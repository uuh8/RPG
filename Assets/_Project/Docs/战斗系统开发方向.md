## 〇、先建立一个心智模型：战斗是一个"闭环"

很多新手把战斗系统理解成"角色能攻击"。但一个**好玩**的战斗系统，本质是一个**双向闭环**：

```
        ┌──────────────── 数值层（多疼） ─────────────────┐
        │                                                  │
   我方输出端  ──攻击/技能──►  时序·命中·空间  ──►  对手端（敌人/AI）
   (你已基本做完)                  │  ▲                  (你目前完全缺)
        ▲                         │  │                       │
        │                         ▼  │                       │
        └──── 防御/博弈 ◄──── 反馈层（看得懂·打得爽）◄────────┘
                (你目前缺)            (你部分有)
```

"好玩"几乎不来自单点功能，而来自这个环里**各层之间的张力**：

- 没有**对手**，输出再华丽也没有意义（打木桩）。
- 没有**数值**，伤害只是数字搬运，谈不上平衡与成长。
- 没有**博弈**（防御选项），战斗是单向碾压，不是"对话"。
- 没有**反馈**，玩家"看不懂威胁、打不出爽感"。

所以判断战斗系统完不完整，不是看招式多不多，而是看**这个环闭没闭、每层够不够厚**。下面按"层"展开。

# 第一部分：一个好玩的战斗系统由什么构成

把战斗系统拆成 **6 层 + 1 个对手端**。每层我都按 `是什么 / 为什么影响好玩 / 业界常见做法 / 关键设计抉择 / 怎么算做得好` 来讲。

---

## 一层：数值与属性层（"多疼" —— 战斗的数学内核）

**是什么**：把"造成伤害"变成一套可设计、可平衡的数字系统。属性（攻击/防御/暴击…）+ 伤害公式 + 成长曲线。

**为什么影响好玩**：数值是"成长感"和"平衡"的来源。同一招，1 级打 10 点和 60 级打 1 万点，靠的就是数值系统。暴击带来的随机惊喜、减伤带来的坦克体验，全在这一层。

**业界常见做法**：
- **属性块（Stats）**：攻击力、防御/护甲、暴击率、暴击伤害、攻速/施法速度、穿透、各伤害类型抗性、命中/闪避。
- **减伤公式**两大流派：
  - **减法**：`实际伤害 = 攻击 - 防御`。直观，但高低防差距悬殊、容易出现"破不了防"。
  - **乘法/比例**：`减伤率 = 护甲 / (护甲 + K)`。永远不会完全免疫、边际递减平滑，**主流 ARPG 多用这种**。
- **暴击**：roll 一个随机数 < 暴击率 → 伤害 × 暴击倍率。
- **伤害浮动**：±5%~10% 随机，避免数字太死板。
- **伤害类型 × 抗性**：物理/魔法/真实 × 对应抗性（你已有 `DamageType` 枚举和 `DefenseProfile` 预留位）。

**关键设计抉择**：
- 减法还是乘法减伤？（决定整个平衡手感）
- 暴击是"高方差刺激"还是"稳定输出"导向？
- 数值要不要随等级指数膨胀（数值膨胀的利弊）？

**怎么算做得好**：公式能用**纯函数**表达、能写单元测试、改一个参数能预测对战斗的影响。你的 `DamagePipeline.Resolve` 已经是纯静态函数 + 有 EditMode 测试——这层你**地基已铺好，只差把 passthrough 换成真实公式**。

## 二层：时序与帧数据层（"打得准不准、爽不爽" —— 手感内核）

**是什么**：把一次攻击在时间轴上拆成阶段，并定义"哪一刻能造成伤害、哪一刻能被打断、哪一刻能接下一招"。

**为什么影响好玩**：动作游戏的"手感"几乎全在这一层。同样一刀，**前摇多久、命中判定开多久、后摇能不能取消**，决定了它是"笨重"还是"流畅"。

**业界常见做法**：
- **三段式帧数据**：**前摇(startup) → 命中(active) → 后摇(recovery)**。这是格斗/动作游戏的通用语言。
- **取消窗口(cancel window)**：后摇期间允许用某些操作（闪避/下一段连招）打断，产生"连招"和"取消"。
- **顿帧 / Hit-Stop（帧停）**：命中瞬间双方定格 2~5 帧。**打击感性价比最高的一招**，几乎零成本。
- **无敌帧(i-frames)**：某些动作（翻滚）期间免疫伤害。
- **霸体(super armor) / 削韧(poise)**：被打不被打断 / 攒够削韧才被打断。决定"谁能强吃一拳继续输出"。

**关键设计抉择**：
- 时序数据放哪？你的项目用 **`normalizedTime` 阈值 + ScriptableObject**（而非 Animation Event），统一、可单测——延续这套即可。
- 后摇能否被移动/闪避取消？（流畅度 vs 硬核感）
- 谁有霸体？（普通敌人没有、Boss 大招有，是难度设计的杠杆）

**怎么算做得好**：玩家能"感觉到"攻击的轻重缓急；高手能通过取消把招式打出连贯节奏。

> 你已有 hit window / combo input window / 单点生成 这套时序框架，**最该补的是 hit-stop 和 i-frame**——立刻让战斗"有重量、有攻防"。

## 三层：攻防博弈层（"战斗是不是一场对话"）

**是什么**：给玩家**防御与反制**的选项，让战斗从"单向输出"变成"读招—应对"的双向博弈。

**为什么影响好玩**：这是"硬核战斗"爽感的核心。魂系、只狼、猎天使魔女的精髓都在这层——**风险与回报的瞬时决策**。

**业界常见做法**：
- **闪避(dodge)**：带 **i-frame 的翻滚**。你已有 Dash，**加几帧无敌就是翻滚闪避**，改动极小、收益极大。
- **格挡(block)**：减伤 + 消耗耐力 + 削韧。
- **招架/弹反(parry)**：**精确时机**格挡 → 弹开敌人/造成破防 → 反击窗口。经典的"高风险高回报"。
- **完美闪避 → 子弹时间 / 反击**（猎天使魔女 Witch Time）。
- **破防(guard break) / 处决(execution)**：削韧满 → 敌人硬直 → 高伤处决。

**关键设计抉择**：
- 闪避是"位移型"（魂系翻滚）还是"原地无敌型"（弹幕擦弹）？
- 弹反的时机窗口多宽？（宽 = 容易上手，窄 = 硬核）
- 防御消耗什么资源？（耐力是约束防御滥用的关键）

**怎么算做得好**：玩家会主动"读"敌人出招、用对的应对方式，而不是无脑输出。

---

## 四层：状态效果与控制层（系统深度的放大器）

**是什么**：一套通用的"在目标身上施加一个有持续时间的效果"的框架。增益(buff)、减益(debuff)、持续伤害/治疗(DoT/HoT)、控制(CC)。

**为什么影响好玩**：它是**内容产能的放大器**。有了通用框架，你设计一个新技能 = 拼装几个 effect，而不是写新代码。也是构建 build/流派/连携的基础。

**业界常见做法**：
- **通用 Status/Effect 组件**：持续时间、**叠层(stacking)**、刷新规则、互斥/覆盖、可驱散(dispel)。
- **DoT/HoT**：你已有 `BurnStatus`（燃烧）作为雏形——**把它泛化**就能得到中毒、流血、再生等一整类。
- **控制(CC)**：眩晕(stun)、定身(root)、减速(slow)、沉默(silence)、击飞(knockup)、恐惧。CC 是 PvE/PvP 深度的核心。
- **进阶：元素反应**（原神式）——不同元素状态叠加触发新效果，制造系统涌现性。

**关键设计抉择**：
- 重复施加：**刷新时长**还是**独立叠层**还是**叠层加伤**？（你在法师文档里已分析过这点）
- CC 要不要递减(diminishing returns)防连控？
- buff 上限、是否可被覆盖？

**怎么算做得好**：新效果几乎不写新框架代码就能配出来；效果之间能产生有趣的组合。

> 你已经有 `BurnStatus` + `DamageReceivedEvent` 的雏形，**把它抽象成通用 StatusEffect 框架**，是性价比极高的一步。

---

## 五层：资源与节奏层（战斗的"经济系统"）

**是什么**：给技能与防御加上"成本"，制造取舍与节奏。

**为什么影响好玩**：免费的能力没有决策。**法力/耐力/能量/怒气**让玩家必须思考"现在该用什么、留着什么"。耐力还能约束闪避/格挡/攻击的频率，是魂系节奏的命脉。

**业界常见做法**：
- 资源类型：**法力(MP)**（技能消耗）、**耐力(Stamina)**（攻击/闪避/格挡消耗，魂系核心）、**能量/怒气**（攒大招）。
- 回复：自然回复、攻击回复、击杀回复。
- **冷却(cooldown) / 充能层数(charges) / 公共冷却(GCD)**。你刚给普攻加的攻击冷却，就是这套的雏形。
- **吸血、护盾、过量治疗**。

**关键设计抉择**：
- 用"冷却"约束还是"资源"约束，还是两者叠加？
- 耐力恢复速度（决定战斗是"喘息式"还是"无脑连打"）。

**怎么算做得好**：玩家在战斗中有"资源管理"的决策密度，而不是一招循环到底。

---

## 六层：技能系统层（把一切封装成可配置技能）

**是什么**：用**数据驱动**的方式，把"伤害 + 效果 + 时序 + 消耗 + 冷却 + 目标方式"打包成一个"技能"，策划改数据就能造新技能。

**为什么影响好玩**：这是战斗内容**可扩展性**的关键，也是大厂战斗程序/策划最核心的能力——**把零散机制抽象成统一可配置格式**。

**业界常见做法**：
- **Skill ScriptableObject**：消耗、冷却、施法方式（瞬发/吟唱 cast/引导 channel）、目标类型、**效果列表(effect list)**。
- **效果组合**：一个技能 = 一串 effect（造成伤害 + 施加 debuff + 位移 + 召唤），数据拼装。
- **连招树 / 派生招 / 技能取消**。

**关键设计抉择**：
- 效果系统的颗粒度（太细配置繁琐，太粗复用差）。
- 施法方式如何统一（你的陨石"引导态"已经是 channel 的实现，可复用）。

**怎么算做得好**：新技能 90% 靠配数据、不写或少写代码。

> 你的 asmdef 里 `Game.Skills` 和 `CombatDamage` 接缝**一直是空的**——这层就是它们的归宿。

---

## 七、对手端：敌人与 AI（最重要、你完全缺的一块）

**是什么**：会感知玩家、会决策、会出招、能被击败的敌人。

**为什么影响好玩**：**没有对手，前面 6 层全部无法被检验**。战斗的乐趣 90% 来自"应对一个有威胁、可读、可击败的对手"。这也是你目前**最大的缺口**。

**业界常见做法**：
- **感知(perception)**：视野/距离/仇恨(aggro)。
- **决策**：先用**有限状态机**（巡逻→追击→攻击→受击→死亡），复杂后再上**行为树(Behavior Tree)**。
- **攻击模式 + 前摇预警(telegraph)**：敌人出招前给"地面指示器/起手动作"，玩家才有机会闪/格——**PvE 可读性的关键**。
- **Boss 阶段(phase) / 狂暴(enrage)**：分血量阶段切换招式。
- **刷怪器(spawner) / 波次**。

**关键设计抉择**：
- 敌人有没有霸体？（决定能不能被普攻打断）
- telegraph 给多长？（难度旋钮）
- 群战 AI 的"攻击令牌"机制（避免所有敌人同时扑上来）。

**怎么算做得好**：玩家能"看懂"敌人要干什么、并用学到的防御手段应对；难度有曲线。

> 好消息：你的 `HealthComponent` / `IDamageable` / 伤害管线 / 受击死亡反馈**本来就是通用的**，敌人可以直接复用，不用重写。

---

## 八、反馈与游戏感层（"看得懂 + 打得爽" —— juice）

**是什么**：把战斗中发生的事**清晰、爽快地传达**给玩家。低成本、高回报，但极其影响体感。

**为什么影响好玩**："爽"这个字几乎全在这层。同样的伤害，加上**飘字 + 顿帧 + 震屏 + 特效音效**，体感天差地别。

**业界常见做法**：
- **伤害飘字**（暴击特殊样式）、**血条**（头顶浮动 + 玩家 UI）。
- **顿帧(hit-stop)**（同二层）、**震屏(screen shake)**、**FOV 冲击**。
- **命中 VFX/SFX 分层**（轻击/重击/暴击/不同部位不同表现）。
- **受击反馈**：闪白、击退、硬直动画（你已有闪红 + 受击/死亡动画 + 刚加的流血）。

**关键设计抉择**：
- 反馈强度的"度"（过度震屏会晕、过多飘字会糊）。
- 信息优先级（玩家最该看到什么）。

**怎么算做得好**：玩家闭着眼睛都能"感觉"到自己打中了、打了重的还是轻的、暴击没暴击。

> 你的 `EventBus`（`DamageReceivedEvent` / `DeathEvent`）让这一层可以**零侵入地挂上去**——表现层订阅事件即可，gameplay 不用改。

---

## 小结：好玩战斗系统的"完整度自检表"

| 层 | 一句话 | 没有它会怎样 |
|----|--------|--------------|
| 数值 | 多疼、能不能成长 | 伤害只是数字搬运，无平衡 |
| 时序/帧 | 打得准不准、流不流畅 | 招式笨重、无连招无取消 |
| 攻防博弈 | 是不是双向对话 | 单向碾压，无硬核爽感 |
| 状态/CC | 系统有多深 | 内容产能低，无 build/连携 |
| 资源/节奏 | 有没有取舍 | 一招循环到底，无决策 |
| 技能系统 | 内容好不好扩展 | 每个技能都写死，难扩展 |
| 敌人/AI | 有没有对手 | 打木桩，前面全白做 |
| 反馈/juice | 看不看得懂、爽不爽 | 闷、糊、不知道发生了什么 |

# 第三部分：后续开发路线图（分阶段、带优先级）

延续你项目的"里程碑 M#"习惯（前面到 M3）。每个阶段我标注：**目标 / 包含 / 为什么这个顺序 / 挂到你的架构 / 简历价值 / 验收**。建议严格按顺序，因为后一阶段依赖前一阶段才能被检验。

---

## 🎯 M4：敌人最小可玩 + 真实数值（最高优先级，先做这个）

**目标**：让战斗第一次"成环"——有一个会动会打、能被你的三职业击败的敌人，且伤害是真实算出来的。

**包含**：
1. **EnemyController**：复用 `HealthComponent`/`IDamageable`（已通用），加一个**有限状态机**：`Idle/巡逻 → 侦测玩家 → 追击 → 攻击(带前摇) → 受击 → 死亡`。
2. **简单感知**：距离/视野检测玩家、进入仇恨。
3. **敌人攻击**：一个带**前摇预警**的近战攻击，走和玩家**同一条** `ReceiveHit` 管线打到玩家。
4. **真实伤害公式**：把 `DamagePipeline.Resolve` 从 passthrough 换成"攻击力 - 减伤 + 暴击 + 浮动"，并补单元测试。
5. **StatDefinition SO**：攻击/防御/暴击等属性，挂到 `HealthComponent`。

**为什么先做**：没有对手，你已完成的 80% 都在打木桩；没有真实数值，伤害没有意义。这两件事**一旦做完，整个项目立刻"像个游戏"**。

**挂到你的架构**：敌人 = 一个新的 MonoBehaviour + 一套手写状态机（和你玩家状态机同思路）；伤害公式纯函数化 + 单测（和 `DamagePipeline` 现有测试同套路）。

**简历价值**：⭐⭐⭐⭐⭐ "实现了数据驱动的伤害结算管线（攻防/暴击/减伤，纯函数 + 单元测试）"和"敌人 AI 状态机"是战斗岗位最硬的两个点。

**验收**：能被一个会追你、会打你的敌人威胁到；伤害数字符合公式预期；敌人和玩家共用同一伤害管线。

---

## 🎯 M5：攻防博弈（闪避 i-frame + 格挡/弹反）

**目标**：把单向输出变成"读招—应对"的双向战斗。

**包含**：
1. **翻滚闪避**：给现有 Dash 加 **i-frame 窗口**（用你已有的 `normalizedTime`/计时器思路）。
2. **Hit-Stop（顿帧）**：命中瞬间短暂 `Time.timeScale` 压低或局部定格。**性价比最高的爽感**。
3. **格挡(block)**：减伤 + 消耗资源 + 朝向判定。
4. **弹反/招架(parry)**（进阶）：精确时机 → 敌人破防 + 反击窗口。
5. **削韧(poise)**：给敌人/玩家加韧性条，决定能否被打断。

**为什么这个顺序**：有了敌人(M4)才有"被打"，才谈得上防御。

**挂到你的架构**：i-frame = 受击时检查一个"无敌中"标志；hit-stop = 命中事件触发的全局/局部时停；这些都能挂在 `DamageReceivedEvent` 上。

**简历价值**：⭐⭐⭐⭐ "i-frame 翻滚 / 弹反 / 顿帧"是动作战斗手感的招牌词，面试官一听就懂你的水平。

**验收**：翻滚能无伤穿过敌人攻击；命中有"咔"的顿挫；格挡/弹反成立。

---

## 🎯 M6：通用状态效果与控制(CC)框架

**目标**：把 `BurnStatus` 泛化成"能配出任意 buff/debuff/CC"的通用框架。

**包含**：
1. **StatusEffect 框架**：持续时间、叠层、刷新规则、互斥、可驱散。
2. **常见效果**：中毒/流血(DoT)、再生(HoT)、攻防速 buff、护盾。
3. **控制(CC)**：眩晕/定身/减速/击飞——接到角色状态机（被控时禁用输入）。
4. **(可选进阶) 元素反应**：两种状态叠加触发新效果。

**挂到你的架构**：现有 `BurnStatus` 重构为继承自一个 `StatusEffectBase`；CC 通过 EventBus 或直接控制状态机进入"被控状态"。

**简历价值**：⭐⭐⭐⭐ "可扩展的状态效果/Buff 系统（叠层、刷新、CC）"体现系统抽象能力。

**验收**：新增一个 debuff 几乎不写框架代码；CC 能真正打断玩家/敌人行动。

---

## 🎯 M7：资源 + 技能系统（`Game.Skills` 上线）

**目标**：让技能有成本、有冷却，并把战斗机制封装成数据驱动的"技能"。

**包含**：
1. **资源**：法力 MP / 耐力 Stamina（攻击/闪避/格挡消耗）。
2. **SkillDefinition SO**：消耗、冷却、施法方式（瞬发/吟唱/引导，复用陨石引导态）、**效果列表**。
3. **效果组合**：技能 = 伤害 effect + buff effect + 位移 effect 的数据拼装。
4. **技能栏 / 冷却 UI**。

**挂到你的架构**：填上 `Game.Skills` 和 `CombatDamage` 这两个空接缝；技能效果调用 M6 的状态框架 + M4 的伤害管线。

**简历价值**：⭐⭐⭐⭐⭐ "数据驱动技能系统（效果可组合、策划零代码配技能）"是战斗程序的核心卖点。

**验收**：能纯靠配 SO 造出一个新技能（消耗法力、有冷却、造成伤害并附加减速）。

---

## 🎯 M8：目标与空间（锁定 / telegraph / 投射物变体）

**目标**：提升近战定位与 PvE 可读性。

**包含**：锁定(lock-on)/软锁定、敌人攻击的**地面预警 telegraph**、AOE 形状库、投射物变体（追踪/弹射/穿透——扩展 `ProjectileBase`）。

**简历价值**：⭐⭐⭐ 锁定系统和 telegraph 体现"玩家体验/可读性"意识。

---

## 🎯 M9：游戏感打磨（juice）

**目标**：把已有反馈升级成"爽"。

**包含**：伤害飘字（暴击特殊样式）、头顶血条 + 玩家 HUD、震屏、命中 VFX/SFX 分层。**全部订阅你已有的 EventBus，零侵入。**

**简历价值**：⭐⭐⭐ "game feel / 打击感"是体验向岗位的加分项；做一个对比视频(加 juice 前后)非常有说服力。

---

## 🎯 M10：工程化收口

**目标**：把项目从"能跑"提升到"工程严谨"。

**包含**：投射物/VFX/飘字**对象池**（解决高频 Instantiate 的 GC）、场景生命周期管理（`EventBus<T>.Clear()`）、更多伤害公式单测、调试可视化（hitbox/telegraph Gizmos）。

**简历价值**：⭐⭐⭐⭐ "对象池 / 零 GC / 模块隔离 / 单元测试"是面试**必问**的工程素养，能把你和"只会堆功能"的人区分开。

# 第四部分：把这个项目写进简历 / 面试怎么讲

你的目标是补战斗系统经验、写进简历。这里给你"变现"建议。

## 简历可写的亮点（按含金量排序）

挑你**真正做完**的写，用"做了什么 + 怎么做 + 量化/技术名词"的句式：

- **数据驱动的战斗框架**：伤害结算管线（值快照 `DamageRequest` → 纯函数 `DamagePipeline` → 事件广播），攻防/暴击/减伤公式，**纯函数 + NUnit 单元测试**。
- **多角色战斗架构**：base/subclass + 虚方法接缝(`TryStartAttack`)统一近战/远程/法术三套攻击，共享一套手写状态机。
- **数据驱动动画时序**：基于 `normalizedTime` 阈值 + ScriptableObject 的命中/连段/蓄力/单点生成时序系统（替代 Animation Event，可单测）。
- **投射物系统**：模板方法抽象基类，支持抛物/直射/防穿地(连续碰撞)/命中残留，复用于箭/火球/陨石。
- **解耦的表现层**：基于泛型 `EventBus<T>`（struct 事件、零装箱）的发布订阅，受击/死亡/流血/UI 反馈零侵入挂载。
- **性能纪律**：热路径零 GC（`NonAlloc` 系列、预分配、`StringToHash` 缓存、`[Conditional]` 日志），Profiler 验证。
- **状态效果系统**（做完 M6 后写）：可叠层/刷新/驱散的通用 buff/debuff/CC 框架。
- **敌人 AI**（做完 M4 后写）：感知—决策—攻击—受击—死亡的状态机，与玩家共用伤害管线。

## 面试官（尤其米哈游这类）会重点追问

1. **"你的伤害是怎么算的、怎么保证可测试？"** → 讲值快照 + 纯函数管线 + 单测。
2. **"攻击时序怎么做的？为什么不用 Animation Event？"** → 讲 normalizedTime + SO + 可单测 + 统一性。
3. **"怎么保证战斗这种高频场景不卡（GC）？"** → 讲零 GC 纪律。
4. **"模块怎么解耦的？UI 怎么不依赖 gameplay？"** → 讲 asmdef 单向依赖 + EventBus。
5. **"如果再加一个职业/一个技能要改哪里？"** → 讲接缝设计 + 数据驱动（加子类/配 SO）。
6. **"打击感怎么做的？"** → 讲 hit-stop / i-frame / 反馈分层（做完 M5/M9 后底气最足）。

> 你项目里已经有大量 `*_System_Design.md` 和 `*_Interview_QA.md`——**这些文档本身就是简历的加分项和面试素材库**，证明你不只是会写代码，还能系统性地设计与复盘。

## 一个建议

做完 **M4 + M5 + M9** 后，录一段 **1~2 分钟的战斗演示视频**（三职业 × 打会反击的敌人 × 翻滚弹反 × 顿帧飘字），放进简历/作品集。**动作战斗岗位，一段会动的演示胜过千言万语。**

## 附：值得了解的概念关键词（自学检索用）

- 数值：`armor mitigation formula`、`crit multiplier`、`damage variance`、`damage type & resistance`
- 手感：`frame data (startup/active/recovery)`、`hit-stop / hitstun`、`i-frames`、`poise / super armor`、`animation canceling`
- 博弈：`parry / riposte`、`perfect dodge`、`stamina-based combat`、`guard break`
- 系统：`status effect / buff stacking`、`crowd control & diminishing returns`、`ability/effect system (GAS)`、`elemental reaction`
- AI：`perception / aggro`、`behavior tree vs FSM`、`attack telegraph`、`boss phase / enrage`、`attack token (group AI)`
- 工程：`object pooling`、`event bus / observer`、`data-driven design (ScriptableObject)`、`zero-allocation gameplay`

经典可参考的战斗手感标杆：**只狼/黑魂（攻防博弈、削韧）、猎天使魔女（完美闪避）、鬼泣（连招取消）、原神（元素反应、技能系统）、怪物猎人（帧数据与武器 moveset）**。
