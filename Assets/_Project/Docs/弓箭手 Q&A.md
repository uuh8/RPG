## 一、架构与设计模式

#### **Q1：你这个项目有战士和弓箭手两个角色，代码上是怎么组织的？为什么这么做？**

我用的是「继承 + 多态」的 base/subclass 结构。抽了一个抽象基类 `PlayerControllerBase`，把两个角色共有的能力——移动、跳跃、冲刺、相机、输入计时器、状态机基础设施，以及接地/空中/滑坡/冲刺这四个共享状态——全部放进去；然后 `WarriorController` 和 `ArcherController` 各自继承它，只补自己的攻击部分。我之所以选继承而不是「每个角色复制一份脚本」，是因为复制会导致共享逻辑改一处要同步多份、迟早不同步出 bug；之所以没上更重的「组件化/组合」或「泛型基类」，是因为现在只有两个角色，差异本质就是「近战一种连段 vs 远程两种攻击」这种行为型差异，继承最直观、成本最低。组合（ability system 那种）是角色数量多、能力要自由拼装时才划算的方案，现在上属于过度设计。

#### **Q2 ⭐：基类里有移动逻辑要触发攻击，但基类并不知道"攻击"到底是近战还是射箭，你怎么解决的？**

我用了一个虚方法做「接缝（seam）」。基类暴露 `public virtual bool TryStartAttack()`，默认返回 false。共享的地面状态在它的转换优先级链里（Dash → Attack → Jump）只调用 `if (_player.TryStartAttack()) return;`，它完全不关心攻击具体是什么，只负责把「攻击」这个优先级槽位占住。然后战士子类 override 它去进连段状态，弓箭手 override 它去做点按/按住的路由。这样做的好处是：Dash→Attack→Jump 这个**优先级规则只写在共享处一次**，三个角色都遵守，而「攻击具体干什么」被多态分发到各自子类。这是典型的「把策略外包给子类、把流程骨架留在基类」，本质是模板方法 + 多态。

#### **Q3：状态对象怎么访问子类专属的数据？你没用泛型，会不会到处强转？**

不会。我做了一个分层：共享状态（接地/空中等）里持有的 `_player` 字段类型就是基类 `PlayerControllerBase`，它们只用基类成员，所以对谁都通用；而角色专属状态，比如弓箭手的 `PlayerChargeAttackState`，构造时传入的是**具体子类** `ArcherController`，它在 `base(player)` 存一份基类引用的同时，自己再存一份 typed 的 `_archer` 引用。于是 **`_player` 走共享成员、`_archer` 走专属成员**，各取所需，全程不强转、不用泛型。我特意避开泛型基类 `Base<T>`，因为泛型会传染到状态机、序列化、Inspector，让可读性和 Unity 兼容性变差，换不来等价收益。

## 二、状态机

#### **Q4：描述一下你的角色状态机，它和 Unity 的 Animator 状态机是什么关系？**

我的是一套手写的、纯 C# 的逻辑状态机，和 Animator 是两套独立的东西。每个状态是普通 C# 对象（不是 MonoBehaviour），在 `Awake` 里预先实例化好、运行时只切换引用，所以**切状态零 GC**。状态机核心就一个 `ChangeState`：先调旧状态的 `Exit()`，再换引用，再调新状态的 `Enter()`，**无条件执行**。

逻辑状态机负责「角色现在该按什么规则移动、能不能攻击」；Animator 负责「现在该播哪段动画」。两者通过参数（speed/isGrounded）和代码 CrossFade 对接，但职责分开。这种分离很重要——我后面修过一个 bug 就是因为把这两件事混为一谈（动画在播但人没动）。

#### **Q5 ⭐：为什么资源释放（比如冲刺冷却、准心隐藏）你都放在 Exit()？**

因为 `ChangeState` 保证「离开任何状态必然经过它的 `Exit()`」，所以 `Exit()` 是**唯一一个能覆盖所有离开路径**的收尾点。举两个例子：冲刺的冷却计时器我在 `PlayerDashState.Exit()` 里启动，这样不管冲刺是自然结束、还是将来新增一条「被击打断」的退出路径，冷却都一定会启动，我永远不用去每条分支里补「记得启动冷却」。同理，蓄力瞄准的准心隐藏放在 `PlayerChargeAttackState.Exit()`，保证**不管蓄力是正常放完还是被打断，准心都不会残留在屏幕上**。把收尾绑在「唯一必经的出口」，是状态机里最可靠的一种写法。

## 三、输入系统

**Q6 ⭐：弓箭手点按是普攻、按住是蓄力，同一个左键，你怎么分辨这两个意图？**

我用**「事件回调 + 每帧轮询」**两种读法配合。按下那一帧我用 Input System 的 `performed` 回调写一个 `AttackBufferCounter = 0.15s`，这个缓冲计数器保证「哪怕只按一帧的快速点击也不会丢」，这是亚帧容错。同时，在每帧调用的 `TryStartAttack` 里，只要 `IsAttackHeld`（轮询 `IsPressed()`）为真，我就 `_attackHeldTime += Time.deltaTime` 累计按住时长，一旦越过 `TapThreshold` 就**立刻**切到蓄力状态（不等松手，这是为了「按住到点马上拉弓」的即时手感）；如果在越阈之前就松手了，下一帧判定为点按，进入普通攻击。

**Q7：你提到「缓冲计数器」，这个 buffer 的设计在项目里是不是反复出现？**

是的，这是一个统一的「buffer counter」模式，项目里用了至少五处：**跳跃缓冲、土狼时间（coyote time）、攻击缓冲、冲刺缓冲、冲刺冷却**。本质都是一个会自己过期的 float 信号：输入回调把它设成一个时长，`PlayerControllerBase.Update` 每帧统一 `-= Time.deltaTime`，状态读「> 0 表示有效」、消费时置 0。它比一个纯 bool 多了「时间维度」。我把所有这类计数器集中在控制器的 Update 递减，而不是放在各自状态里，是因为它们是**全局事实**——比如冲刺冷却，不管你现在在地面、空中还是攻击中都该继续走，放进某个状态的 Update 会导致那个状态不活跃时计时停掉，这是个隐蔽 bug 源。

## 四、战斗与伤害

**Q8 ⭐：近战是挥砍判定，远程是飞行的箭，两套命中机制，你的伤害结算是怎么统一的？**

我把「怎么发现命中」和「命中之后怎么算伤害」彻底分开。发现命中可以多样：近战是 `OverlapBox` 每帧扫武器附近的盒子，远程是箭的 `OnCollisionEnter`。但两者发现目标后，都构造同一个 `DamageRequest`（一个值快照结构体）去调用同一个接口 `IDamageable.ReceiveHit(in DamageRequest)`，后面的伤害管线 `DamagePipeline.Resolve` 是个纯静态函数，谁来都一样算。这样的好处是伤害公式只有一份、可单元测试，新增任何攻击形态时我只要回答「我能不能复用 ReceiveHit 这条管线」，几乎都能。这就是「发现方式可以不同，结算路径必须统一」。

**Q9：你说 DamageRequest 是「值快照」，为什么强调值快照？**

因为箭是飞行的，从射出到命中可能有近一秒，这期间射手完全可能已经死了、被销毁了。如果命中时我才回头去问「射手的攻击力是多少、是什么阵营」，就可能访问到一个已经不存在的对象。所以我在箭**生成的瞬间**就把攻击者 ID、阵营、伤害值、伤害类型全部拷进箭自己身上（`Init` 注入），命中时直接用这份快照构造 `DamageRequest`，绝不回查射手。这是一种「在数据还有效时就拍照存档」的思路，近战的 `DamageRequest` 也是同一个哲学，只是近战命中是瞬时的，矛盾没那么尖锐。

**Q10：怎么避免箭射到自己或队友？**

两道防线。第一道是出膛防自撞：生成时调 `Physics.IgnoreCollision(箭的 collider, 射手的 collider)`，避免箭一出现就撞到射手身体直接销毁。第二道是阵营过滤：每个可受击物体有 `TeamId`，箭命中时先比 `target.TeamId == _attackerTeam`，相同就直接 return——穿过去，不结算也不销毁。这样队友之间互不误伤，箭也能飞过队友继续打后面的敌人。

## 五、投射物与物理

**Q11 ⭐：实现「子弹/箭命中」业界有哪几种做法？你选了哪种，为什么？**

主要三种。第一种是**物理投射物**（physical projectile）：场景里真有一个带 Rigidbody 的物体在飞，靠碰撞回调命中——我选的就是这个。第二种是**射线命中**（hitscan）：开火瞬间发一条 Raycast，打到谁就是谁，没有飞行时间，FPS 步枪常用，优点是绝不穿模、零飞行物体，缺点是没有飞行表现、不能被闪避。第三种是逻辑用射线/插值算命中、视觉只播特效的「伪投射物」，性能最好但实现最绕。我选物理投射物，是因为弓箭手要的就是「看得见箭飞、能形成抛物线手感、未来能被打落或闪避」，这只有真实飞行物体最贴合，也最容易做出「普攻抛物线 vs 蓄力直射」的差异。

**Q12 ⭐：物理投射物高速飞行会穿模（一帧穿过薄墙），你怎么解决？**

根因是默认的离散碰撞检测每个物理步只看「这一刻有没有重叠」，速度快时两帧之间就跨过了薄墙。工业上常见两招：一是把 Rigidbody 的碰撞检测模式改成 **Continuous（连续检测）**，让物理引擎用扫掠（sweep）方式检测这一步移动路径上有没有碰撞；二是自己每帧从上一帧位置到当前位置做一条 Raycast 补检，命中就手动结算。我当前项目箭速不高，暂时用默认离散检测够用，但如果做高速狙击箭或子弹，我会优先开 Continuous，必要时再加 raycast sweep 兜薄物体。

**Q13：角色用 CharacterController，箭用 Rigidbody，为什么不统一？**

因为它们要的运动模型不同。角色要的是「精确、不被物理乱推、贴着地面走台阶」的可控移动，`CharacterController` 提供 `Move` 接口、自带胶囊碰撞和台阶处理，不走物理积分，手感最稳——而且它必须在 `Update` 里 Move 才能每帧响应输入。箭要的是「受初速度和重力驱动、撞东西有真实碰撞回调」的物理运动，这正是 Rigidbody 的强项。所以项目的分工就是「角色 CharacterController、投射物 Rigidbody」，各用各擅长的。

## 六、3D 数学（重点）

**Q14 ⭐：箭飞出去时怎么让模型朝向飞行方向？你踩过什么坑？**

我用 `Quaternion.LookRotation(velocity)` 来生成朝向，它返回一个让物体**本地 +Z 轴**对齐到 velocity 方向的旋转——因为 Unity 约定 +Z 是 forward。但我踩了个坑：这个箭的美术模型「箭尖」是沿**本地 +Y** 建模的，于是 LookRotation 把 +Z 对准了飞行方向，箭尖（+Y）却指向侧上方，看起来箭是「竖着」飞的。我的解法是加一个修正旋转：`transform.rotation = Quaternion.LookRotation(velocity) * Quaternion.Euler(90, 0, 0)`，先 LookRotation 把 +Z 对准方向，再绕本地 X 转 90° 把 +Y 掰到飞行方向。我把这个修正角做成了 Inspector 可调字段，因为「模型箭尖朝哪」是资产属性，换个模型可能要改成 -90，让美术能自己调比改代码好。通用教训是：凡是用 LookRotation 对齐的都是 +Z，模型若不是 +Z 朝前就要补一个本地修正旋转。

**Q15 ⭐：四元数和欧拉角的区别？为什么旋转插值要用四元数？什么是万向锁？**

欧拉角是用绕 X/Y/Z 三个轴的角度来描述旋转，直观、好填，但有两个硬伤：一是**万向锁（gimbal lock）**——当中间那个轴转到 90° 时，剩下两个旋转轴会重合，丢失一个自由度，导致某些朝向无法平滑表达；二是欧拉角直接插值会因为轴的耦合产生抽搐、不走最短路径。四元数用 (x,y,z,w) 四个分量表示「绕某个轴转某个角度」，数学上 `q = (sin(θ/2)·axis, cos(θ/2))`，它没有万向锁、能稳定表示任意朝向，而且能做球面插值得到匀速、最短路径的平滑过渡。所以引擎内部旋转都用四元数，欧拉角只用在 Inspector 输入这种人类可读的场合。我项目里角色转向、箭朝向全用四元数，相机的 yaw/pitch 我虽然用 float 存角度，但每帧是用 `Quaternion.Euler` 重新构造成四元数去赋值的，且把 pitch 钳制在 [-30, 70]，本质上规避了万向锁的危险区。

**Q16 ⭐：Slerp 和 Lerp 的区别？你的角色转向用的哪个？它帧率无关吗？**

Lerp 是线性插值，在两个值之间走直线，对旋转来说是对四元数分量线性插值再归一化，速度快但角速度不恒定（中间快两头慢）；Slerp 是球面线性插值，沿单位四元数球面的大圆弧走，角速度恒定、是真正的「匀速转向」。我角色转向用的是 Slerp：`transform.rotation = Quaternion.Slerp(current, target, RotationSpeed * Time.deltaTime)`。但要诚实说一点：`speed * Time.deltaTime` 这种写法**严格来说不是帧率无关的**——它是一种指数逼近（每帧朝目标插值固定比例），帧率不同最终曲线略有差异。真正帧率无关的写法应该是 `t = 1 - Mathf.Exp(-k * Time.deltaTime)`。我项目沿用了 `speed*deltaTime` 是为了和代码库既有风格一致、且差异在正常帧率下可忽略；但如果面试官要求严格帧率无关，我会换成 exp 那种形式。

**Q17 ⭐：蓄力时角色要转向相机的水平朝向，"水平"你在数学上是怎么取的？为什么只取水平？**

我取相机的 forward 向量后，把它的 y 分量清零再归一化：`Vector3 f = cam.transform.forward; f.y = 0; f.Normalize();`。这一步几何上是把相机朝向**投影到水平面（XZ 平面）**，去掉俯仰分量，只保留绕世界 Y 轴的方位角（yaw）。然后 `Quaternion.LookRotation(f)` 得到目标朝向去 Slerp。只取水平是因为角色身体不应该跟着相机俯仰前倾后仰，那样很怪；但要注意，我让身体只 yaw，箭的实际发射方向却是**含俯仰**的（另算，见瞄准部分）——这样身体好看、又能往上往下射，是「转向只取水平、射击方向含俯仰」的拆分。

**Q18 ⭐：怎么判断一个敌人在角色的左边还是右边？怎么判断敌人在不在角色面前？**

判断左右用**叉乘**：`Vector3 cross = Vector3.Cross(角色forward, 指向敌人的方向)`，看 `cross.y` 的符号——大于 0 在一侧、小于 0 在另一侧（具体左右取决于坐标系手性，Unity 左手系下可以现场验一下）。判断前后/视野用**点乘**：`float d = Vector3.Dot(角色forward, 指向敌人的方向归一化)`，d 接近 1 是正前方，大于 0 是前半球，小于 0 在背后；如果要判断「在不在某个视野锥里」，把 d 和 `cos(半视角)` 比即可。点乘的几何意义是「一个向量在另一个向量上的投影长度」，等于 |a||b|cosθ，所以归一化后它直接就是夹角余弦，特别适合判方向、判视野；叉乘给出垂直于两向量的向量，模长是 |a||b|sinθ，方向编码了「转向哪一侧」，所以适合判左右和算法线。

#### **Q19：从屏幕准心位置怎么算出世界空间的瞄准方向？**

我用 `camera.ViewportPointToRay(new Vector3(0.5f, 0.5f, 0))`。视口坐标是归一化的，(0.5, 0.5) 就是屏幕正中央、也就是准心所在；这个 API 返回一条从相机出发、穿过该屏幕点射向世界的射线（起点+方向）。我拿这条射线做 `Physics.Raycast` 求命中点 `targetPoint`，命中就用命中点、没命中就取射线上 `AimMaxDistance` 处的远点。最后箭的方向是 `dir = (targetPoint - 箭生成点).normalized`。这里有个关键细节：方向是从**箭的生成点（弓弦）**指向目标点，而不是从相机指向——因为相机和弓弦不在同一点，存在视差；让箭从弓弦出发指向准心命中的那个目标点，才能保证箭真的命中准心所指的东西，这是一种「让发射源和命中点连线收敛到目标」的处理。

**Q20：方向向量为什么几乎都要 Normalize？不归一化会怎样？**

因为我经常把方向乘以一个标量速度来得到速度向量，比如 `velocity = dir * speed`。如果 dir 不是单位向量，它的长度就会混进速度里——距离越远、dir 越长、箭越快，速度就失控了。归一化把方向的「长度信息」剥掉、只保留「方向信息」，speed 才能精确控制大小。另外做点乘判夹角时也必须先归一化，否则 `Dot` 得到的是 |a||b|cosθ 而不是 cosθ，没法直接当余弦用。归一化前我还会查 `sqrMagnitude` 是否接近 0，避免对零向量归一化得到 NaN，并准备一个退化兜底方向（比如用角色 forward）。

**Q21：你代码里常用 sqrMagnitude 而不是 magnitude，为什么？**

因为 `magnitude` 要开平方根（sqrt），而 `sqrMagnitude` 只是分量平方和，省掉了开方。当我只是做「比较」或「判断是否接近 0」时，比如 `if (dir.sqrMagnitude < 1e-6f)`，完全不需要真实长度，和一个同样平方过的阈值比就行，省一次 sqrt。在每帧、每个对象都可能跑的逻辑里，这种小优化累积起来有意义。只有当我真的需要「长度这个数值本身」时才用 magnitude。

## 七、瞄准系统

**Q22 ★：你说过有个"蓄力箭往身后飞"的 bug，完整讲一下原因和你的排查、修复过程。**

现象是蓄力箭不往准心方向飞、反而朝角色背后射。我先看方向是怎么算的：`dir = targetPoint - 箭生成点`。targetPoint 来自屏幕中心射线的命中点。问题就出在这条射线上——我的相机是第三人称轨道相机，在角色**身后**，屏幕中心射线从相机出发往前射，第一个撞到的其实是**角色自己的碰撞体**，于是命中点落在角色身体上；而身体在弓弦生成点（在身前）的**后方**，dir = 身后的点 - 身前的点，自然指向后方。再加上当时 AimMask 配成了 Everything 没排除自己，射线确实会打到自身。修复我没有只靠「让美术把 Player 层从 Mask 里排掉」，因为那依赖一个外部配置永远正确的隐含前提，忘配又会反向。我改成在代码里稳健地剔除自身：用 `Physics.RaycastNonAlloc` 取**沿途所有**命中，遍历时跳过「属于射手的碰撞体」（`hit.collider.transform.IsChildOf(_player.transform)`，IsChildOf 含自身，所以 CharacterController 和身体网格都会被跳过），再在剩下的命中里取最近的。这样无论 Mask 怎么配都对。

**Q23：用 RaycastNonAlloc 有个坑，结果是不排序的，你怎么处理的？**

对，`Physics.RaycastNonAlloc` 把命中写进我预分配的数组里，但**不保证按距离排序**，也可能在数组填满时截断。所以我不能直接拿第 0 个当最近命中，而是遍历所有有效命中、用一个 `nearest` 变量手动求 distance 最小的那个。数组我预分配成 `RaycastHit[16]`，一是遵守项目「热路径零 GC」的纪律（NonAlloc 不产生垃圾），二是 16 这个容量足够一条射线沿途的碰撞体数。

**Q24：普攻是抛物线、蓄力是直线命中，这个差异你代码上怎么实现的？**

就一个 bool 开关。`Arrow.Init` 有个 `useGravity` 参数：普攻箭传 true，箭的 Rigidbody 开重力，自然走抛物线，有真实的远近下坠手感；蓄力瞄准箭传 false，关重力，箭走直线，配合前面算出的精确方向，实现「指哪打哪」的精确命中。同一个 Arrow 类，一个参数切两种弹道，不用写两套投射物。这个开关其实是我在做普攻（阶段 3）时就预留的伏笔，到做瞄准（阶段 5）时正好复用。

## 八、相机

**Q25 ★：原神弓箭手蓄力时角色会移到画面一侧露出准心，你是怎么实现的？讲讲数学关系。**

我没有移动角色，而是移动相机——用 Cinemachine 的 `CinemachineCameraOffset` 扩展给相机加一个相机空间的偏移。核心数学关系是：**相机平移和画面里静止物体的视觉移动永远相反**。就像你举手机拍一个站着不动的人，手机往右挪，取景框里的人就往左跑。所以相机空间偏移 +X（相机右移），角色在屏幕上左移；偏移 +Y（相机上移），角色在屏幕上下移。我要把角色推到左下，就给偏移 (+X, +Y)。具体实现是订阅蓄力时发布的 `AimStateChangedEvent`，瞄准时把目标偏移设成配置值、退出时设回零，每帧用 Lerp 平滑逼近，所以进入和归位都是顺滑过渡而不是瞬移。我特意让那个偏移扩展的 `PreserveComposition` 保持关闭，因为开启它会让相机偏移后重新转回来对准角色、产生旋转感，而我要的是干净的平移让角色真的滑出去。

**Q26：这个相机效果是加在哪个模块的？有没有动到战斗代码？**

加在表现层 `Game.Rendering`，是个独立组件 `AimFramingController`，它只订阅 `Game.Core` 里的 `AimStateChangedEvent`，完全不认识弓箭手。我做这个功能时**一行 gameplay 代码都没改**，只是给那个早就存在的事件多挂了一个监听者——和准心 UI 是同一个事件的两个独立订阅者。这正是我想强调的：因为瞄准开始/结束在更早的阶段就被广播成了事件，后来加任何瞄准相关的表现（准心、相机偏移、未来的 FOV 收窄、音效）都只要订阅它，不必碰玩法代码。这是事件解耦埋下的红利。

## 九、动画与时序

**Q27 ★：箭应该在放箭动画的某一帧生成，而不是按键瞬间。你怎么对齐这个时机？为什么不用 Animation Event？**

我用 `normalizedTime` 阈值轮询。每帧读 `animator.GetCurrentAnimatorStateInfo(0).normalizedTime`（动画归一化进度，0 到 1），和 SO 里配的 `ArrowSpawnTime`（比如 0.4）比，越过就生成一支箭。我没用 Unity 官方的 Animation Event（在动画时间轴上打标记回调函数），是因为本项目刻意让所有战斗时机——近战命中窗口、连段输入窗口、刀光窗口、箭矢生成——全部统一用「读 normalizedTime 比阈值」这一套机制，集中写在 ScriptableObject 里，好统一管理、还能把判定逻辑抽成纯函数做单元测试（比如 ComboResolver）。Animation Event 的「所见即所得」是优点，但它把时机和数据散落进动画文件、跨角色不一致、不好测试，统一性的收益盖过了它。

**Q28：normalizedTime 这种单点触发，有什么容易踩的细节？**

两个。第一，单点必须配「只触发一次」的去重位：进度越过 0.4 之后，后续每帧都满足 `t >= 0.4`，不加一个 `bool _arrowSpawned` 守卫就会一帧生成一串箭。第二，要排除动画过渡期：两个动画 CrossFade 切换时，`GetCurrentAnimatorStateInfo(0)` 返回的「当前状态」可能还是旧状态，它的 normalizedTime 跟你要判断的新动画无关，可能恰好接近 1 导致一进来就误触发。我的处理是要么 `if (animator.IsInTransition(0)) return;`，要么校验 `info.shortNameHash == 目标状态hash`，确认「我真的在我以为的那个状态里」再读进度。蓄力放箭因为前面有个「满弓保持」态，我用的是 shortNameHash 校验。

**Q29 ★：你提到 Animator 状态名是"数据驱动"的，还说会"静默失败"，展开讲讲。**

我进入攻击/冲刺/放箭这些状态用的是代码 `CrossFadeInFixedTime(stateHash, ...)`，直接点名要播哪个状态，不需要在 Animator 里画进入连线。这个状态名我不写死，而是序列化成字符串字段（基类的 `_dashStateName`、SO 里的 `AnimationStateName`），让每个角色在 Inspector 填自己 Controller 里的真实节点名——所以同一份代码配不同名字就能驱动不同角色的动画。代价是 CrossFade 到一个不存在的状态时 Unity 不报错、只是不切动画，也就是「静默失败」。我真踩过这个坑：弓箭手冲刺时只平移没动画，因为 `_dashStateName` 还留着战士的默认值，弓箭手 Controller 里没那个状态。所以我在预 hash 时对空字符串加了 `GameLog.Warn` 告警，让配置错误尽量自曝。还有个相关的关键点：代码 CrossFade 只负责**进入**，**离开**它不管——所以每个这样进入的状态在 Animator 里仍然必须自己画**退出**连线（比如 Dash → Idle，勾 Has Exit Time），否则角色会定格在那个姿势。口诀是「进入靠代码、退出靠连线」。

## 十、性能

**Q30 ★：战斗逻辑每帧跑，你怎么保证不产生 GC？**

项目有条硬规则：Update、FixedUpdate 或任何每帧热路径里，禁止 new 分配、禁止 LINQ、禁止装箱。具体落实：状态对象全在 Awake 预实例化、运行时只切引用，不在运行时 new 状态；射线检测用 `RaycastNonAlloc` 写进预分配数组，近战命中用 `OverlapBoxNonAlloc` 写进预分配的 `Collider[]` 加复用的 HashSet 去重；Animator 状态名在 Awake 就 `StringToHash` 成 int 缓存，绝不每帧 StringToHash（字符串转 hash 既有开销又产生临时垃圾）；事件用 struct 不用 class，配合泛型 EventBus 按值传递、零装箱。一次性的离散分配是可以的，比如松手放箭时 Instantiate 一支箭——那是响应离散输入、不是每帧，没问题。我的验收标准是用 Unity Profiler 确认目标功能 GC Alloc 为 0。

**Q31：日志为什么不用 Debug.Log，你那个 GameLog 有什么讲究？**

`GameLog` 的三个方法都标了 `[Conditional("DEVELOPMENT_BUILD")]` 加 `[Conditional("UNITY_EDITOR")]`。Conditional 特性的效果是：在 Release 构建里，编译器会把整个调用点**连同参数一起在 IL 层删除**——包括字符串插值和它带来的装箱。也就是说 `GameLog.Info($"伤害{dmg}")` 在 Release 里不只是「不打印」，而是连那次字符串拼接和装箱都不存在了，零开销。如果直接用 `Debug.Log`，即使日志被关掉，参数的字符串拼接和装箱仍然会执行，白白产生 GC。这是一个很容易被忽视但在战斗这种高频场景里很值的优化。

## 十一、解耦与工程

**Q32 ★：你的模块之间怎么通信？为什么准心 UI 不直接被攻击代码调用？**

项目按域拆成多个 assembly definition，依赖是单向的：Core 是地基谁都不依赖，上层可以依赖下层，下层绝不依赖上层。跨模块通信只走 Core 里的 `EventBus<T>`，发布-订阅。准心 UI 在 `Game.Rendering`，攻击逻辑在 `Game.Character`，按规矩 gameplay 不许认识 UI，所以攻击代码不可能直接 `crosshair.SetActive`。解法是：蓄力状态 Enter/Exit 时 `Publish` 一个 `AimStateChangedEvent`，准心 UI 自己 `Subscribe` 这个事件、自己显隐。双方只认识 Core 里那个事件 struct，互不认识对方。这样做的扩展性体现在：我后来加相机偏移效果，又是同一个事件多挂一个订阅者，gameplay 完全没动。订阅和退订我严格成对放在 OnEnable/OnDisable，防止对象销毁后还被回调造成悬挂引用。

**Q33：EventBus 的事件为什么强制是 struct 而不是 class？**

为了零 GC。`EventBus<T>` 把 T 约束成 `struct, IGameEvent`，Publish 时按值传递，JIT 不装箱、不在堆上分配。如果事件是 class，每次 Publish 都要 new 一个堆对象，高频事件（比如每次命中都发 DamageReceivedEvent）会制造大量垃圾、触发 GC 卡顿。用 struct 让事件成为栈上的值，发布是零分配的，这在战斗这种事件密集的系统里很关键。

## 十二、开放性 / 拔高

**Q34：如果现在要再加第三个角色，比如法师，你的架构要改哪里？哪里不用动？**

不用动的是：整个 `PlayerControllerBase` 的移动/跳跃/冲刺/相机/状态机、伤害管线、EventBus、那四个共享状态。要加的是：一个 `MageController : PlayerControllerBase`，override `TryStartAttack` 路由它的施法逻辑；一两个法师专属攻击状态；法师的攻击数据 SO；以及它自己的 Animator Controller 和动画状态名（在 Inspector 填）。如果法术也是投射物，大概率能复用 Arrow 或仿照它再写一个平行类。这正是当初选 base/subclass 架构的回报——加角色主要是「加子类 + 配数据」，共享地基纹丝不动。唯一要警惕的是：如果法师的能力开始需要和别的角色自由组合（比如「会飞 + 会射」要能任意搭配），那继承会开始吃力，就该考虑往组件化/ability system 迁移了。

**Q35：你这个项目里哪个设计你觉得最得意，哪个如果重来你会改？**

最得意的是「攻击接缝 TryStartAttack」——它用一个虚方法就把「攻击优先级这个共性」和「攻击内容这个差异」干净地切开，让共享的地面状态完全不需要知道角色是谁，加角色零侵入。如果重来，我会更早地把转向插值写成严格帧率无关的 `1 - exp(-k·dt)` 形式，而不是沿用 `speed·deltaTime`；虽然现在差异可忽略，但作为战斗手感相关的基础设施，一开始就做对更好。另外瞄准那个「射线打到自己」的 bug，其实在设计阶段我就在注释和文档里写了「务必排除 Player 层」，但还是依赖了人工配置——更好的做法是从一开始就在代码里剔除自身，而不是等 bug 出现才补。这让我体会到：凡是依赖「外部配置永远正确」的设计，都应该尽量在代码里加一道兜底。

**Q36：怎么测试你的战斗逻辑？纯数学部分能不能脱离 Unity 测？**

能，而且这是我刻意的设计。我把可以纯函数化的逻辑都抽成了不依赖 MonoBehaviour/Animator 的静态类：伤害计算 `DamagePipeline.Resolve`、连段判定 `ComboResolver.Resolve` 都是「输入参数进、结果出」的纯函数。这些在 `Game.Combat.Tests` 里用 NUnit 写 EditMode 单元测试，不需要进 Play 模式、不需要场景，跑得飞快。比如伤害管线我可以直接断言「攻击力 50、防御 passthrough，结果就是 50」。需要 Unity 运行时环境的部分（投射物飞行、动画时序、输入）才在 Play 模式里人工验证。这种「纯逻辑下沉、可单测；表现/物理上浮、人工验」的分层，让战斗的核心数学有回归保障。

## 附：高频追问的"一句话武器库"（速记）

- **LookRotation 对齐的是 +Z**（Unity forward 约定），模型不是 +Z 朝前就乘一个本地修正旋转。
- **四元数无万向锁、可球面插值**；欧拉角直观但有 gimbal lock 和插值抽搐，只用于人类输入。
- **Slerp 匀速、Lerp 不匀速**；`speed·dt` 不是严格帧率无关，严格写法是 `1-exp(-k·dt)`。
- **点乘判方向/视野**（=|a||b|cosθ，归一化后即夹角余弦）；**叉乘判左右/求法线**（模=|a||b|sinθ）。
- **去 y 再归一化 = 投影到水平面**，取 yaw、滤掉俯仰。
- **相机平移 → 画面物体反向移动**（相机右移角色左移）。
- **方向必归一化**再乘速度，否则长度污染速度；比较用 **sqrMagnitude** 省 sqrt。
- **值快照**：飞行物在生成瞬间拍照存攻击者数据，命中时不回查（攻击者可能已销毁）。
- **发现命中可多样、结算路径必统一**（都汇入 ReceiveHit / DamagePipeline）。
- **收尾放 Exit()**——状态机唯一必经出口，覆盖所有离开路径。
- **单点触发必配去重 bool + 排除过渡帧**（IsInTransition / shortNameHash 校验）。
- **NonAlloc 系列**零 GC，但结果不排序、要自己求最近；**StringToHash 只在 Awake**。
- **RaycastNonAlloc 剔除自身**用 `transform.IsChildOf`（含自身），别只靠 LayerMask。
- **进入靠代码 CrossFade、退出靠 Animator 连线**，否则定格姿势。
