# 法师系统 — 模拟面试 Q&A（战斗系统 + 投射物架构 + 技能时序方向）

> 用途：以「米哈游战斗/Gameplay 程序面试官」的口吻，针对本项目**法师**功能提问，覆盖抽象复用、双阶段技能状态机、tap/hold 输入边沿、持续伤害 DoT、落点瞄准、物理穿透、贴地表现、动画时序、性能与解耦。
> 答案以**面试者第一人称、段落作答**，做到「先答要点 → 给原因 → 必要时讲取舍 / 追问应对」。
> 这份与 `Archer_Interview_QA.md` 互补：弓箭手那份侧重 3D 数学与瞄准，本份侧重**架构抽象、技能时序、输入设计、DoT**。共性知识（四元数、点乘叉乘、Slerp、EventBus、零 GC 通则）不再重复，只在法师独有的语境里展开。
> 建议用法：先盖住答案自己说一遍，再对照。带 ★ 的是高频追问点，重点准备。

## 一、抽象与复用：投射物基类（本角色的标志性设计）

#### **Q1 ⭐：你现在有箭、火球、陨石三种投射物，代码上你是怎么组织的？为什么这么做？**

我抽了一个抽象基类 `ProjectileBase`，把所有飞行投射物的公共骨架放进去：`Init` 注入伤害快照、初速度、忽略施法者碰撞、定向、计时；`OnCollisionEnter` 统一处理「同阵营穿过 / 敌方结算一次 `IDamageable.ReceiveHit` / 命中后销毁」。子类只重写一个扩展点 `OnImpact`，决定「命中后、销毁前」的额外表现——箭啥都不加（空实现）、火球加爆炸特效 + 点燃、陨石加爆炸特效。`Arrow`、`Fireball`、`NovaFireball` 都派生自它。这其实是**模板方法模式**：父类把「不变的流程骨架」固化下来，把「会变的那一步」开成虚方法外包给子类。

我之所以抽，是因为在写第三个投射物（陨石）之前，我发现箭和火球的代码有大约八成是逐字重复的——同阵营过滤、构造 `DamageRequest`、`ReceiveHit`、防重复结算、超时自毁，全都一样。重复代码的真正危害不是「行数多」，而是「改一处要同步多处、迟早漏一处出 bug」：比如我后来修一个穿地 bug，要给所有投射物开连续碰撞检测，抽了基类我只在 `Init` 里改一行，三种投射物全好了；没抽的话我要改三个文件，还很可能漏掉将来第四个。

#### **Q2 ⭐：什么时候该抽基类，什么时候不该？你判断的依据是什么？**

我的经验法则是「**Rule of Three（事不过三）**」：出现第一次就实现它，第二次先忍着、容忍少量重复，等到**第三次**重复出现、并且我能看清「哪些是真正不变的共性、哪些是会变的差异」时，再抽象。过早抽象比重复更危险——你会基于只有一两个样本的猜测去设计接口，结果第三个用例一来就不匹配，抽象反而成了枷锁。

我这次正好卡在这个点：箭和火球是前两次，本来可以再忍，但陨石是明确要做的第三个，而且三者的差异已经收敛得很清楚——**唯一的变化点就是「命中后做什么」**，其余完全一致。差异点这么单一、这么明确，正是抽模板方法的理想时机，所以我趁写第三个之前一次性抽掉，避免又复制一份。如果三个投射物的差异是发散的、每个都不一样（飞行方式、命中判定、销毁条件各不同），我反而会犹豫，可能改用组合而不是继承。

#### **Q3：`OnImpact` 这个扩展点的签名你是怎么设计的？为什么传那几个参数？**

签名是 `protected virtual void OnImpact(Collision collision, IDamageable target, Vector3 hitPoint, bool damaged)`。我刻意把基类已经算好的信息都传给子类，避免子类重复计算或重复查询：`collision` 给需要碰撞法线的子类（比如想让爆炸贴合斜面）；`target` 是基类已经 `GetComponentInParent<IDamageable>` 拿到的命中目标，可能为 null（撞的是环境）；`hitPoint` 是接触点，爆炸特效就生成在这；`damaged` 是个布尔，告诉子类「本次有没有真的结算伤害」。这个 `damaged` 很关键——火球的逻辑是「撞到任何东西都炸（表现），但只有打到敌人才点燃（伤害效果）」，所以它用 `damaged` 区分：爆炸无条件放，`ApplyBurn` 只在 `damaged` 为真时调。把「命中环境也走到 OnImpact」这点设计进去，就是为了支持「撞地也要有爆炸表现」这种需求。

#### **Q4 ⭐：你把箭、火球的逻辑挪进基类时，Prefab 上原来配好的参数（比如箭的模型修正角）会不会丢？**

不会，这是我敢做这次重构的前提。Unity 的序列化是**按字段名**匹配的，而且能跨类继承层级——只要字段名不变，我把 `_modelForwardOffsetEuler`、`_maxLifetime` 这些字段从 `Arrow` 上移到父类 `ProjectileBase`，Prefab 里序列化的值依然能按名字找回来，箭的 (90,0,0) 修正角不会丢。同理，重命名脚本时我会把 `.cs` 和它的 `.meta` 一起 `git mv`，因为 Prefab 引用的是 `.meta` 里的 GUID，GUID 不变引用就不断。理解「序列化按字段名、引用按 GUID」这两条，是放心做大重构而不破坏美术已配资产的关键；不懂这个的人重构完往往一堆 Prefab 引用丢失、字段清零，只能手动重配。

## 二、状态机：引导 → 施法的双阶段技能

#### **Q5 ⭐：法师重击是「长按引导、松手放陨石」，这种有蓄力/引导过程的技能，状态机你怎么设计？**

我用**一个状态**承载两个阶段，而不是拆成两个状态。`PlayerWizardHeavyState` 里有一个 `_released` 布尔标志：进入时 `_released = false`，处于引导阶段——原地定身、脚下生成光圈、地面生成落点圆框、每帧用屏幕中心射线更新落点、转向落点；检测到松手（`!IsAttackHeld`）就调 `BeginRelease()` 把 `_released` 置真，进入施法阶段——收起引导视觉、（可选）播施法动画、靠计时器在前摇结束时生成陨石、再到总时长结束时切回移动状态。

之所以用单状态 + 标志而不是「引导态 + 施法态」两个状态，是因为这两个阶段共享大量上下文：落点 `_aimPoint`、光圈和圆框的实例引用、定身逻辑。拆成两个状态的话，这些数据要么重复存、要么得在状态间传递，反而更绕。当一个技能的几个阶段是「同一段连续过程、共享同一批数据、只是行为分支」时，单状态 + 阶段标志更内聚；只有当各阶段差异巨大、几乎不共享数据时，我才会拆成独立状态。

#### **Q6 ⭐：施法阶段「松手 0.3 秒后生成陨石、0.6 秒后结束」，这个时序你是用动画进度还是计时器驱动的？为什么和弓箭手不一样？**

我用**计时器**驱动：`_releaseTimer += Time.deltaTime`，到 `MeteorSpawnDelay` 生成陨石、到 `ReleaseDuration` 切回移动。这跟弓箭手「读 `normalizedTime` 比阈值」是两套不同的驱动方式，原因是**法师的施法动画是可选的**——`ChannelStateName` / `ReleaseStateName` 在 SO 里允许留空，留空就不 CrossFade。如果我也去读 `normalizedTime`，那当美术还没做施法动画、状态名为空时，当前播的是 Idle，它的 `normalizedTime` 跟我的施法节奏毫无关系，时序就全乱了。所以凡是「节奏不应该依赖某段动画是否存在」的逻辑，我就用独立计时器；只有「触发点本来就该精确贴合某一动画帧」（比如箭要在拉弓动画 0.4 处离弦）才用 `normalizedTime`。这是一个「时序的权威应该是谁」的判断：是动画，还是逻辑时钟。

#### **Q7：引导期间角色「定身」，这个你是怎么实现的？为什么不是简单地不调 Move？**

我实现了一个 `HandleRooted()`：它仍然调 `CharacterController.Move`，但只施加垂直方向的贴地速度（`Vector3.up * VerticalVelocity`），完全不响应水平移动输入。之所以还要调 Move 而不是干脆什么都不做，是因为 `CharacterController` 需要持续的向下微小速度来保持和地面的接触判定、处理脚下的细微高度变化；如果引导期间完全不 Move，离地判定、斜坡贴合这些会出问题。所以「定身」的准确含义是「锁住水平、保留垂直贴地」，而不是「冻结整个角色」。同时我还在引导期每帧调 `FaceTarget` 让角色平滑转向落点方向（只 yaw），这样松手放陨石时朝向是对的。

#### **Q8：引导时生成的光圈和落点圆框，怎么保证它们不会泄漏（比如技能被打断后还留在场景里）？**

我把清理放在状态的 `Exit()` 里。`ChangeState` 保证「离开任何状态必然经过它的 `Exit()`」，所以不管这个重击是正常放完、还是将来新增一条「被击打断」的退出路径，`Exit()` 里 `Destroy(_ring)` 和 `Destroy(_indicator)` 都一定会执行，视觉不会残留。这跟弓箭手把准心隐藏放 `Exit()` 是同一个原则——**把收尾绑定在「唯一必经的出口」上**。另外在 `BeginRelease()`（引导转施法）时我也会提前销毁圆框和光圈，因为它们只属于引导阶段；`Exit()` 里再判一次 null 兜底，双保险。

## 三、输入：tap/hold 边沿门控（含 bug 复盘）

#### **Q9 ⭐：同一个按键，点按是火球、长按是陨石，你怎么区分这两个意图？**

最初我用「累计按住时长 + 阈值」：每帧只要 `IsAttackHeld`（轮询 `IsPressed()`）为真，就 `_attackHeldTime += Time.deltaTime`，越过 `TapThreshold`（比如 0.15 秒）立刻切到陨石引导态（不等松手，为了「按到点马上起手」的即时感）；在越阈前松手就判定为点按、放火球。思路跟弓箭手蓄力一致。但这个最初版本有一个我后来排查出来的 bug，引出了我对「电平 vs 边沿」的理解——要不要我展开讲？

#### **Q10 ★：（追问）展开讲那个 bug。**

现象是：上一发是重击之后，紧接着的那一发我明明只是点按，却又触发了重击。我排查后定位到根因：旧逻辑判定「长按」的方式是**每帧只要按键处于按下电平（level）就累加时长**，它**没有「新按下边沿（edge）」的概念**——分不清「这一帧是我刚按下的」还是「我从上一个动作一直按着没松手」。

而重击的输入天然会「跨状态边界还按着」：你是按住触发重击的；引导和施法期间地面状态不跑、`TryStartAttack` 不被调用、累加器不动；等陨石放完回到地面状态时，如果按键还按着，或者你在施法尾段就提前重新按下、这次按压「骑跨」了状态切换的边界——地面状态一恢复就对着这颗「残留的按下电平」从头累加，很快又过阈值，于是又判成重击。本质是：**回到可攻击状态时按键已是按下态，被误当成一次新的长按。**

#### **Q11 ⭐：（追问）你怎么修的？为什么这样修是对的？**

我把判定从「电平触发」改成「**边沿触发**」：用 Input System 的 `WasPressedThisFrame()` 检测「本帧刚按下」的上升沿，只有出现上升沿才开始跟踪这一次输入（置一个 `_attackTracking` 标志、把累计清零）；`_attackTracking` 为假时直接返回，不累加、不触发任何攻击。这样一颗从上个重击残留下来的「按住」，因为没有新的上升沿，就不会启动新的蓄力计时，bug 根除。触发任一攻击后我用一个 `EndAttackTracking()` 统一清掉跟踪标志、累计时长和攻击缓冲，防止下一发误触发。

这样改还白送了**亚帧点按容错**：如果按下和松开发生在同一帧，`WasPressedThisFrame` 这一帧仍返回真（它记录的是「这帧发生过按下」），而此时 `IsPressed` 已经是假——于是直接走点按分支放火球，连原来用 `AttackBufferCounter` 兜亚帧的依赖都可以摘掉。通用结论是：**「这一刻是不是按着」（电平）和「这一刻是不是刚按下」（边沿）是两回事**，凡是「一次性意图的开始」（开始蓄力、触发跳跃）都应该用边沿，凡是「持续状态」（是否仍在蓄力、是否仍按住移动）才用电平。我最初混用了，把「开始一次长按」错误地建立在电平上，才会被残留电平骗到。

#### **Q12：这个修复你只改了法师吗？**

不止。弓箭手的蓄力路由跟法师是逐字平行的同一套逻辑，所以是同一个 bug，我同步改了 `ArcherController`。我还把 `WasPressedThisFrame` 这个上升沿封装在基类 `PlayerControllerBase` 上暴露成 `AttackPressedThisFrame` 属性，两个子类共用——既然这是「攻击输入的读法」这种共性，就该放在基类，避免两处各写一遍又不同步。这也呼应了前面投射物抽基类的思路：**发现两个角色在做同一件事且容易不同步时，就上移到共享处。**

#### **Q13：键位上为什么重击用「长按左键」而不是「按住右键」？**

这是被现有输入配置约束出来的取舍。我查了 `InputSystem_Actions`，右键已经绑给了 Dash（和左 Shift 一起），左键是 Attack。如果硬把重击塞给「按住右键」，就会和冲刺撞键，得改输入资产、还要处理「按住右键到底是冲刺还是蓄力」的歧义。而「长按左键蓄力、点按左键普攻」复用的是同一个攻击键的 tap/hold 路由，零输入资产改动，也和弓箭手保持一致的操作心智。所以我选了长按左键。面试里我会强调：输入方案不是凭空设计的，要先看清现有键位占用和冲突，能不动配置就不动。

## 四、持续伤害 DoT（点燃）

#### **Q14 ⭐：火球命中后目标会「燃烧」持续掉血，这种 DoT（Damage over Time）你怎么实现的？**

我做了一个 `BurnStatus` 组件，本质是「**挂在目标身上的计时循环**」。火球命中、确认造成伤害后，对目标 `GetComponent<HealthComponent>` 找到血量组件，再 `GetComponent`/`AddComponent` 一个 `BurnStatus`，调它的 `Apply(...)` 传入每跳伤害、间隔、持续时长、点燃特效。`BurnStatus` 在 `Awake` 缓存目标的 `IDamageable`，`Update` 里用一个计时器到点就构造 `DamageRequest` 走 `ReceiveHit` 结算一跳伤害，总时长到了就 `Stop()` 销毁特效和自身。关键点是 **DoT 的每一跳都复用同一条伤害管线 `ReceiveHit`**，而不是另写一套扣血逻辑——这样防御、阵营、伤害类型、事件广播全都自动一致。

#### **Q15：重复点燃（还在烧的时候又被一发火球打中）你怎么处理？**

我用「**刷新而非叠加**」：`Apply` 时如果目标已经有 `BurnStatus`，就只刷新持续时长，不再生成第二个特效、不叠加第二条独立的燃烧。这是最简单也最常见的 DoT 续期策略。当然业界还有别的方案——独立多层叠加（每层各自计时，暗黑那种）、按层数提升每跳伤害（DK 疾病那种）——具体选哪种是数值策划的设计，但我把接口留成「Apply 即续期」，要扩展成叠层只是在 `Apply` 里加层数字段的事。

#### **Q16：你提到 DoT 走 ReceiveHit，那它会不会每跳都触发受击表现（比如受击闪白、硬直）？这合理吗？**

会，这正是我目前实现的一个**已知局限**，我在文档里标注了。因为每跳都走完整的 `ReceiveHit`，订阅了 `DamageReceivedEvent` 的受击反馈组件每跳都会响应一次，可能导致燃烧时角色一直闪白。直接命中和 DoT 在「是否该触发受击表现」上需求是不同的：直接命中该有顿挫反馈，DoT 通常只该飘个跳字、不该硬直。更好的做法是在 `DamageRequest` 上加一个「静默 / DoT 标记」，让受击表现订阅者据此决定要不要响应。我现在没做是因为受击反馈本身还在早期，先保证伤害正确、表现层留作下一步——但我能清楚说出它现在的边界和将来怎么补，这比假装它完美更重要。

#### **Q17：为什么 `BurnStatus` 放在 `Game.Combat` 而不是法师所在的 `Game.Character`？**

因为 DoT 是一个**通用战斗概念**，不是法师专属。放在 `Game.Combat` 后，它既能被火球挂在敌人身上，将来也能被陷阱、毒池、敌人技能挂在玩家身上——任何带 `IDamageable` 的对象都能燃烧。如果我把它放进 `Game.Character`，就违反了依赖方向（Combat 不能依赖 Character），而且把一个通用机制锁死在了角色模块里。判断一段逻辑该放哪个程序集，我的依据是「它服务的概念属于哪一层、会被谁复用」，而不是「我现在是在做哪个角色时写的它」。

## 五、落点瞄准与陨石几何（3D 数学）

#### **Q18 ⭐：引导时鼠标控制陨石落点，这个「屏幕中心射到地面」的落点你是怎么算的？和弓箭手瞄准有什么不同？**

我用 `camera.ViewportPointToRay(0.5, 0.5)` 取屏幕正中央（准心）射出的射线，做 `Physics.RaycastNonAlloc` 求与**地面**的命中点作为落点；没命中就取射线在 `AimMaxDistance` 处的远点、并把它的 y 拉到角色脚高。和弓箭手不同的关键有两点：第一，法师的 `AimMask` 我只勾**地面层**，因为我要的是「打到地面的那个点」，不是打到敌人——陨石是范围落点技能，瞄的是地，不是单体目标。第二，我对落点做了**水平距离上限钳制**：把「落点 - 角色」的向量去掉 y 投影到水平面，如果水平距离超过 `AimMaxDistance` 就按比例缩回来，防止落点被甩到很远。同样地，遍历射线命中时我用 `IsChildOf(_player.transform)` 跳过自身，不依赖 LayerMask 配置永远正确。

#### **Q19 ⭐：陨石要从落点的「斜上方天空」砸下来，这个生成点和方向你在数学上怎么构造？**

我要的是一条斜入射的弹道，所以生成点不能在落点正上方（那是垂直落下），得在「正上方再往后退一点」。我先算水平方向 `horizDir = (落点 - 角色).y 清零后归一化`，作为「角色看向落点」的水平朝向。生成点就是 `落点 + Vector3.up * SpawnHeight - horizDir * SpawnHorizontalBack`——即落点上方 `SpawnHeight` 的高度，再沿水平朝向**反方向**后退 `SpawnHorizontalBack`，这样生成点位于落点的斜后上方。陨石的飞行方向就是 `(落点 - 生成点).normalized`，乘以速度作为初速度，并用 `Quaternion.LookRotation(dir)` 让模型朝向飞行方向。把高度和后退量都做成 SO 参数，策划可以调出「几乎垂直砸」还是「很斜地划过来」的不同观感。退化兜底也做了：水平方向接近零向量（角色就站在落点上）时用角色 forward，方向接近零时用 `Vector3.down`，避免 LookRotation 收到零向量。

#### **Q20 ★：落点圆框和爆炸特效本来是「竖着」的，你怎么让它们贴合地面？讲讲旋转。**

这些环形/平面特效大多是在**竖直平面（XY 平面）**建模的，直接以 `Quaternion.identity` 生成就立在地上像一面墙。要平铺到地面（XZ 平面），绕世界/本地 X 轴转 90° 即可：`Quaternion.Euler(90, 0, 0)`。我把这个旋转角做成可配置字段——落点圆框走 SO 的 `IndicatorRotationEuler`、陨石爆炸走 `NovaFireball` 上的 `_explosionRotationEuler`，默认都是 (90,0,0)，美术觉得还歪就自己在 Inspector 调，不用改代码。我没有把爆炸的旋转去复用落点 SO 的字段，因为 `NovaFireball` 作为 `Game.Combat` 的通用投射物，`Init` 只注入伤害快照，**拿不到也不该认识**那份属于法师状态的 SO；给投射物自己一个旋转字段，它对「谁发射的」零依赖，将来别的技能复用陨石也不受影响。如果要更进一步贴合斜坡，可以用碰撞法线 `Quaternion.FromToRotation(Vector3.up, contactNormal)` 再乘上这个修正角，让爆炸跟着斜面倾斜——平地两者等价，斜坡上前者才贴。

## 六、物理：穿透与连续碰撞检测

#### **Q21 ⭐：陨石高速下落时会「穿过地面」不爆炸，你怎么排查和解决的？**

现象很典型：陨石撞到斜坡或角色能正确爆炸，但落到平地会直接穿下去。我先做对比——能炸的都是有一定厚度或正对飞行方向的碰撞体，穿过去的是薄地面，于是判断是**隧穿（tunneling）**：默认的离散碰撞检测每个物理步只检查「这一刻有没有重叠」，陨石速度快，两个物理步之间它就整个跨过了薄薄的地面碰撞体，引擎那一刻没检测到重叠，就漏掉了。

解决我在 `ProjectileBase.Init` 里把刚体的 `collisionDetectionMode` 设成 `ContinuousDynamic`——连续碰撞检测会用**扫掠（sweep）**的方式检查这一步移动路径上有没有碰撞，而不只是看终点那一刻，所以能抓住快速穿过薄物体的命中。因为我设在基类，所有投射物自动获得这个保护。前提是地面得真的有 Collider；如果地面太薄又没碰撞体，那是资产问题，不是代码能兜的。

#### **Q22：连续碰撞检测不是有性能开销吗？无脑全开合适吗？**

不合适，但用在投射物上是划算的。连续检测确实比离散贵，因为要做扫掠。工业上的取舍是：**只给「高速且不可穿透很重要」的物体开**——子弹、箭、陨石这种；而大量低速或可以容忍偶尔穿一下的物体保持离散。我的投射物数量少、生命周期短、又恰恰是最怕穿模的，所以开 `ContinuousDynamic` 收益远大于开销。如果将来有海量高速弹幕（弹幕类游戏），我就不会每颗都开物理连续检测了，会转向「自己每帧从上一帧位置到当前位置做一条 Raycast 补检」的伪投射物方案，性能可控得多。

## 七、动画与时序

#### **Q23 ⭐：冲刺是个「前滚翻」，你说原来角色会先瞬移、动画再滞后，怎么对齐的？**

现象是按下冲刺后角色立刻平移过去，翻滚动画才慢半拍跟上，位移和动作对不上。根因是位移和动画是两条独立时间线：冲刺状态一进入就开始全速施加位移，而翻滚动画的「滚起来」那一刻在动画稍靠后的位置。我的修法是给位移加一个**起始延迟**，让它和翻滚的发力点对齐：加了一个可配置的 `_dashMoveDelay`，在 `PlayerDashState` 里，`elapsed < DashMoveDelay` 这段时间水平速度给 0（只有动画在动、人不冲出去），过了延迟才给全速 `DashSpeed`；同时把状态结束条件改成 `elapsed >= DashMoveDelay + DashDuration`，保证延迟不会吃掉冲刺的有效位移时长。延迟量做成 Inspector 可调，因为不同角色的翻滚动画发力点不同，这是手感参数该留给数据层。这本质是「让两条时间线的关键帧相位对齐」。

#### **Q24：法师边走边放火球，火球在动画哪一刻生成？怎么保证不会一帧生成一串？**

火球在普攻动画的 `ArrowSpawnTime`（SO 配的归一化进度）那一刻生成，做法跟弓箭手一致：每帧读 `GetCurrentAnimatorStateInfo(0).normalizedTime` 和阈值比，越过就生成。防止「一帧一串」用一个 `bool` 守卫——越过阈值后置真，本次播放内不再生成；并校验 `shortNameHash` 确认确实在普攻状态里、排除 CrossFade 过渡帧的误触发。这套「单点触发 = 进度阈值 + 去重 bool + 排除过渡帧」是项目里所有单点战斗事件的统一范式（近战命中、连段窗口、箭/火球生成都用它）。

#### **Q25：法师引导/施法动画你允许「留空」，代码上怎么优雅处理动画可有可无？**

我在 SO 里把 `ChannelStateName` / `ReleaseStateName` 设为可空字符串，`Awake` 预 hash 时空串就存 0，且**不告警**（区别于普攻段名为空时会 `GameLog.Warn`，因为普攻没动画是配置错误，而引导/施法动画是「可选润色」）。运行时状态里 `if (hash != 0) CrossFade(...)`，为 0 就跳过、不切动画但逻辑照常用计时器推进。这样美术没做施法动画时整套重击逻辑依然能跑（陨石照样生成、伤害照样结算），动画做好了填上名字就自动接上。把「可选表现」和「必需逻辑」解耦，是让功能能分阶段交付、不被美术进度卡住的关键。

## 八、性能、解耦与工程

#### **Q26：法师这些新功能里，每帧跑的逻辑你怎么守住零 GC？**

延续项目通则，落到法师的具体点：落点射线用 `Physics.RaycastNonAlloc` 写进预分配的 `RaycastHit[16]`，不每帧 new 数组；引导态每帧的落点更新、转向都只做向量运算和结构体传递，不产生堆分配；动画状态名（普攻段名、引导/施法名）在 `Awake` 就 `StringToHash` 缓存成 int，绝不每帧转字符串；伤害事件是 struct 走泛型 `EventBus`，零装箱。而像生成火球、生成陨石、给目标 `AddComponent<BurnStatus>` 这种是**响应离散输入/事件的一次性分配**，不在每帧热路径上，是允许的。验收还是看 Profiler 的 GC Alloc 是否为 0。

#### **Q27 ★：你提到运行时 Console 一直刷「Can't remove Light because UniversalAdditionalLightData depends on it」，这是什么原因？严重吗？**

不是致命错误，但会刷屏、且说明有段代码在反复做无效操作。根因是第三方特效脚本 `ETFXLightFade` 在灯光淡出后调 `Destroy(light)` 想删掉 Light 组件，但 URP 下每个 Light 都带一个 `[RequireComponent]` 的 `UniversalAdditionalLightData`——它依赖 Light，所以引擎拒绝单独删 Light，每帧重试就每帧报一次。修法是**先删依赖方、再删被依赖方**：先 `Destroy` 掉 `UniversalAdditionalLightData`，再 `Destroy` 掉 Light，然后把这个脚本 `enabled = false` 停掉避免继续每帧重试。这题考的是对 Unity `[RequireComponent]` 依赖约束的理解——有依赖关系的组件，销毁顺序必须从依赖方往被依赖方走。

#### **Q28：你提到把脚本按类型重新分了文件夹，这种纯整理对架构有影响吗？做的时候要注意什么？**

对编译和运行**零影响**，因为 C# 的命名空间和程序集（asmdef）边界跟磁盘文件夹无关——`Game.Combat` 这个程序集是按 asmdef 的位置和引用决定的，不是按子文件夹。我把 Combat 拆成 Damage / Definitions / Combo / Projectiles / Melee / Status / Events 等子文件夹，纯粹是为了人找文件方便。要注意的唯一一点是：移动 `.cs` 时必须**连同它的 `.meta` 一起移动**（我用 `git mv` 成对移动），因为 `.meta` 存着 GUID，Prefab/场景/asmdef 的引用都认 GUID；只移 `.cs` 不移 `.meta`，Unity 会重新生成一个新 GUID，所有引用就断了。理解「文件夹只是给人看的、引用靠 GUID」，这种整理就很安全。

## 九、开放性 / 拔高

#### **Q29 ⭐：现在箭、火球、陨石都继承 ProjectileBase，如果来一个「会拐弯追踪的导弹」或「命中后分裂成多发」的投射物，你的架构扛得住吗？**

追踪导弹基本扛得住：飞行中转向是「每帧修改速度方向」的逻辑，我可以在基类开一个 `protected virtual void OnFlightUpdate()`（默认空）的扩展点，让追踪子类在 `FixedUpdate` 里重写它去把速度转向目标，骨架不动。命中分裂也扛得住：分裂是「命中后的额外行为」，正好落在现有的 `OnImpact` 扩展点里，子类在那 `Instantiate` 几个小投射物即可。但我会诚实指出边界：如果新投射物的**命中判定方式**本身变了（比如范围 AOE 投射物要的是 `OverlapSphere` 而不是 `OnCollisionEnter`），那它和现有基类「靠碰撞回调命中」的骨架就不一致了，硬塞进同一个基类会让骨架长出一堆 if 分支。那时我会考虑：要么把「命中检测」也抽成一个可替换的策略（组合），要么让 AOE 走另一条独立基类。**继承适合「流程相同、某步不同」，一旦流程本身分叉，就该考虑组合。**

#### **Q30：法师的重击（引导→落点→范围）和弓箭手的蓄力（拉弓→瞄准→直射）有不少相似，你有没有想过把「蓄力/引导」也抽象成共享的东西？**

想过，但我**现在没抽**，这是个克制的决定。表面上两者都是「按住起手、松手放招」，但细看差异是发散的：弓箭手瞄的是单体方向（含俯仰）、放的是直线箭、蓄力期间能移动并改变镜头构图；法师瞄的是地面落点（范围）、放的是斜落陨石、引导期间完全定身。它们共享的其实只有最浅的「tap/hold 输入路由」——而这部分我**已经**抽到基类了（`AttackPressedThisFrame` + 各自的 `TryStartAttack`）。再往上把整个「蓄力技能」抽成模板，收益不大、还得设计一堆扩展点去容纳两者发散的中段行为，属于过早抽象。我的判断标准始终是：**抽真正稳定的共性（输入读法、伤害管线、投射物骨架），放过会发散的差异（每个技能的中段表现）。** 等出现第三个、第四个蓄力技能、共性收敛清楚了，再考虑抽「蓄力技能基类」也不迟。

#### **Q31：回顾整个法师的开发，你觉得哪个决定最关键，哪个如果重来会改？**

最关键的是「**在写第三个投射物之前抽 `ProjectileBase`**」——这个时机抓得好，既没过早（已经有两个样本看清了共性）也没过晚（赶在复制第三份之前），让后续修穿地、加旋转字段都变成改一处而非改三处。另一个我很满意的是把 tap/hold 的 bug 从「电平」纠正到「边沿」，它让我真正想清楚了输入设计里「持续状态」和「一次性意图」必须用不同读法。

如果重来，我会**一开始就用边沿触发**，而不是先写电平、踩了「重击后误触发」的 bug 再回头改——这个 bug 其实在我设计「按住触发、且触发后会离开地面状态一段时间」的那一刻就埋下了，只要当时多问一句「回到地面态时按键可能还按着吗」，就能提前避免。这让我体会到：**凡是「会让输入跨越状态边界」的设计，都要立刻问一句『状态切回来时输入残留会不会被误读』。** 另外 DoT 每跳触发受击表现这个局限，我会更早地在 `DamageRequest` 上预留「静默标记」，而不是留到表现层做完再补。

## 附：法师专属「一句话武器库」（速记）

- **Rule of Three**：第一次实现、第二次容忍、第三次（共性收敛清楚时）才抽象；过早抽象比重复更危险。
- **模板方法**：父类固化流程骨架（`Init`/`OnCollisionEnter`），开一个虚方法（`OnImpact`）外包变化的那一步给子类。
- **序列化按字段名、引用按 GUID**：字段移到父类不丢 Prefab 值；脚本改名连 `.meta` 一起 `git mv`。
- **双阶段技能 = 单状态 + 阶段标志**（`_released`），共享上下文时比拆两个状态更内聚。
- **时序权威是谁**：依赖某动画帧 → 读 `normalizedTime`；不该依赖动画是否存在 → 用独立计时器。
- **定身 = 锁水平、留垂直贴地**（仍调 Move 给向下微速），不是冻结整个角色。
- **电平 vs 边沿**：「是否按着」用 `IsPressed`（电平）；「是否刚按下」用 `WasPressedThisFrame`（边沿）。一次性意图的开始必须用边沿，否则会被跨状态残留电平误触发。
- **DoT = 挂在目标上的计时循环组件**，每跳复用同一条 `ReceiveHit` 管线；重复施加默认「刷新时长而非叠加」。
- **通用战斗概念放 Game.Combat**（BurnStatus），不锁进角色模块；判断归属看「服务哪一层、被谁复用」。
- **范围落点瞄准**：屏幕中心射线只打**地面层**，命中点即落点 + 水平距离钳制；`IsChildOf` 剔除自身。
- **斜入射弹道**：生成点 = 落点 + up·高度 − 水平朝向·后退量；方向 = 落点 − 生成点。
- **竖直特效贴地**：绕 X 转 90°（`Euler(90,0,0)`）把 XY 平面掰到 XZ；要贴斜坡再乘 `FromToRotation(up, normal)`。
- **隧穿**：高速投射物穿薄地面 → 刚体 `collisionDetectionMode = ContinuousDynamic`（扫掠检测），只给少量高速物体开。
- **相位对齐**：冲刺先瞬移后动画 → 给位移加起始延迟对齐发力帧；延迟做成数据。
- **可选动画**：状态名留空 → hash 为 0 → 跳过 CrossFade 但逻辑照跑，让功能不被美术进度卡住。
- **URP 删 Light**：先删依赖方 `UniversalAdditionalLightData` 再删 Light，并停用脚本停止每帧重试。
- **继承 vs 组合的边界**：流程相同、某步不同 → 继承/模板方法；流程本身分叉（命中方式都变了）→ 组合或独立基类。
