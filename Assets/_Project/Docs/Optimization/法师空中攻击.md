# 法师空中普攻 —— 功能实现解析

> **本文定位**：解析"让法师在空中也能释放火球普攻"这个功能的从需求到落地的完整过程。
> 区别于 `Combat_And_Movement_Optimizations.md`（打磨已有功能的手感），本功能是在现有攻击系统上**新增一条入口（空中起手）并复用已有攻击态**——重点在"如何用最小代价、最小重复，给状态机加一条新分支"。
> 适合作为"状态机扩展 / 架构复用 / 工程取舍"方向的面试素材。

---

## 1. 需求拆解 + 涉及的基础知识与原理

### 1.1 需求拆解

一句话需求：**法师在空中（跳跃/下落）时，点左键也能放火球普攻**，并播放专门的空中施法动画 `JumpAttack_MagicWand`。

拆成几个可独立验证的小点：

| # | 子需求 | 含义 |
|---|---|---|
| A | **空中能触发攻击** | 当前攻击只能在地面起手——空中按左键没反应。要给"空中"也开一条攻击入口。 |
| B | **空中播专属动画** | 地面普攻播 `Attack01_MagicWand`，空中要播 `JumpAttack_MagicWand`（带跳跃姿态）。 |
| C | **空中要继续受重力** | 攻击期间不能"凭空悬停"——边下落边施法才自然。 |
| D | **空中不混入重击** | 法师长按是"陨石重击"（地面落点技能），空中放陨石没意义，空中只保留点按火球。 |
| E | **复用发射逻辑** | 火球生成、屏幕中心瞄准、阵营、射速冷却，空中和地面应当一致，不重写。 |
| F | **结束后正确归位** | 空中攻击结束，若还在空中 → 回空中态；若已落地 → 回地面态。 |

### 1.2 涉及的基础知识与原理

**① 手写有限状态机（FSM）**
本项目角色用的是**纯 C# 状态对象**（不是 MonoBehaviour），在 `Awake` 里预先 new 好、运行时只切引用（零 GC）。`ChangeState = Exit() → 换引用 → Enter()`，**无条件执行**。
- 关键点：状态的"进入逻辑"全在 `Enter()`，"每帧逻辑"在 `Update()`，"收尾"在 `Exit()`。
- 状态之间不能传参（`ChangeState` 不带参数）——要在进入时区分情景，只能在 `Enter()` 里**读当前上下文**（如"现在接没接地"）来分支。这是本功能"用一个态同时服务地面/空中"的技术前提。

**② 状态转移的"优先级链"**
地面态每帧按固定优先级检查能不能切走：`Dash → Attack → Jump → 离地 → 滑坡`。空中态原本只检查"落地"。给空中加攻击，本质就是在空中态的检查链里**插一个新优先级位**，且要想清楚它和"土狼跳（Coyote Jump）"谁先谁后。

**③ 抽象 seam（接缝）模式**
共享基类 `PlayerControllerBase` 不知道每个角色具体怎么攻击，只暴露一个虚方法钩子 `TryStartAttack()`：基类返回 `false`（不攻击），各角色子类重写。共享的地面态调用这个钩子，于是"攻击优先级"写一遍、各角色攻击语义各异。
- 本功能要新增一条空中入口，最自然的做法就是**再开一个平行的 seam** `TryStartAirAttack()`，而不是把空中判断硬塞进某个具体角色。

**④ 动画的"代码驱动进入 + 连线驱动退出"**
本项目攻击/冲刺动画用 `Animator.CrossFadeInFixedTime(stateHash, ...)` 由代码**直接点名目标状态**进入——所以 Animator 里**不需要画进入连线**，目标状态名是数据驱动、`Awake` 里 `StringToHash` 预哈希一次。
- 但**离开**不是代码做的：代码只切 FSM 状态，不会 CrossFade 回去。所以每个这样进入的 Animator 状态**必须自带"退出连线"**（如 `JumpAttack_MagicWand → Idle_Bow`，带 Has Exit Time），否则角色会卡在施法 pose。

**⑤ 分段重力（Split-phase Gravity）**
空中移动用半隐式欧拉积分 `v += g * multiplier * dt`，上升用 `GravityMultiplier`、下落用更大的 `FallGravityMultiplier`（下落更"脆"）。空中攻击要"边落边放"，就得在攻击态里复用这同一套重力，而不是地面态那种"贴地 -2"。

**⑥ normalizedTime 单点触发 + 去重位**
火球在动画 `ArrowSpawnTime`（归一化时间 0~1 的某点，如 0.4）生成一次。用一个 `bool _fireballSpawned` 防止同一次播放里重复生成，并用 `IsInTransition(0)` 跳过 CrossFade 过渡期（过渡期 `normalizedTime` 不可信）。

## 2. 最终选择的实现方案

核心决策：**新增一条"空中攻击 seam"，复用已有的 `PlayerWizardAttackState`，让该状态在 `Enter()` 时自己读"接没接地"来切换动画与重力分支。**

> 一句话：**入口加一条，状态不新建。**

### 2.1 为什么这样选（方案对比）

| 方案 | 做法 | 评价 |
|---|---|---|
| **A. 新建 `PlayerWizardAirAttackState`** | 再写一个空中攻击状态类 | ❌ 和地面普攻态 90% 重复（火球生成、瞄准、射速冷却、结束判定全一样），维护要改两处 |
| **B. 复用普攻态 + 接地分支**（采用） | 同一个 `PlayerWizardAttackState`，`Enter` 读 `IsGrounded` 决定动画/重力 | ✅ 零重复，发射逻辑唯一；差异只有"动画名 + 重力公式"两处小分支 |
| **C. 在地面 `TryStartAttack` 里判空中** | 不开新 seam，复用旧钩子 | ❌ 旧钩子只被地面态调用，空中态根本不会调它；硬接会破坏"地面/空中两条独立入口"的清晰度 |

选 B + 新开一条 seam，得到的结构最干净：
- **入口分两条**：`TryStartAttack()`（地面态调）/ `TryStartAirAttack()`（空中态调），各角色自行决定是否实现空中分支（法师实现、战士/弓箭手默认不支持）。
- **状态只一个**：`PlayerWizardAttackState` 用一个 `_airborne` 标志吸收地面/空中的两点差异。

### 2.2 空中攻击的取舍（呼应需求 D）

空中 `TryStartAirAttack` **只做点按火球，不做长按陨石**：
- 陨石是"地面落点引导"技能，空中没有落点语义；
- 触发用**本帧上升沿** `AttackPressedThisFrame`，不吃旧缓冲——避免把跳跃前残留的"按住/缓冲"误判成空中起手（这点和地面法师攻击的"边沿门控"是同一个防御思路）；
- 仍然走**同一个射速冷却** `AttackCooldown`，所以地面/空中连发的节奏一致。

## 3. 实现过程（改了哪些文件、为什么）

一共动 4 个文件：2 个加"新入口"，1 个角色"接线"，1 个状态"吸收差异"。

### 3.1 `PlayerControllerBase.cs` —— 新增空中攻击 seam

```csharp
public virtual bool TryStartAttack() => false;      // 已有：地面攻击钩子

/// <summary>
/// 空中攻击触发 seam：共享的 AirborneState 在落地检测前调用本钩子。
/// 基类默认不支持空中攻击（返回 false）；具体角色在子类重写（如法师空中火球普攻）。
/// </summary>
public virtual bool TryStartAirAttack() => false;   // 新增：与地面钩子平行
```
**为什么**：把"空中攻击"做成基类的一个虚钩子，和地面钩子对称。基类默认 `false`，意味着战士/弓箭手不写就是"不能空中攻击"，零侵入；只有法师重写它。这是 seam 模式的标准扩展方式——共享层只定义"有这么个时机点"，具体语义下沉到子类。

### 3.2 `PlayerAirborneState.cs` —— 在空中态调用新钩子

```csharp
public override void Update()
{
    // 土狼跳（Coyote Jump）：保留最高优先级
    if (_player.JumpBufferCounter > 0f && _player.CoyoteTimeCounter > 0f)
    {
        ExecuteCoyoteJump();
        return;
    }

    // 新增：空中攻击（角色子类决定是否支持）。放在落地检测之前、土狼跳之后：
    // 保留跳跃宽限优先级，攻击则替代本帧的重力/移动/落地处理。
    if (_player.TryStartAirAttack())
        return;

    HandleGravity();
    HandleMovement();
    base.HandleRotation();
    CheckTransition();   // 落地检测
}
```
**为什么**：空中态原本只管重力+落地，根本不会去问"要不要攻击"。这里在它的逻辑链里**插一个优先级位**。位置很关键——放在**土狼跳之后**（跳跃宽限是更高优先的玩家意图，不能被攻击吞掉），放在**落地检测之前**（攻击触发后本帧直接 `return`，不再跑这一帧的重力/移动/落地，交给攻击态接管）。

### 3.3 `WizardController.cs` —— 接线：空中动画名 + 重写空中钩子

**(a) 空中动画状态名（数据驱动 + 预哈希）**
```csharp
[Tooltip("空中普攻动画状态名（Animator 节点 JumpAttack_MagicWand）；空 → 0 → 退回地面普攻动画")]
[SerializeField] private string _airAttackStateName = "JumpAttack_MagicWand";

private int _airAttackStateHash;            // Awake 预哈希一次
public int AirAttackStateHash => _airAttackStateHash;

// Awake 内：
_airAttackStateHash = string.IsNullOrEmpty(_airAttackStateName)
    ? 0 : Animator.StringToHash(_airAttackStateName);   // 空 → 0 → 状态据此退回地面动画，不报错
```
**为什么**：动画状态名做成 `SerializeField` 而非写死，和项目里所有 CrossFade 目标一样——美术改了节点名，策划在 Inspector 改即可，不动代码。`StringToHash` 在 `Awake` 算一次、绝不每帧/每次触发算（性能规则）。空字符串退化成 0，攻击态据此安全回退到地面动画，不会因为漏配崩。

**(b) 重写空中攻击钩子**
```csharp
public override bool TryStartAirAttack()
{
    if (!AttackPressedThisFrame) return false;          // 空中必须本帧新按下，不吃旧缓冲（防误触）
    if (AttackCooldownCounter > 0f) return false;       // 射速冷却中：本次点按作废
    AttackCooldownCounter = _combo != null ? _combo.AttackCooldown : 0f;  // 与地面普攻共用射速冷却
    AttackBufferCounter = 0f;
    StateMachine.ChangeState(_wizardAttackState);        // 复用普攻态：其 Enter 据接地与否切动画
    return true;
}
```
**为什么**：这里集中表达了需求 D（只点按、不蓄力陨石）、防误触（上升沿门控）、射速一致（共用 `AttackCooldown`）。最后切到的是**已有的** `_wizardAttackState`，而不是新状态——复用的关键落点。

### 3.4 `PlayerWizardAttackState.cs` —— 用一个标志吸收地面/空中差异

**(a) 进入时锁定情景**
```csharp
private bool _airborne;   // 本次起手是否在空中：决定播空中动画 + 用真实重力

public override void Enter()
{
    _comboIndex = 0;
    _player.AttackBufferCounter = 0f;
    _fireballSpawned = false;
    _airborne = !_player.GroundChecker.IsGrounded;   // 起手瞬间锁定（状态不能传参，只能读上下文）
    ...
    StartSegment(0);
}
```
**为什么**：FSM 的 `ChangeState` 不带参数，所以"这次是地面还是空中起手"只能在 `Enter()` 里读 `IsGrounded` 得到，并锁存进 `_airborne`，供本次播放全程使用。

**(b) 动画分支**
```csharp
private void StartSegment(int index)
{
    _fireballSpawned = false;
    // 空中起手且配了空中动画 → 播 JumpAttack_MagicWand；否则退回地面普攻段动画
    int hash = _airborne && _wizard.AirAttackStateHash != 0
        ? _wizard.AirAttackStateHash
        : _wizard.GetComboStateHash(index);
    _player.Animator.CrossFadeInFixedTime(hash, CrossFadeDuration, 0);
}
```

**(c) 重力分支**
```csharp
private void HandleGravity()
{
    if (_airborne)
    {
        // 空中起手：继续真实分段重力下落（与 AirborneState 同款），边落边施法
        float multiplier = _player.VerticalVelocity < 0f
            ? _player.FallGravityMultiplier
            : _player.GravityMultiplier;
        _player.VerticalVelocity += Physics.gravity.y * multiplier * Time.deltaTime;
    }
    else if (_player.VerticalVelocity < 0f)
    {
        _player.VerticalVelocity = -2f;   // 地面起手：贴地微压
    }
}
```
**为什么**：地面普攻是"贴地不掉"，空中普攻要"边落边放"。差异只有重力这一处公式，用 `_airborne` 一分二即可。火球生成、屏幕中心瞄准、`CheckEnd`（射出即交还控制权）、`TransitionToMovement`（结束后按 `IsGrounded` 回地面/空中态）**全部不动**——这正是"复用一个状态"省下来的部分。

### 3.5 需要在 Editor 里手动做的（代码之外）

> 本项目约定：Animator 连线、Play 模式测试由开发者在 Editor 完成；代码只数据驱动"进入动画名"。

1. **Animator 新建 `JumpAttack_MagicWand` 状态**（已建）。
2. **给它加退出连线** `JumpAttack_MagicWand → Idle_Bow`，勾 **Has Exit Time**——否则代码只切 FSM、动画不淡出，角色卡在施法 pose（呼应原理 ④）。
3. **不需要进入连线**——进入由代码 CrossFade 点名。
4. `JumpAttack_MagicWand` 的"挥杖出手帧"尽量对齐火球的 `ArrowSpawnTime`（当前 0.4，地面/空中共用）。

## 4. 模拟大厂面试 Q&A

**Q1：给"空中也能攻击"这个需求，你的第一反应是新建一个状态还是复用？为什么？**
A：我先看新旧两种情景的"差异面"有多大。空中攻击和地面攻击在火球生成、屏幕中心瞄准、阵营判定、射速冷却、结束判定上完全一样，只有"播哪个动画"和"重力怎么算"两点不同。差异这么小，新建一个状态类会带来 90% 的重复代码，以后改发射逻辑得改两处、容易漏。所以我选复用同一个攻击状态，用一个 `_airborne` 标志把那两点差异在状态内部一分为二。新建状态我会留给"行为结构本质不同"的情况，而不是"只是参数/分支不同"。

**Q2：你的状态机 `ChangeState` 不能传参，那"这次是空中起手"这个信息怎么进到攻击状态里？**
A：靠状态在 `Enter()` 里读上下文，而不是靠外部传。我在 `Enter()` 里 `_airborne = !GroundChecker.IsGrounded` 把当前接地情况锁存下来，整次播放都用它。这样做的额外好处是状态自洽——它不依赖"谁、用什么参数"切进来，任何入口切进来它都能根据当下环境正确表现。如果将来要传更复杂的数据，我才会考虑在状态上加显式的配置字段，在 `ChangeState` 前设好；但对一个布尔情景，读上下文是最轻的。

**Q3：空中攻击的入口，你为什么不复用已有的 `TryStartAttack`，而要新开一个 `TryStartAirAttack`？**
A：因为 `TryStartAttack` 只被地面态调用，空中态根本不会触发它——这俩是两条独立的状态分支。如果硬把空中判断塞进地面钩子，语义会混淆，而且地面态也不该关心空中的事。我选择开一个对称的 seam：地面态调地面钩子、空中态调空中钩子，基类两个都给默认 `false` 实现。这样每个角色可以独立决定"我支不支持空中攻击"——法师重写空中钩子，战士、弓箭手不写就是不支持，零侵入、可扩展。这是接缝模式的标准用法：共享层只定义"有这么个时机点"，具体语义下沉子类。

**Q4：空中攻击的检查，你放在空中态 Update 的哪个位置？为什么不能随便放？**
A：放在"土狼跳之后、落地检测之前"。顺序是优先级的体现。土狼跳是玩家"想跳"的宽限补偿，是更高优先的意图，不能被攻击吞掉，所以攻击在它后面。而一旦攻击触发，我当帧直接 `return`，跳过这一帧的重力、移动、落地检测——因为控制权已经交给攻击态了，再跑这些就会和攻击态打架（比如落地检测可能当帧把状态又切回地面）。这跟地面态里 `Dash → Attack → Jump` 的优先级链是同一个设计思路：状态转移顺序即优先级。

**Q5：空中攻击时角色还要继续下落，你怎么处理重力？为什么不直接复用地面攻击的重力代码？**
A：地面攻击的"重力"其实是 `if (v<0) v=-2` 的贴地微压，目的是让 CharacterController 稳稳贴在地面上、不抖。空中攻击要的是"边落边放"，得用真实的分段重力积分 `v += g*multiplier*dt`，且上升/下落用不同倍率，和空中态完全一致。这俩语义相反，不能共用，所以我在攻击态的 `HandleGravity` 里按 `_airborne` 一分二：空中走真实重力、地面走贴地微压。水平移动那部分本来就是统一的 `Move(velocity)`，不用动。

**Q6：动画是怎么进、怎么出的？为什么代码里只管"进"不管"出"？**
A：进入用 `Animator.CrossFadeInFixedTime` 由代码直接点名目标状态，所以 Animator 里不用画进入连线，状态名是数据驱动、`Awake` 里 `StringToHash` 预哈希一次。但"出"是 Animator 自己的事——代码切 FSM 状态时不会 CrossFade 回去，所以每个这样进入的状态必须自带退出连线（带 Has Exit Time），否则会卡在施法 pose。这是一个很容易踩的坑：加了新动画状态，进入由代码保证了，但忘了在 Animator 里连退出线，结果人物定格。所以我在交付时会专门提醒开发者去 Animator 补这条退出连线。

**Q7：空中为什么不支持长按陨石重击？触发上你做了什么防护？**
A：陨石是地面落点引导技能，需要一个地面落点，空中放它没有语义，所以空中只保留点按火球。触发上我用本帧上升沿 `AttackPressedThisFrame` 而不是攻击缓冲来判定——这能避免把跳跃前残留的"按住"或缓冲误判成空中起手。地面法师攻击里也用了同样的"边沿门控"来防"重击后点按被误判为重击"，是一致的防御思路。另外空中攻击仍然走和地面同一个射速冷却 `AttackCooldown`，所以无论在地面还是空中，连发的最小间隔是一致的，不会因为换了入口就破节奏。

**Q8：这个功能你是怎么保证零 GC 的？**
A：几个点。状态对象在 `Awake` 预先 new 好，运行时切状态只改引用、不分配。动画状态名的 `StringToHash` 在 `Awake` 算一次缓存成 int，绝不每帧或每次攻击算。火球的射线命中缓冲 `RaycastHit[]` 也是状态对象的字段、只在构造时分配一次，用 `RaycastNonAlloc` 写进去，避免每次瞄准产生数组垃圾。火球本体的 `Instantiate` 是离散输入上的一次性分配（按一次左键生成一个），不在每帧热路径上，是可接受的。整个空中攻击的每帧逻辑里没有 new、没有 LINQ、没有装箱。
