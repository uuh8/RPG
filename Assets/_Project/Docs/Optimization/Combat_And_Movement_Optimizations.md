# 战斗与移动 —— 已有功能优化解析

> **本文定位**：把我们这几次对"**已有功能**"做的优化集中解析。区别于 `Wizard_System_Design.md` / `Enemy_System_Design.md`（那是新功能从 0 到 1），本文聚焦"功能已经能跑、但手感/正确性有缺陷，如何打磨"。
> 每个优化按四部分写：① 之前状态与弊端 → 优化后效果；② 改/加了哪些文件、用到什么技术；③ 我补充的、对你重要的要点；④ 模拟大厂面试 Q&A。
> 适合作为"手感打磨 / Bug 修复 / 工程取舍"方向的面试素材与复习资料。

## 目录
1. 冲刺位移延迟 —— 对齐翻滚动画（消除"先闪后翻"）

3. 攻击输入边沿门控 —— 修"重击后点按被误判为重击"
4. 角色转向锁存 —— 短按也转到位，不停在中途
5. DoT 受击反应抑制 —— 修"持续伤害时角色站桩/锁死"

> 贯穿全文的一个心智模型：**手感问题几乎都是"两条时间线/两种语义没对齐"**——位移与动画、射速与动画、输入电平与边沿、伤害与受击反应。优化的本质是把它们正确地解耦或对齐。

## 1. 冲刺位移延迟 —— 对齐翻滚动画

### ① 之前状态与弊端 → 优化后效果
- **之前**：冲刺(Dash)是一个"前滚翻"。按下冲刺键的**那一瞬间角色就整体平移过去了**，而翻滚动画的"翻起来"在动画稍靠后的时间点——结果是"人先闪过去、动作再滞后跟上"，割裂感强。
- **根因**：位移和动画是两条独立时间线。冲刺状态一进入就开始全速位移，而翻滚发力点在动画中后段。
- **优化后**：位移**延迟零点几秒**再启动，对齐翻滚的发力帧；延迟段只播动画+贴地、人不移动，过了延迟才全速冲出。位移有效距离不缩水。观感变成"蓄势→翻滚冲出"，连贯自然。

### ② 改/加了哪些文件、用到什么技术
- `PlayerControllerBase.cs`：新增 `[SerializeField] float _dashMoveDelay`（默认 0.1s）+ 暴露 `DashMoveDelay`。
- `PlayerDashState.cs`：
  - `HandleDashMovement`：`float horizontalSpeed = _elapsed >= _player.DashMoveDelay ? _player.DashSpeed : 0f;`（延迟内水平速度为 0，仍施加 -2 贴地压力）。
  - 结束条件：`if (_elapsed >= _player.DashMoveDelay + _player.DashDuration)` —— **把延迟加进总时长**，保证位移段时长不变、冲刺距离 = `DashSpeed × DashDuration` 不缩水。
- **技术**：计时器/相位对齐；数据驱动（延迟量 `SerializeField`，每个角色可调，因为不同翻滚动画发力点不同）。

### ③ 补充：为什么"延迟要加进总时长"
新手最容易写成"前 0.1s 不动、总时长不变"——那样会**吃掉 0.1s 的位移**，冲刺变短。正确做法是 `总时长 = 延迟 + 位移时长`，延迟只是把位移段整体往后推，距离守恒。这种"调相位但保总量"的思路，在很多手感打磨里都会用到。

### ④ 面试 Q&A
**Q：动作游戏里位移和动画对不齐，你怎么处理？**
A：本质是两条时间线相位错位。我的做法是给位移加一个可配置的启动延迟，让它和动画的发力帧对齐——延迟段只播动画、不位移，过了延迟再施加位移；同时把延迟加进状态总时长，保证位移的有效时长和距离不被延迟吃掉。延迟量做成数据（Inspector 可调），因为每个动作的发力帧不同，这是手感参数该留给策划/美术，而不是写死。

**Q：为什么不用 Animation Event 在发力帧那一刻触发位移？**
A：可以，但本项目刻意让所有时序统一走"代码计时/normalizedTime 比阈值"，集中、可单测、跨角色一致；Animation Event 把时机散进动画文件、不好统一管理。对"延迟一个固定量"这种简单相位对齐，一个计时器最直接。

## 2. 攻击射速冷却 —— 把射速从动画长度解耦

### ① 之前状态与弊端 → 优化后效果
- **之前**：远程普攻（箭/火球）的发射频率**完全由攻击动画长度决定**——投射物在动画 `ArrowSpawnTime`（约 0.4）就射出去了，但状态要等动画播到 `EndThreshold`(0.85) 才交还控制权。于是 **0.4→0.85 这段是纯空窗**：明明已经射完，却还不能再射，两发间隔被动画绑死，手感拖沓。
- **优化后**：引入**数据驱动的攻击冷却 `AttackCooldown`**，射速由它决定、与动画长度无关；投射物**一射出就立刻交还控制权**（不再干等 0.85），下一发由冷却闸门控制。射速变成一个可独立调的数值旋钮。

### ② 改/加了哪些文件、用到什么技术
- `ComboDefinition.cs`：新增 `float AttackCooldown`（射速旋钮，0=旧行为）。
- `PlayerControllerBase.cs`：新增 `AttackCooldownCounter`（每帧在 Update 递减），与 Dash 的 buffer/cooldown 同一套"正交计数器"模式。
- `ArcherController.cs` / `WizardController.cs`：普攻分支冷却门控——冷却中点按作废，否则启动冷却再进攻击态（重击/蓄力不受此冷却限制）。
- `PlayerBowAttackState.cs` / `PlayerWizardAttackState.cs`：把"等动画 0.85 才结束"的 `CheckCombo` 换成 `CheckEnd`——**投射物射出即 `TransitionToMovement`**，并保留"始终没射出时用 0.85 兜底"防卡死。
- **技术**：数据驱动；buffer/cooldown 正交计数器模式；`normalizedTime` 单点生成 + 提前结束。

### ③ 补充：射速的真正下限
两发最小间隔 = `max(AttackCooldown, 出膛所需时间)`。"出膛时间"(CrossFade + `ArrowSpawnTime`×动画长) 是物理下限——箭还没离手当然不能再射。所以把 `AttackCooldown` 设得 ≥ 出膛时间，它才是唯一权威；设得比出膛时间还小则无效（被物理下限兜住）。另外，因为长按已被蓄力/重击占用，普攻连发靠**连续点按**实现，点多快由冷却封顶，而不是"按住自动连射"。

### ④ 面试 Q&A
**Q：你的攻击频率一开始是被动画长度决定的，为什么这是问题？怎么解的？**
A：射速本该是一个独立的战斗数值，不该"动画多长射速就多慢"。原实现要等攻击动画播到 85% 才允许下一次输入，可投射物在 40% 就射出了，中间 45% 是无意义空窗。我把它改成：投射物一射出就交还控制权，两发间隔交给一个数据驱动的攻击冷却来管——射速从此和动画解耦、可独立调，还顺带支持"动画取消"的观感（下一发 CrossFade 盖过上一发动画尾巴）。

**Q：buffer 和 cooldown 有什么区别？**
A：两个正交的计数器。buffer 是"亚帧输入容错"——提前一点按下也不丢；cooldown 是"能力锁"——多久才能再用一次。冲刺同时用了两者（缓冲 + 冷却），攻击原来只有 buffer、缺 cooldown，这次把 cooldown 补上。两者都用"输入设时长 / Update 每帧递减 / >0 视为有效"的统一模式。

## 3. 攻击输入边沿门控 —— 修"重击后点按被误判为重击"

### ① 之前状态与弊端 → 优化后效果
- **之前**：点按=普攻、长按=重击/蓄力，靠"每帧只要 `IsPressed()` 为真就累加按住时长，过阈值进重击"。这套用的是**电平(level)**判断，**没有"刚按下"的概念**——分不清"这一帧我刚按下"和"我从上一个动作一直按着没松手"。
- **弊端**：重击是按住触发的，且引导/施法期间地面态不跑、累加器不动；等重击结束回到地面态时按键**还按着**（或在施法尾段提前重按、按压跨过了状态边界），地面态一恢复就对这颗"残留按住"从头累加，很快又过阈值 → **点按被误判成重击**。
- **优化后**：改用**上升沿(edge)**——只有"本帧刚按下"才开始跟踪一次输入；从上一个动作残留下来的按住没有新边沿，**不会启动新的蓄力**。误触发根除，还白送了亚帧点按容错。

### ② 改/加了哪些文件、用到什么技术
- `PlayerControllerBase.cs`：新增 `AttackPressedThisFrame => _inputActions.Player.Attack.WasPressedThisFrame()`（上升沿）。
- `ArcherController.cs` / `WizardController.cs`：新增 `_attackTracking` 标志，只有 `AttackPressedThisFrame` 才置真并清零计时；`_attackTracking` 为假直接返回（残留按住不触发）；抽 `EndAttackTracking()` 统一收尾。
- **技术**：Unity Input System 的 `WasPressedThisFrame`（边沿）vs `IsPressed`（电平）；状态边界的输入残留处理。

### ③ 补充：电平 vs 边沿，是输入设计的分水岭
**"这一刻是不是按着"(电平)** 和 **"这一刻是不是刚按下"(边沿)** 是两回事。凡是"一次性意图的开始"（开始蓄力、触发跳跃、确认）都该用边沿；凡是"持续状态"（是否仍在蓄力、是否仍按住移动）才用电平。最初的 bug 就是把"开始一次长按"错误地建立在电平上。可迁移的经验：**任何"会让输入跨越状态边界"的设计，都要立刻问一句『状态切回来时输入残留会不会被误读』。**

### ④ 面试 Q&A
**Q：同一个键，点按和长按两种意图，你怎么区分？踩过什么坑？**
A：按住累计时长、过阈值算长按。坑在于我最初用"每帧 IsPressed 就累加"，这是电平判断，分不清"刚按下"和"一直按着"。重击是按住触发、且触发后会离开可攻击状态一段时间，等状态切回来时按键往往还按着，于是这颗残留电平被当成新的一次长按、把紧接着的点按误判成重击。修复是改用上升沿 WasPressedThisFrame：只有真正"刚按下"才开始跟踪，残留按住没有新边沿就不触发。顺带，按下+松开同帧的亚帧点按因为有上升沿、且当帧 IsPressed 已为假，会正确走点按分支。

**Q：这个修复只改了一个角色吗？**
A：弓箭手和法师是同一套路由，所以是同一个 bug，一起改了；并把上升沿读取上移到基类 `PlayerControllerBase` 暴露成属性，两个子类共用，避免两处各写一遍又不同步。

## 4. 角色转向锁存 —— 短按也转到位，不停在中途

### ① 之前状态与弊端 → 优化后效果
- **之前**：转向 `HandleRotation` 只在"有移动输入时"朝移动方向 Slerp。短按一个方向（尤其后退 S），**松手瞬间 `MoveDirection→0`，方法直接早退，Slerp 停在转身中途** —— 角色卡在一个尴尬的半转角度。
- **优化后**：按下方向时把该方向**锁存**为目标朝向；之后**即使松开也继续朝它转到位**。无论按多短，角色最终都面向所按方向；转完即停、不抖动。

### ② 改/加了哪些文件、用到什么技术
- `PlayerControllerBase.cs`：新增 `Vector3 TargetFacing`（锁存的目标朝向）；`Start` 里初始化为 `transform.forward`（避免开局自转到世界 +Z）。
- `PlayerStateBase.cs`：`HandleRotation` 改为——有输入时更新 `TargetFacing`；**每帧始终朝 `TargetFacing` Slerp**（与是否按住无关），已对齐时 Slerp 本身是无操作。
- **技术**：四元数 `Quaternion.LookRotation` + `Slerp` 球面插值；锁存状态。

### ③ 补充：锁存为什么放在角色而不是状态里
锁存放在 `PlayerControllerBase`（角色）上，能**跨状态保留**：比如"轻点 S 然后立刻跳"，松手时已经在空中，但因为目标朝向锁存在角色上，空中状态继续调 `HandleRotation` 会把这次转身补完。若锁存在某个状态对象里，切状态就丢了。这是"把需要跨状态存活的数据放在 Context(角色) 上、而非放在易失的状态里"的通用取舍。

### ④ 面试 Q&A
**Q：角色短按方向键会停在转身中途，怎么修？**
A：原来转向只在有移动输入时执行，松手就停了。我把"目标朝向"锁存到角色上：按下方向时更新锁存值，然后每帧始终朝锁存值 Slerp，与当前是否按住无关，所以转身一定会转完。锁存放在角色（Context）而不是状态里，是为了跨状态保留——边跑边跳时空中状态也能把转身补完。已对齐时 Slerp 是无操作，所以不会�u动。

**Q：为什么转向用 Slerp 而不是 Lerp？**
A：Slerp 沿单位四元数球面的大圆弧走，角速度恒定，是真正的匀速转向；Lerp 对四元数分量线性插值，角速度不恒定（中间快两头慢）。转向这种要"匀速顺滑"的，用 Slerp 更自然。

## 5. DoT 受击反应抑制 —— 修"持续伤害时角色站桩/锁死"

### ① 之前状态与弊端 → 优化后效果
- **之前**：被火球点燃(BurnStatus)或站在陨石火场(AreaDamage)里时，**每一跳 DoT 都走和直击完全相同的 `ReceiveHit` 路径** → 每跳都发 `DamageReceivedEvent` → 表现层每跳都 CrossFade 受击动画、敌人每跳都进硬直态。结果：玩家被钉在受击姿态几乎不能动、敌人被持续 DoT **永久锁死(stun-lock)**。
- **优化后**：给伤害打一个**"是否触发受击反应"标记**。DoT 标记为 false → **跳过受击动画、跳过敌人硬直**，但仍正常扣血、仍闪红（轻量反馈）。被点燃/站火场里可以自由走动；直击和陨石爆炸仍有受击顿挫。

### ② 改/加了哪些文件、用到什么技术
- `DamageRequest.cs`：新增 `bool TriggerHitReaction`，**构造函数可选参数默认 true** → 所有现有直击调用零改动、照常受击。
- `DamageReceivedEvent.cs` / `HealthComponent.cs`：把标记透传进事件。
- `CharacterCombatFeedback.cs`：闪红始终触发；标记为 false 时**不放血、不播 GetHit 动画**。
- `EnemyController.cs`：标记为 false 时**不进 HurtState**。
- `BurnStatus.cs`：每跳传 `triggerHitReaction: false`。
- `AreaDamage.cs`：新增序列化开关 `_triggerHitReaction`（爆炸默认 true=会震；火场设 false）。
- **技术**：伤害标记 / 通道语义分离；C# 可选构造参数（向后兼容，不破坏既有调用）；EventBus 透传。

### ③ 补充：这是 DoT 的经典问题，标记法是标准解
"持续伤害不该触发受击反应"是 DoT 的通病。业界三类解法：① **打标记**（直击 vs DoT，最主流最灵活，本项目选）；② 受击反应冷却/去抖（简单但会误限流真实连击）；③ 削韧/硬直阈值（攒够硬直值才踉跄，DoT 贡献为 0，最高级）。这个布尔标记其实就是方案③(poise)的雏形——以后做削韧时，把"是否硬直"从布尔升级成"硬直值累计"，DoT 贡献 0、直击贡献高即可平滑演进。另外，**用可选参数默认 true** 是关键工程技巧：给共享结构体加字段却不破坏几十处既有调用，新语义只在 DoT 来源显式开启。

### ④ 面试 Q&A
**Q：持续伤害(DoT)导致角色一直播受击、无法移动，一般怎么解决？**
A：根因是 DoT 每跳都走了和直击相同的受击反应路径。标准解法是给伤害标注性质——加一个"是否触发受击反应"的标志，由伤害来源声明：直击为真、DoT 为假。表现层和硬直系统据此判断，DoT 跳过受击动画与硬直，只扣血、只做轻量反馈（闪红/飘字）。我用的是这个标记法。更进阶可以做削韧系统，让硬直取决于累计硬直值而非单次，DoT 贡献为 0——这个布尔标志正是它的雏形。

**Q：给 DamageRequest 这种被到处使用的结构体加字段，怎么不破坏现有代码？**
A：把新字段做成构造函数的**可选参数、默认值取"保持原行为"**（这里默认 true=照常受击）。这样几十处现有的直击调用一行都不用改、行为不变；只有 DoT 来源显式传 false 开启新语义。向后兼容的同时引入新能力。

## 6. 箭矢飞行朝向 + 命中残留

### ① 之前状态与弊端 → 优化后效果
- **之前**：① 弓箭手射出的箭**竖着飞**（箭尖朝天），不符常理；② 箭命中即消失，没有"扎中"的实感。
- **根因(①)**：箭的模型"箭尖"沿**本地 +Y**建模，而 `Quaternion.LookRotation(velocity)` 把模型的**本地 +Z** 对齐到飞行方向（Unity 约定 +Z 为前）。于是 +Z 对准了飞行方向、+Y 箭尖却指向侧上方 → 看着竖着飞。预制体上又没有序列化修正角（默认 0）。
- **优化后**：① 加 `(90,0,0)` 修正旋转把 +Y 箭尖掰到飞行方向；并让箭**飞行中持续对齐速度方向**（抛物线下坠时机头跟着俯冲）。② 命中后**冻结物理、关碰撞、挂到命中物体下，残留一小段时间再销毁**——形成"插在目标/地面上"的效果。

### ② 改/加了哪些文件、用到什么技术
- `ProjectileBase.cs`：新增 `_impactLingerTime`（命中后停留多久再销毁，0=立即）；`FaceVelocityInFlight`（虚属性，抛物线箭重写为 true）+ `FixedUpdate` 每帧把模型朝向对齐当前速度。
- `Arrow.cs`：`override FaceVelocityInFlight => true`；`OnImpact` 里冻结 `Rigidbody`(kinematic + 清速度)、关 `Collider`、`SetParent` 到命中物体、由基类按 `_impactLingerTime` 延时销毁。
- `Arrow.prefab`：序列化 `_modelForwardOffsetEuler:(90,0,0)`、`_impactLingerTime:3`。
- **技术**：四元数 `LookRotation` + **模型本地修正旋转**；`Rigidbody.isKinematic` 冻结；`Transform.SetParent`；`Destroy(go, delay)`。

### ③ 补充：LookRotation 永远对齐 +Z，模型不是 +Z 朝前就要补修正
这是 3D 数学里极常见的坑：`LookRotation` 让物体**本地 +Z** 指向目标方向（Unity forward 约定）。任何用它对齐朝向的物体，只要美术建模的"前方"不是 +Z，就要乘一个本地修正旋转（箭尖 +Y → 补 `(90,0,0)`，某些 -Z 模型补 `(0,180,0)`）。把修正角做成 Inspector 可调字段，让美术换模型时自己调，比改代码好。

### ④ 面试 Q&A
**Q：箭飞出去是竖着的，为什么？怎么修？**
A：`Quaternion.LookRotation(velocity)` 对齐的是物体的本地 +Z（Unity 约定 +Z 为前），而这支箭的模型箭尖是沿本地 +Y 建的，于是 +Z 对准了飞行方向、+Y 箭尖指向侧上方，看着就竖着飞。修复是乘一个本地修正旋转 `Euler(90,0,0)` 把 +Y 掰到飞行方向，并把这个修正角做成预制体上的可调字段。通用结论：凡是用 LookRotation 对齐的，模型若不是 +Z 朝前就要补一个本地修正旋转。

**Q：命中后"插在身上停留"怎么实现，要注意什么？**
A：命中瞬间把刚体设为 kinematic 并清零速度、关掉碰撞体，再 SetParent 到被命中物体下随它移动，然后延时销毁。注意要关碰撞体、设 kinematic，否则残留的箭还会参与物理、被二次碰撞或推动；延时销毁的时长做成数据可调。

## 7. 近战命中防自伤 —— 纵深防御

### ① 之前状态与弊端 → 优化后效果
- **之前**：`MeleeHitDetector` 只按**阵营**过滤（跳过 `TeamId == _attackerTeam`）。联调敌人时出现"敌人出招把自己打死"：敌人 detector 的 `_attackerTeam` 和它自己 `HealthComponent.TeamId` **配得不一致**，挥击的 OverlapBox 扫到自己身体时没被当成同队 → **打到了自己**。
- **优化后**：除了阵营过滤，命中循环**额外跳过"攻击者自身层级"**（`IsChildOf(_ownerRoot)`）。这样无论阵营怎么配错，挥击都**绝不会打到自己身体**。对正确配置的角色无副作用。

### ② 改/加了哪些文件、用到什么技术
- `MeleeHitDetector.cs`：`DoOverlap` 命中循环加 `Transform ownerT = _ownerRoot != null ? _ownerRoot : transform; if (_buf[i].transform.IsChildOf(ownerT)) continue;`。
- **技术**：`Transform.IsChildOf`（含自身）；**纵深防御(defense in depth)**——找到根因(配置)后在另一层加保险。

### ③ 补充：根因是配置，但代码也该兜底
直接根因是阵营配错（不变量"detector 队号 = 自己 HealthComponent 队号"被违反），治本是改配置。但"默认队号 0 是有效队号"这件事本身是个陷阱——配错时不会报错，只会诡异自伤。所以除了按数据过滤，再加一道**结构性过滤(不打自己层级)**做纵深防御，让同类错误不再以"自伤"的形式复现。这就是"找到根因后，在多层各加一道验证"的工程习惯。

### ④ 面试 Q&A
**Q：近战出招把自己打死了，怎么排查和修？**
A：现象是攻击时攻击者自己掉血。`MeleeHitDetector` 的 OverlapBox 会扫到攻击者自己的身体，本该被同阵营过滤跳过，但发现它的 `_attackerTeam` 和自己 `HealthComponent.TeamId` 配得不一致，于是自己没被当成同队、被打到了。治本是把两者对齐（不变量：detector 队号必须等于自己角色队号）。我还做了纵深防御：在命中循环里额外用 `IsChildOf(ownerRoot)` 跳过攻击者自身层级，这样即使阵营再配错也绝不自伤——而且对正确配置的角色是无副作用的空操作。

**Q：什么是纵深防御？为什么这里值得加？**
A：在多个层次各设一道防线，单层失效时还有兜底。这里阵营过滤是第一道（按数据），自身层级跳过是第二道（按结构）。值得加是因为"队号 0 既是合法队号又是未初始化默认值"，配错时静默自伤、很难一眼看出；多一道结构性防线把这类低级错误挡死，性价比很高。

## 8. 流血粒子碰撞分层 —— 修"血溅在自己身上而非落地"

### ① 之前状态与弊端 → 优化后效果
- **之前**：流血用 `BloodDripping` 粒子系统（开启了 World **Collision** 模块：血滴下落、碰到表面就"溅开"）。但它被**挂在角色身上**(parent=角色)，而角色自身有碰撞体(CharacterController 胶囊)，且与地面**在粒子碰撞遮罩的同一可碰撞范围内**——于是血滴一生成就撞到角色自己的身体，**在身上摊开溅射**，而不是落到地面。
- **优化后**：把**角色自身所在层**从血液粒子的 `Collision.collidesWith` 遮罩中剔除。血滴穿过自己、继续下落，**只在地面/环境上溅射**，符合"伤口滴血→落地"的物理直觉；同时血液仍随角色移动(挂载关系不变)。

### ② 改/加了哪些文件、用到什么技术
- `CharacterCombatFeedback.cs`：
  - `SpawnBlood` 里两个血效果(`BloodExplosion`/`BloodDripping`)实例化后都调 `ExcludeOwnerFromParticleCollision`。
  - 新增 `ExcludeOwnerFromParticleCollision(GameObject)`：`GetComponentsInChildren<ParticleSystem>` 拿到所有粒子系统，对每个 `var col = ps.collision; col.collidesWith &= ~(1 << gameObject.layer);` 剔除自身层。
- **技术**：Unity 粒子系统 **Collision 模块的 `collidesWith` LayerMask**（可运行时脚本修改——模块结构体的属性写回底层系统）；位运算 `~(1 << layer)` 清除某一层位。

### ③ 补充：为什么是改"层遮罩"而不是"不挂到角色身上"
- 不挂到角色身上(不 parent)会让血液停在出生点、不跟随角色——伤口血本该随角色走，所以保留挂载。真正要解决的是"别撞到自己"，而粒子的 World 碰撞**只能按 LayerMask 过滤**(没有"忽略某个具体碰撞体"的选项，`Physics.IgnoreCollision` 对粒子无效)。所以正解是把自身层从遮罩里剔除。
- **前提(重要)**：角色与地面必须**不在同一层**。本项目角色本就在专用层(Player/Enemy，命中遮罩/瞄准遮罩都依赖它)、地面在 Default，所以剔除角色层后血液照常落地。若你的角色和地面同在 Default，这个剔除会让血液连地面也穿过——那是"层规划"问题，应把角色放到专用层(良好实践本就如此)。
- 用 `gameObject.layer`(角色根层，CharacterController 所在层) 做剔除，零配置、自动适配玩家/敌人各自的层。

### ④ 面试 Q&A
**Q：粒子特效(血、火花)溅在了角色自己身上而不是地面，为什么？怎么修？**
A：那个粒子系统开了 World 碰撞，又被挂在角色身上，角色自身的碰撞体和地面都在它的碰撞遮罩里，于是粒子一出生就撞到自己的身体、在身上溅开。修复是把角色自身所在层从粒子 Collision 模块的 `collidesWith` 里剔除——粒子的世界碰撞只能按 LayerMask 过滤（没有忽略单个碰撞体的接口），所以按层排除自己最直接。前提是角色和地面分属不同层，本项目角色本就在专用层，所以血液剔除自身层后仍正常落到地面。

**Q：为什么不干脆关掉粒子碰撞，或者不把血挂在角色身上？**
A：关碰撞就没有"落地溅射"的效果了，丢表现；不挂在角色身上则血不跟随角色、伤口血看着不对。都改变了想要的表现。真正的问题只是"别撞到自己"，按层排除自身是最小代价、保留全部想要的效果的解。

**Q：`collidesWith` 是 LayerMask，你怎么"去掉一层"？**
A：位运算。层 N 对应的位是 `1 << N`，取反 `~(1 << N)` 得到"除该层外全 1"的掩码，再和原 `collidesWith` 按位与，就把第 N 层清零、其余保留。这里 N 取角色根物体的 `gameObject.layer`。

## 9. 远程攻击屏幕中心瞄准 —— 修"出手前转身导致弹道漂移"，准心常驻

### ① 之前状态与弊端 → 优化后效果
- **之前**：法师火球 / 弓手箭矢的**普通攻击**沿 `_player.transform.forward`（角色当前朝向）发射，且攻击态里每帧 `base.HandleRotation()` 让角色**随移动输入转身**。发射不是按下瞬间，而是动画到 `ArrowSpawnTime`(≈0.4s)那一刻——这段时间里只要玩家拨动方向，角色就转了，弹道方向也跟着变成"转身中途的朝向"，而不是按下时对准的方向。结果：**瞄准漂移、点哪打不中哪**，远程手感差。（弓手蓄力重击早已是屏幕中心瞄准，普攻却没有，体验割裂。）
- **优化后**：普攻火球/箭矢一律**朝屏幕中心（轨道相机朝向）发射**——你用相机对准目标，弹道就飞向准心，与角色身体朝向、与出手前是否转身**完全解耦**。出手时角色身体平滑转向相机水平朝向（"朝哪瞄就朝哪打"，不再随移动乱转）。**准心全程常驻**（控制远程角色时一直显示），便于"先瞄后射"；死亡时自动隐藏。箭矢普攻同时改为**直线飞行**（关重力），才能精确命中准心点。

### ② 改/加了哪些文件、用到什么技术（优化思路与方法）
**思路**：把"弹道方向"的来源从"角色朝向"换成"相机朝向（屏幕中心）"——弓手蓄力重击已有这套，复用并下沉为共享能力即可。
- **新增** `CrosshairVisibilityEvent.cs`（`Game.Core`，`struct : IGameEvent`）：准心显隐**专用**事件。
- **`PlayerStateBase.cs`** 抽出两个 `protected` 共享方法（弓手普攻 / 法师普攻 / 蓄力重击三处复用，消除重复）：
  - `ResolveAimTargetPoint(LayerMask, maxDist, RaycastHit[] buffer)`：用 `Camera.ViewportPointToRay(0.5,0.5)` 取屏幕中心射线 → `Physics.RaycastNonAlloc`（**跳过射手自身碰撞体** `IsChildOf(player)`，因轨道相机在角色身后、射线会先穿过自己）取最近命中点；未命中则 `ray.GetPoint(maxDist)`。
  - `HandleAimRotation()`：把角色平滑 `Slerp` 转向相机水平 forward（只 yaw）。
- **`PlayerBowAttackState.cs` / `PlayerWizardAttackState.cs`**：`Update` 里 `base.HandleRotation()` → `HandleAimRotation()`；发射方向由 `目标点 - 生成点`（含俯仰）算出；各自加一个**预分配** `RaycastHit[16]` 缓冲（状态对象只在 `Awake` 建一次 → 零每帧 GC）；箭矢 `Init(..., useGravity:false)` 改直线。
- **`ArcherController.cs` / `WizardController.cs`**：加 `_aimMaxDistance`（瞄准射线最大距离）；远程角色在 `Start`/`OnEnable` 发 `CrosshairVisibilityEvent{true}`、`OnDisable` 发 `{false}`（为此把基类 `Start/OnEnable/OnDisable` 提为 `protected virtual`）。法师另加 `_fireballAimMask`（火球要瞄到敌人/环境，与陨石那个"只勾地面"的 `_aimMask` 区分）。
- **`CrosshairUI.cs`**：改订 `CrosshairVisibilityEvent`。
- **`PlayerChargeAttackState.cs`**：删掉自带的私有瞄准/转向逻辑，改用上面两个共享方法（行为不变）。
- **技术点**：`ViewportPointToRay` 屏幕→世界射线；`RaycastNonAlloc` + 预分配缓冲（零 GC）+ 自身过滤；`Quaternion.LookRotation`/`Slerp` 转向；`EventBus<T>` 解耦表现层；`LayerMask` 位遮罩；`protected virtual` 生命周期模板方法。

### ③ 补充：为什么"准心"要单独一个事件，而不是复用蓄力的瞄准事件
- 项目里已有 `AimStateChangedEvent`，但它被**两个**系统监听：`CrosshairUI`（显隐准心）**和** `AimFramingController`（蓄力时把角色推到屏幕左下的运镜，仿原神）。若直接用它做"准心常驻"（一直发 `Active=true`），相机会**一直被推到左下**——把"准心可见"和"瞄准运镜"两件事耦合死了。
- 解法是**按职责拆事件**：新增 `CrosshairVisibilityEvent` 只管准心；`AimStateChangedEvent` 退回只管蓄力运镜。普攻只直瞄、**不发**运镜事件（快速点射不晃镜头），蓄力重击仍发运镜事件。这是"一个事件别承担两种语义"的典型——和第 5 条给伤害加 `TriggerHitReaction` 区分语义同理。
- **常驻准心的初始化竞态**：在 `OnEnable` 发"显示"可能早于 `CrosshairUI` 在自己 `OnEnable` 里的订阅（同场景对象 `OnEnable` 顺序不定）→ 首帧漏接。所以**额外在 `Start` 再发一次**（`Start` 必晚于所有 `OnEnable`），保证初始一定可见；`OnEnable` 那次则覆盖运行时重新启用（如复活）。

### ④ 面试 Q&A
**Q：远程攻击"点哪打不中哪"，你怎么定位和解决的？**
A：根因是弹道方向取自角色 `transform.forward`，而真正发射在动画 0.4s 后的某一帧——这中间攻击态还在每帧按移动输入转身，角色朝向变了，弹道就漂了。本质又是"两条线没对齐"：玩家按下时的瞄准意图，和 0.4s 后实际发射时的角色朝向，是两个不同的方向。解法是把弹道来源从"角色朝向"换成"相机朝向（屏幕中心射线）"，瞄准与身体朝向彻底解耦；项目里弓手蓄力重击已有这套，我把它下沉到 `PlayerStateBase` 做成共享方法，普攻和蓄力一起复用。

**Q：屏幕中心射线为什么要跳过射手自己？**
A：轨道相机在角色背后，从屏幕中心打出的射线会**先穿过角色自己的身体**。若不剔除，最近命中点会落在角色身上（还在弓弦/法杖生成点之后），算出来的方向甚至会反向。所以用 `RaycastNonAlloc` 拿到所有命中后，用 `transform.IsChildOf(player)` 跳过射手自身及其子碰撞体，再取最近点。顺带：因为代码已经跳过自己，瞄准遮罩里勾不勾 Player 层都不影响正确性，是纵深防御。

**Q：你在每帧/每次发射做射线，怎么保证不产生 GC？**
A：`Physics.RaycastNonAlloc` 把命中写进**调用方预分配**的 `RaycastHit[]`，不像 `RaycastAll` 每次 new 数组。这个缓冲是状态对象的字段，只在 `Awake`（状态对象创建时）分配一次、之后复用，所以发射时零分配。这和项目里近战 `OverlapBoxNonAlloc`、AOE `OverlapSphereNonAlloc` 是同一套零 GC 模式。

**Q：为什么把箭矢普攻从抛物线改成直线？这算不算改了玩法？**
A：因为目标是"打中准心点"。抛物线箭受重力下坠，飞到准心距离时已经掉下去了，准心就成了摆设。要让"屏幕中心瞄准"名副其实，弹道必须直线（`useGravity:false`），这也和已有的蓄力直射箭、火球一致。它确实改变了普攻箭的飞行手感，是为"精确瞄准"做的有意取舍；如果想保留抛物线，可以改成"按准心方向给初速但仍带重力"，那样准心就只是大致方向而非精确落点——这是数值/手感选择，留给战斗策划定。

**Q：准心常驻为什么不直接复用已有的瞄准事件？**
A：那个 `AimStateChangedEvent` 同时驱动准心显隐和蓄力运镜（把角色推到左下）。如果用它做常驻准心，相机会被一直推偏。一个事件承担两种语义就会这样互相绑架。我按职责把准心拆成独立事件，运镜事件只留给蓄力，普攻只直瞄不晃镜头。另外常驻准心要注意 `OnEnable` 发布可能早于 UI 订阅的竞态，我在 `Start` 补发一次兜底初始可见。

## 附：可复用的优化方法论（速记）
- **手感问题 = 两条时间线/两种语义没对齐**：位移↔动画、射速↔动画、电平↔边沿、伤害↔受击反应。先找出"是哪两者错位"。
- **相位对齐保总量**：延迟位移要把延迟加进总时长，别吃掉有效距离。
- **解耦数值与表现**：射速、转向速度、冷却都该是独立可调数据，不被动画长度等表现细节绑死。
- **电平 vs 边沿**：一次性意图用边沿(WasPressedThisFrame)，持续状态用电平(IsPressed)。
- **跨状态存活的数据放 Context(角色)**，别放易失的状态对象里（转向锁存）。
- **给共享结构体加字段用"可选参数 + 默认保持原行为"**，向后兼容引入新语义（TriggerHitReaction）。
- **DoT/环境伤害要和直击区分语义**（标记法），别让持续伤害触发受击反应。
- **LookRotation 对齐 +Z**，模型非 +Z 朝前补本地修正旋转，且做成可调字段。
- **纵深防御**：找到根因后，在另一层加结构性兜底，挡住同类低级错误。
- **粒子世界碰撞只能按 LayerMask 过滤**：要"别撞到自己/某类对象"就把对应层从 `collidesWith` 剔除；前提是把角色放到与地面不同的专用层。
- **弹道方向别绑死角色朝向**：要精确瞄准就取相机/屏幕中心射线，把瞄准与身体朝向解耦；屏幕中心射线记得跳过射手自身（轨道相机会先穿过自己）。
- **一个事件别承担两种语义**：准心显隐 vs 瞄准运镜要拆成两个事件，否则复用一个会互相绑架（同 `TriggerHitReaction` 区分直击/DoT）。
- **常驻类 UI 信号防初始竞态**：`OnEnable` 发布可能早于订阅方订阅，补一次 `Start` 发布兜底（`Start` 必晚于所有 `OnEnable`）。
- **改完就在 Editor 验证一个明确可见的结果**（Unity 编译/Play 只能人工做），bug 才会在当步暴露。
