## 第一步：先盯住"最终效果"（我们的终点）

请在脑子里（或进 Play 模式）盯住这个具体画面：

> 我在法杖里摆了 **`[三重, 增伤, 散射, 火球, 火球, 火球]`**，按一次鼠标左键 ——
> 法师挥杖，**同时飞出 3 颗火球**，它们**呈扇形散开**、**每颗都更疼**，**朝着屏幕准心方向**飞出去，**命中敌人后敌人掉血**（比单发掉得多）。
> 然后我**只改法杖里的数组**（比如把"三重"删掉、把"增伤"换成"加速"），**不碰任何代码**，按左键就变成了完全不同的效果。

这就是终点。整套系统存在的意义，就是让上面这段话成立。

现在我们**不从代码开始**，而是反过来问：**要让这个画面发生，背后必须依次发生什么？**

## 第二步：把"效果"拆成几个可观察的子现象

先别碰代码。把上面那个画面，拆成你肉眼能分辨的几件事。每件事，其实对应系统里的一个"机制"：

| 你看到的现象 | 它背后的机制 | 对应的"技术问题" |
|---|---|---|
| ① 按一次键 = 施放一次（有节奏、不会乱发） | 输入 → 入队 → 冷却 | 怎么把"按键"变成"恰好一次施放"？ |
| ② 同时飞出 **3 颗** | 多重（Multicast，预算） | 怎么决定"一次开火产出几发"？ |
| ③ 每颗**更疼** | 修正（Modifier，伤害倍率） | 怎么让"增伤"作用到"它后面的火球"？ |
| ④ 呈**扇形** | 散射（把多发铺开） | 多发同方向会重叠，怎么散开？ |
| ⑤ 朝**准心** | 瞄准（按下瞬间锁存方向） | 怎么知道朝哪打？为什么是"按下那一刻"？ |
| ⑥ 命中敌人**掉血** | 伤害结算（复用已有战斗） | 火球碰到敌人怎么扣血？ |
| ⑦ 改数组就换效果、**不写代码** | 数据驱动 + 涌现 | 为什么换排列就换效果，却不用加代码？ |

**关键洞察**：现象 ②③④⑦ 不是各写一段"三连增伤散射火球"的专属代码。它们是**同一个"从左到右运行法杖"的过程**自然跑出来的。这就是整套系统的灵魂——**解释器，不是查表**（详见解析文档〇节）。先记住这句话，下面倒推时你会反复看到它。

## 第三步：顺着一颗火球，往回追因果链（核心）

现在选现象里最"实"的那颗火球，从它往回追：**这颗火球是谁、凭什么、在什么时候被造出来的？** 每往回一层，就遇到一个组件 + 一个技术问题。

### 追问 1：屏幕上这颗火球实体，是谁 `Instantiate` 出来的？
→ **`SpellCaster`**（挂在法师身上的组件，`Game.Character/Spells/SpellCaster.cs`）。
它的 `CastWand(...)` 负责把"该产出什么"变成"真实的飞行物"：`Instantiate` 预制体 → 设方向/速度 → 调 `ProjectileBase.Init(伤害, 阵营, 速度…)`。
- **技术问题**：谁来把"逻辑结果"落地成"Unity 物体"？为什么需要单独一个组件干这事？
- **答案要点**：因为"算什么"和"在引擎里造什么"要分开（见追问 2 的伏笔）。`SpellCaster` 是这两层之间**唯一的桥**。

### 追问 2：`SpellCaster` 凭什么知道要造"3 颗增伤散射火球"？
→ 它手里拿到了一组**产出指令** `List<EmitCommand>`。每条 `EmitCommand` 是一份纯数据：用哪个预制体、最终伤害多少、速度多少、散射角多少。
- **技术问题**：为什么不让 `SpellCaster` 自己一边算一边造，而要先拿到一份"指令清单"？
- **答案要点**：**关注点分离**。"算出该产出什么"是纯逻辑（不依赖 Unity、可单元测试）；"把它造出来"才依赖 Unity。把前者的产物固化成 `EmitCommand` 数据，后者只负责消费。**这是整套设计最重要的一刀**（解析文档"子功能五"）。

### 追问 3：这组 `EmitCommand` 是谁算出来的？
→ **`CastEvaluator`**（纯静态求值器，`Game.Skills/CastEvaluator.cs`）。
`Evaluate(法术序列, 基础预算, 法力, 起始修正, 输出列表)` 从左到右"运行"法杖，把结果填进输出列表。
- **技术问题**：它怎么从 `[三重, 增伤, 散射, 火球, 火球, 火球]` 这串东西，算出"3 条都带增伤、带散射角的火球指令"？
- **答案要点**：一个**单遍循环 + 三条规则**（这就是②③④的来源）：
  - 读到 **三重** → 把"投射物预算"+2（从 1 变 3）。→ 决定了"出 3 颗"（现象②）
  - 读到 **增伤** → 把"当前修正状态"的伤害倍率 ×1.5。→ 之后产出的都更疼（现象③）
  - 读到 **散射** → 把"当前修正状态"的散射角 +20°。→ 之后产出的带散射（现象④的数据来源）
  - 读到每个 **火球** → 若预算>0：产出一条 `EmitCommand`（把"基础值 × 当前修正快照"算成最终伤害/速度/散射），预算 -1。
  - 预算耗尽 → 停。
- **为什么"增伤"要放在"火球"前面？** 因为是从左往右读、修正只影响"它之后"的产出。这条直觉就是"状态从左累积、产出时快照"的自然结果。

### 追问 4：`[三重, 增伤, 散射, 火球…]` 这串数据本身从哪来？
→ **`WandLoadout`**（一个 ScriptableObject 资产，`Game.Skills/WandLoadout.cs`），里面就是一个 `SpellDefinition[]`（法术数组）+ 基础预算。
每个 **`SpellDefinition`** 也是一个资产，描述"这条法术是什么"（是产出/修正/多重？参数多少？用哪个预制体？）。
- **技术问题**：为什么用资产（数据）而不是写死在代码里？
- **答案要点**：**数据驱动**。法杖 = 一个可编辑的数组资产；改数组就改了"程序"，求值器照跑——所以现象⑦"改数组换效果、不写代码"才成立。新增一个法术 = 新建一个 `SpellDefinition` 资产，零代码。

### 追问 5：到底是谁、在什么时候，启动了上面这一切？
→ 一条输入链（`Game.Character`）：
1. **`WizardController.UpdateAttackInput()`**（每帧常驻）：你**按下左键那一帧**，立刻用屏幕中心射线锁存"准心瞄准点"`ClickAimPoint`（现象⑤的"按下那一刻"），并把这次施放**入队**。
2. **`TryStartAttack()`**：若已入队且射速冷却已过 → 进入施法状态（现象①的"恰好一次、有节奏"）。
3. **`PlayerWizardAttackState`**：播放挥杖**动画**，当动画播到设定的时间点（`ArrowSpawnTime`）→ 调 `ReleaseCast()` → 把 `ClickAimPoint`、阵营等传给 `SpellCaster.CastWand(...)`。
- **技术问题**：为什么瞄准方向要在"按下那一刻"锁存，而不是"火球飞出那一刻"再读？
- **答案要点**：挥杖有前摇（动画要播到 0.x 秒才发射）。若发射时才读相机，你在前摇里转动轨道相机就会让火球飞偏。锁"按下瞬间"= 所见即所得。

### 追问 6：火球飞出去，碰到敌人怎么就掉血了？
→ 这一步**完全复用项目已有的战斗系统**，法术系统没另造：火球预制体上的 **`ProjectileBase`**（`Game.Combat`）在 `OnCollisionEnter` 时构造一个伤害请求，调敌人身上的 `IDamageable.ReceiveHit`，由 `HealthComponent` 扣血并发事件（血条 UI 据此更新）。
- **技术问题**：法术系统要不要重写一套伤害？
- **答案要点**：不要。`SpellCaster` 造出来的就是现有 `Fireball` 预制体，命中走现有那条伤害管线。**复用而非重造**，是这次接运行时最省力也最稳的决定。

## 第四步：把链条正过来看（一次开火的完整数据流）

上面是"倒着追因果"。现在把它正过来，就是**一次开火从头到尾的数据流**——这也是你**读代码时该按的顺序**：

```
[你按下左键]
   │
   ▼  WizardController.UpdateAttackInput()      ← 锁存 ClickAimPoint（准心方向）+ 入队
   │
   ▼  WizardController.TryStartAttack()          ← 冷却 OK → FireQueuedCast() → 进施法态
   │
   ▼  PlayerWizardAttackState（播放挥杖动画）    ← 动画到 ArrowSpawnTime → ReleaseCast()
   │
   ▼  SpellCaster.CastWand(生成点, 准心点, 阵营, 攻击者, 碰撞体)
   │        │
   │        ▼  CastEvaluator.Evaluate(法杖序列, 预算, 法力, 起始修正, 输出)   ← 纯逻辑
   │        │        读 [三重(+2) 增伤(×1.5) 散射(+20°) 火球 火球 火球]
   │        │        → 产出 3 条 EmitCommand（各含最终伤害/速度/散射角）
   │        │
   │        ▼  遍历 EmitCommand：
   │             SpellAiming.SpreadOffsetDegrees(i, 3, 20°)  → 第 i 发的偏航角（-10/0/+10 之类）
   │             Quaternion.AngleAxis(偏航, up) * 准心方向    → 这一发的实际方向
   │             Instantiate(火球预制体) → ProjectileBase.Init(...)
   │
   ▼  3 颗火球飞出（扇形、增伤、朝准心）
   │
   ▼  命中敌人 → ProjectileBase.OnCollisionEnter → IDamageable.ReceiveHit → 扣血（已有系统）
```

把这张图和上面的"倒推问答"对照着看，你就有了"地图 + 路线"。

## 第五步：从哪儿下手学（学习路线）

你有两种顺序，目的不同，建议这样组合：

**A. 先"理解"——用本文倒推（已完成）**
从效果倒推一遍，建立"哪个环节负责哪个现象"的整体感。看不懂某个环节的细节时，去查解析文档对应子功能的 7 维度讲解。

**B. 再"读代码"——按数据流正序读（第四步那张图的顺序）**
依次打开这几个文件，边读边对照图：
1. `Game.Character/Controllers/WizardController.cs` —— 输入怎么变成"一次施放"（看 `UpdateAttackInput` / `TryStartAttack` / `FireQueuedCast`）
2. `Game.Character/States/PlayerWizardAttackState.cs` —— 动画时机怎么触发释放（看 `HandleCastRelease` / `ReleaseCast`）
3. `Game.Character/Spells/SpellCaster.cs` —— 数据怎么变成真实投射物（看 `CastWand`）
4. `Game.Skills/CastEvaluator.cs` —— **核心**：法杖怎么被"运行"成 `EmitCommand`（看 `Evaluate` 那个循环）
5. `Game.Skills/SpellDefinition.cs` / `WandLoadout.cs` / `EmitCommand.cs` / `CastModifierState.cs` —— 各种数据长什么样
6. `Game.Skills/SpellAiming.cs` —— 扇形角度怎么算

**C. 想"动手重写/造类似系统"——自底向上建（我们当初的开发顺序）**
和读代码相反，**动手要从最纯、最不依赖别人的那层开始**，因为它能单独测、不卡在 Unity 上：
1. 先建**数据词汇**（`SpellKind`/`SpellDefinition`/`EmitCommand`/`CastModifierState`）——没有依赖。
2. 再写**求值器** `CastEvaluator`——只依赖词汇，纯函数，**立刻写 EditMode 测试**验证"增伤在前生效/在后不生效""三重出几发""法力不足 fizzle"。这一层不进 Play 都能验证对错。
3. 然后写**桥** `SpellCaster` + 纯函数 `SpellAiming`——把求值结果落地成真实投射物。
4. 最后**接输入**——把法师左键改成调 `SpellCaster`。
> 记住这条原则：**先做"能脱离引擎验证"的纯逻辑（可测、稳），再做"必须靠引擎"的副作用（实例化、动画、输入）。** 这是整套开发最值钱的方法论。

## 第六步：亲手改一下（巩固理解的小练习）

进 Play 前先在脑子里**预测**，再改 `WandLoadout.Spells` 验证：

| 把法杖改成 | 预测会发生什么 | 为什么（追到哪一环） |
|---|---|---|
| `[火球]` | 1 颗普通火球朝准心 | 预算 1、无修正 |
| `[增伤, 火球]` | 1 颗更疼的火球 | 修正在产出前累积 |
| `[火球, 增伤]` | 1 颗**普通**火球（增伤无效） | 修正只影响"其后"——顺序敏感 |
| `[三重, 火球]` | 仍只 1 颗（预算 3 但只有 1 个火球） | 单遍不回绕、预算填不满即作废 |
| `[三重, 火球, 火球, 火球]` | 3 颗 | 3 个产出物刚好填满预算 3 |
| `[三重, 散射, 火球, 火球, 火球]` | 3 颗扇形 | 散射给修正状态加了散射角 |
| 把"增伤"换成"加速" | 火球飞更快而非更疼 | 改的是修正状态的不同字段 |

能准确预测，就说明你真的把这条链路打通了。

## 一句话总结这条主线

> 你按左键 → **控制器**锁准心、入队、按节奏进入施法态 → **施法动画**到点 → **SpellCaster** 拿着法杖去问 **CastEvaluator**："这串法术该产出什么？" → 求值器从左到右跑一遍**规则**（多重扩预算、修正改状态、产出读快照）吐出一组 **EmitCommand** → SpellCaster 按这些指令、用 **SpellAiming** 散成扇形、**Instantiate** 出真实火球 → 火球走**已有战斗系统**命中扣血。
>
> 而"换法杖数组就换效果、不写代码"之所以成立，是因为复杂度被收进了**少数几条求值规则**里，而不是散在"每一种组合"上——这就是**解释器**的胜利。
