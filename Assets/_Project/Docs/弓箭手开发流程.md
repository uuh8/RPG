# 弓箭手开发流程 — 从零到完成的时间线（配套阅读指南）

> 本文是 `Archer_System_Design.md`（按功能切开的六维解析）的**配套总览**。
> 那篇文档把弓箭手拆成 9 个独立子功能逐个深挖，好处是讲得透，坏处是**缺少"先后顺序"**——你不知道该从哪读、这些功能是按什么次序做出来的、为什么是这个次序。
>
> 这篇就补这个：用**开发时间线**把所有子功能重新串起来，回答三个问题——
> 1. 整个弓箭手是**按什么流程**一步步做出来的？
> 2. 每一步**为什么必须排在这个位置**（依赖关系）？
> 3. 我该**按什么顺序阅读**那篇解析文档？
>
> 读完这篇，你再回去读 `Archer_System_Design.md` 的各个子功能，就有了"地图"。

## 一、先理解开发哲学：为什么要"分阶段 + 每阶段先想后做"

在写任何代码之前，先记住本项目做复杂功能的**统一节奏**。弓箭手不是"一口气写完"的，而是切成若干**阶段（Phase）**，每个阶段都走同一套五步循环：

```
头脑风暴(brainstorm) → 设计文档(spec) → 实现计划(plan) → 执行(execute) → 在 Editor 里验证(verify)
        │                  │                │              │                  │
   想清楚要什么        定方案与取舍       拆成可执行步骤    写代码          开发者亲手玩一遍
        └──────────────────┴────────────────┴──────────────┴── 通过了才进下一阶段 ──┘
```

**为什么要这样，而不是直接开干？**

1. **复杂功能直接写必然返工。** 弓箭手牵涉架构（多角色）、物理（箭矢）、输入（点按/按住）、相机（瞄准）、表现（准心）……一次性想全不可能。切成阶段，每阶段只解决一类问题，脑子才装得下。

2. **每阶段先想后做，是为了"把贵的错误提前到便宜的时候犯"。** 改一句设计文档的话 5 秒钟；改已经写好、还串进别处的代码可能 1 小时。**头脑风暴和写计划，本质是用"想"的成本换"返工"的成本。**

3. **每阶段都要"在 Editor 里亲手验证通过"才进下一阶段（gate，关卡）。** 因为后一阶段会站在前一阶段的肩膀上——地基没验稳就盖二楼，塌了都不知道是哪层的锅。所以每过一关，开发者都进 Play 模式实测：战士还能不能正常打？弓箭手能不能走/跳/冲？这一关的新功能对不对？

> 关键分工：**Claude（我）负责写 `.cs` 和文本配置、写文档；编译、Animator 连线、Play 模式实测由开发者（你）在 Editor 里做。** 我只能保证"代码逻辑上正确"，不能替你按下播放键。所以每阶段末尾的"验证"是你的活，也是关卡。

下面就按真实的开发时间线，一阶段一阶段走。

## 二、开发时间线（按真实顺序）

### 阶段 0：动手前的关键决策（头脑风暴）

在写第一行代码前，先做了几个**岔路口选择**。这些选择决定了后面所有阶段的形状：

| 岔路口 | 选项 | 选了哪个 / 为什么 |
|---|---|---|
| 两个角色怎么组织代码？ | 复制粘贴 / 继承 / 组合 / 泛型 | **继承（基类+子类）**：角色少、差异是"攻击不同"这种行为型差异，继承最省力 |
| 弓箭手攻击态用什么类？ | 复用战士的 `PlayerAttackState` / 平行写新类 | **平行新类**：远程"命中发现"方式（箭矢）和近战（盒子）根本不同，硬复用会拧巴 |
| 点按/按住怎么分辨？ | 纯缓冲 / 手动计时阈值 / Input System 交互 | **手动计时 + 阈值门控**：要"按住到点立刻拉弓"的即时手感，且边界完全可控 |
| 瞄准怎么做？ | 轻量(不换相机) / 专用瞄准相机 | **轻量**：先拿到"指哪打哪"的核心手感，不背相机系统的复杂度 |

**这一步的产物不是代码，是"想清楚了"。** 对应解析文档里每个子功能的「第 3 节：实现方案有哪些 + 选型理由」——那些就是这里做的决策的记录。

---

### 阶段 1：搭地基 —— 抽出共享基类（架构）

**做了什么：** 把原来只属于战士的 `PlayerController.cs` 拆成两层：共享的 `PlayerControllerBase`（抽象基类，装走/跳/冲刺/相机/状态机）+ `WarriorController`（战士子类）+ `ArcherController`（当时还只是空壳子类）。并埋下**攻击接缝** `TryStartAttack()`。

**为什么排第一：** 这是**所有后续阶段的地基**。弓箭手的走、跳、冲刺全靠继承这个基类白拿；攻击逻辑全靠那个接缝挂进去。地基不先建好，后面无从谈起。

**依赖谁：** 不依赖弓箭手任何东西，反而是它**反向要求**"不能改坏战士"（硬约束）。

**对应解析文档：** 👉 **子功能一（多角色架构）**，以及 **子功能二（数据驱动 Animator 状态名）** 的一半（这阶段把冲刺动画名 `_dashStateName` 改成了数据驱动）。

**这一关的验证：** 战士进 Play，走/跳/冲刺/连段**和重构前一模一样**——零行为变化才算过关。

> ⚠️ 这一阶段踩了两个最值得记的坑：① 改脚本名必须 `.cs` 和 `.meta` 一起 `git mv` 保住 GUID，否则战士预制体引用全丢；② "抽基类"和"把状态的 `_player` 改成基类类型"是一次不可分割的编译改动。详见子功能一第 6 节。

---

### 阶段 2：让弓箭手动起来 —— locomotion（动画接线）

**做了什么：** 给阶段 1 那个"空壳" `ArcherController` 接上弓箭手自己的一套动画（`BowHero` Animator Controller：`Idle_Bow`/`Run_Bow`/`Dash_Bow`/`Jump*_Bow`…）。这阶段**几乎没写代码**，主要是在 Animator 窗口连移动动画的过渡线 + 把 `ArcherController` 挂到 BowHero 预制体上、填好各种 Inspector 引用。

**为什么排第二：** 攻击之前，得先有个"能在场景里跑来跑去的弓箭手"。这是把阶段 1 的共享能力**第一次真正落到弓箭手身上**——验证"继承的那套移动逻辑，配上弓箭手的动画，能不能跑通"。

**依赖谁：** 依赖阶段 1 的基类（移动逻辑）和它的数据驱动动画名机制。

**对应解析文档：** 👉 **子功能二（数据驱动 Animator 状态名）**——尤其是那条铁律：**进入靠代码 CrossFade、退出靠 Animator 连线**。

**这一关的验证：** 弓箭手能走/跑/跳/冲，动画都对。

> ⚠️ 这阶段冒出"冲刺只平移没动画"的坑：`_dashStateName` 还留着战士的默认值，CrossFade 到弓箭手没有的状态 = 静默失效。修复就是 Inspector 里填 `Dash_Bow`。这正是"数据驱动 + 静默失败"的典型代价（详见子功能二第 6 节）。

---

### 阶段 3：第一种攻击 —— 普通射击 + 箭矢系统（玩法核心）

**做了什么：** 两件事一起做——
1. **箭矢本身**：新增 `Arrow.cs`（带 Rigidbody 的飞行投射物，撞到敌人走伤害管线、撞环境销毁、同阵营穿过）。
2. **普攻状态**：新增 `PlayerBowAttackState.cs`（按 `ArcherController` 的接缝进入；播放攻击动画，到 `ArrowSpawnTime` 那一刻生成一支箭）。

**为什么排第三：** 这是弓箭手**第一次有"攻击"**，也是把阶段 0 那些攻击相关决策（平行新类、箭矢投射物）第一次变成代码。它必须排在 locomotion 之后（攻击要从"地面态"经接缝进入），且必须在蓄力之前（蓄力是普攻的"加强变体"，共用箭矢系统和单点生成机制——先有简单版才好做复杂版）。

**依赖谁：** 依赖阶段 1 的接缝 `TryStartAttack()`（攻击怎么挂进优先级链）；依赖既有的伤害管线 `IDamageable.ReceiveHit`（箭命中后怎么算伤害，直接复用，不重写）。

**对应解析文档：** 👉 **子功能三（箭矢投射物）** + **子功能四（动画驱动的单点时序）**。

**这一关的验证：** 点左键能射出箭、能打到敌人扣血。

> ⚠️ 这阶段的坑：箭射出去是"竖着"飞的——模型箭尖沿 +Y、而 `LookRotation` 对齐 +Z。加一个修正旋转 `_modelForwardOffsetEuler` 解决（详见子功能三第 6 节）。

---

### 阶段 4：第二种攻击 —— 蓄力重击（输入 + 数值）

**做了什么：** 新增 `PlayerChargeAttackState.cs` + `ChargeAttackDefinition.cs`（数据 SO）。在 `ArcherController` 的接缝里加**输入路由**：点按→普攻、按住过阈值→蓄力。蓄力越久，伤害/箭速越高（线性映射）。

**为什么排第四：** 它是普攻的"豪华版"——**复用**阶段 3 的箭矢系统和单点生成（`ArrowSpawnTime`），只是多了"按住蓄力→松手放箭"的输入流程和"蓄力程度→数值"的换算。所以必须站在阶段 3 的肩膀上。

**依赖谁：** 依赖阶段 3 的 `Arrow`（蓄力箭也是它）、单点生成机制；依赖阶段 1 暴露的 `IsAttackHeld`（轮询按住）。

**对应解析文档：** 👉 **子功能五（点按/按住输入路由）** + **子功能六（蓄力→伤害/箭速线性映射）**。

**这一关的验证：** 轻点出普攻、按住出蓄力，蓄越久箭越疼越快。

---

### 阶段 5：让蓄力"指哪打哪" —— 瞄准 + 准心（手感 + 表现）

**做了什么：** 给阶段 4 的蓄力加**瞄准**——蓄力时角色转向相机朝向、显示准心；松手时用"屏幕中心射线"求命中点，箭走直线精确命中准心点（关重力）。新增 `AimStateChangedEvent`（事件）+ `CrosshairUI`（准心 UI 订阅事件显隐）。

**为什么排第五：** 它是蓄力的"精准化"升级，必须先有阶段 4 的蓄力流程才能往上加瞄准。而且这一阶段第一次引入"表现层"（准心 UI），用到了事件解耦——它逻辑上依赖蓄力，物理上却通过事件和蓄力解耦。

**依赖谁：** 依赖阶段 4 的蓄力状态（瞄准挂在蓄力的生命周期上：Enter 显准心、Exit 隐准心）；依赖阶段 3 埋的 `useGravity` 开关（直射 vs 抛物）。

**对应解析文档：** 👉 **子功能七（屏幕中心射线瞄准）** + **子功能八（准心 UI 事件解耦）**。

**这一关的验证：** 蓄力时准心出现、角色转向相机方向、箭命中准心点。

> ⚠️ 这阶段最核心的 bug：蓄力箭往身后飞——轨道相机在角色身后，屏幕中心射线先撞到角色自己，命中点落在身体上（在弓弦生成点后方），方向算反。修复是用 `RaycastNonAlloc` 取全部命中、跳过自身再取最近（详见子功能七第 6 节）。

---

### 阶段 6：手感打磨 —— bug 修复（边走边射 + 方向修正）

**做了什么：** 完整功能跑通后，实测发现两个手感问题，针对性修：
1. **普攻不能移动**（"边走边射"）：普攻状态当时照抄了近战的"锁脚"，于是射箭时走不动。改成完整移动 + 转向。
2. **蓄力箭方向反**（即阶段 5 那个坑的最终修复）。

**为什么排最后：** 这些是"功能都在了，但玩起来不对"的体验问题，只有等整个链路跑通、亲手玩过才会暴露。**实测 → 发现手感问题 → 回头修**，是开发的常态尾声。

**对应解析文档：** 👉 **子功能九（边走边射）**，以及子功能七第 6 节的方向修复。

---

### 阶段 7（本次新增）：相机取景增强 —— 角色让位

**做了什么：** 蓄力瞄准时让角色滑到画面左下、露出准心（仿原神）。新增 `AimFramingController.cs`，**复用阶段 5 早就发布的 `AimStateChangedEvent`**，gameplay 代码一行没改，只多挂一个相机表现监听者。

**为什么能这么轻松加：** 因为阶段 5 已经把"瞄准开始/结束"广播成了事件。新表现只要订阅它即可——**这就是事件解耦在阶段 5 埋下、在阶段 7 收获的红利**。

**对应文档：** 👉 单独的 `Archer_Aim_Framing_Design.md`。

## 三、一张图看懂"为什么是这个顺序"（依赖关系）

每个阶段都**站在前面阶段的肩膀上**，箭头表示"依赖/复用"：

```
阶段1 架构地基(基类+接缝+数据驱动动画名)
   │  提供：移动/跳跃/冲刺/状态机/TryStartAttack 接缝
   ▼
阶段2 弓箭手 locomotion(接动画)
   │  验证：基类能力落到弓箭手身上能跑通
   ▼
阶段3 普攻 + 箭矢系统 ──────────────┐
   │  提供：Arrow 投射物 / 单点生成 / useGravity 开关
   ▼                                  │ 复用
阶段4 蓄力重击(输入路由+线性数值) ◄──┘
   │  提供：蓄力状态生命周期
   ▼
阶段5 瞄准 + 准心(射线+事件解耦)
   │  提供：AimStateChangedEvent 广播
   ├──────────────► 阶段7 相机取景增强(订阅同一事件)
   ▼
阶段6 手感修复(边走边射 / 方向修正)
```

**三条主干依赖，记住就懂了顺序：**
1. **一切先有地基**（阶段 1）：没有基类和接缝，弓箭手连走路都没有。
2. **先能动、再能打**（2 → 3）：攻击要从"地面态"经接缝进入，所以移动必须先通。
3. **先简单、再复杂**（3 → 4 → 5）：普攻 < 蓄力 < 瞄准蓄力，后者复用前者的箭矢系统、单点生成、状态生命周期，层层加码。

## 四、阅读地图：建议你按这个顺序看

你现在面对 `Archer_System_Design.md` 里 9 个子功能不知从何下手——按下面这条线读，**和开发顺序一致，每篇都为下一篇铺垫**：

**第一站：建立全局直觉（5 分钟）**
- 先读 `Archer_System_Design.md` 开头的「全局心智模型」那张数据流图。看不懂细节没关系，建立"按下左键之后发生了什么"的直觉。

**第二站：地基（架构，必读，最重要）**
- 👉 子功能一（多角色架构）——理解"基类+子类+接缝"。这是整个弓箭手的骨架，不懂它后面都悬空。
- 👉 子功能二（数据驱动 Animator 状态名）——理解"代码怎么不写死动画名地驱动动画"。

**第三站：玩法核心（攻击怎么打出去）**
- 👉 子功能三（箭矢投射物）——远程攻击的"载体"。
- 👉 子功能四（单点时序）——箭"什么时候"生成。
- 这两篇连着读，是"普攻"的完整拼图。

**第四站：进阶攻击（蓄力）**
- 👉 子功能五（点按/按住输入路由）——一个键怎么两用。
- 👉 子功能六（蓄力→数值映射）——蓄越久越强怎么算。

**第五站：手感与表现**
- 👉 子功能七（屏幕中心射线瞄准）——"指哪打哪"。
- 👉 子功能八（准心 UI 事件解耦）——表现层怎么不污染玩法。
- 👉 子功能九（边走边射）——"动画在播 ≠ 角色在动"这个易错概念。

**第六站：相机增强（最新）**
- 👉 单独读 `Archer_Aim_Framing_Design.md`。

**最后：回到套路**
- 读 `Archer_System_Design.md` 末尾的「贯穿全程的 10 条设计套路」。读完前面所有子功能再看这 10 条，会有"原来都是同一招"的恍然大悟感。这 10 条才是最该带走、能迁移到下一个角色的东西。

## 五、一句话总结这套流程

> **先想清楚要什么（头脑风暴）→ 定好方案和取舍（设计文档）→ 拆成可执行步骤（计划）→ 写代码（执行）→ 亲手玩一遍验证（关卡）；复杂功能切成阶段，每阶段站在前一阶段肩上，先有地基再盖楼，先能动再能打，先简单再复杂，每过一关才进下一关。**

把这套节奏记住，比记住任何一段弓箭手代码都值钱——它是做**任何**复杂功能都能套用的工作流。
