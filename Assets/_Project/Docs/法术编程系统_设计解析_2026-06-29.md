## 〇、总览：这套系统是什么，以及最重要的一个认知

### 它解决什么问题
传统技能系统 = 一堆写死的技能按钮（Q/E/R），想加新效果就得程序员再写一个技能。我们做的是 **Noita（女巫）式的"法杖编程"**：把若干"法术积木"（火球、增伤、三重、散射…）按顺序摆进一根"法杖"，按一次左键，系统从左到右"运行"这串法术，**组合**出花样百出的效果。换个排列 = 换个效果，**不需要程序员加代码**。

### 全系统一句话
> **法杖是一段"程序"，法术是"指令"，开火 = 让一个"解释器"从左到右求值这段程序，产出投射物。**

### 最重要的认知：解释器（interpreter），不是查表（lookup table）
这是整套系统的"命门"，新手最容易理解错的地方：

- ❌ **查表式**（错误心智）：以为"A+B = 触发隐藏技能 C"，每个组合手写一条规则。N 个法术要维护约 N² 张组合表，**组合爆炸、不可扩展**。
- ✅ **解释器式**（我们做的）：只定义 **少数几类指令的"求值规则"**（产出 / 修正 / 多重 / 法力），所有花样都是"从左到右求值"这一个过程**自然涌现**的副产品。加一个新法术 = 加一条数据，**零新规则**。这才是 Noita 能有 400+ 法术却不崩的原因。

举例：法杖 `[三重, 增伤, 火球]` → "3 颗都带增伤的火球"。**没有人写过"三连增伤火球"这个技能**，它是 `三重`（预算+2）+ `增伤`（改修正状态）+ `火球`（产出，读当前修正）三条规则组合跑出来的。

> 💡 这套设计的简历含金量不在"我做了很多技能"，而在"我把一个看似复杂的涌现系统，抽象成了一个干净、可单元测试的解释器内核"——和项目里 `DamagePipeline`/`ComboResolver` 那种"纯函数、可单测"是同一种工程品味。

### 开发顺序（本文的拆解顺序）
分两大阶段，每阶段几个子任务，**严格按依赖顺序**：

```
阶段 A（纯逻辑内核，可单测，不碰 Unity）
  ① 法术数据词汇   ② 求值器内核(产出+修正+预算)   ③ 多重   ④ 法力/fizzle
阶段 B（接运行时，复用现有投射物，法师能真打出来）
  ⑤ 纯逻辑↔实例化分离(SpellCaster 桥)   ⑥ 多发散射(纯函数)   ⑦ 接入法师输入(左键运行法杖)
```

## 一、法术数据词汇（SpellKind / SpellDefinition / EmitCommand / CastModifierState）

### 1. 需求拆解
解释器要"运行"法术，第一步得先定义**法术是什么数据**。我们需要：一个能描述任意法术的数据容器（投射物？修正？多重？各自参数不同）、一个"求值时累积的修正状态"、以及一个"求值产出什么"的结果数据。

### 2. 基础知识与原理
- **数据驱动（Data-Driven Design）**：游戏数据放在 ScriptableObject（Unity 的可序列化数据资产），不写死在 MonoBehaviour 里。新增内容 = 新建资产，不改代码、不重新编译。
- **值类型 vs 引用类型**：`struct`（值类型）按值拷贝、在栈上、零 GC；`class`（引用类型）在堆上、会触发垃圾回收。求值要频繁传"修正状态"，用 `readonly struct` 避免堆分配。
- **不可变（immutability）**：`readonly struct` 的字段只读，"修改"= 返回一个新值。这让"快照"（snapshot）天然安全——产出投射物时把当前修正拷一份钉死给它，之后再改修正也不影响它。

### 3. 业界常见方案
- **Unreal GAS（Gameplay Ability System）**：用 `GameplayEffect` 数据资产 + `GameplayTag` 描述能力，高度数据驱动——同样是"数据定义 + 运行时解释"。
- **ScriptableObject 架构**（Ryan Hipple, Unite 2017）：Unity 社区主流的"用 SO 当数据/事件容器"范式，本项目全程沿用。
- **组件化效果（effect composition）**：把一个技能拆成若干可组合的小 effect 数据，运行时叠加——和我们"法术 = 指令"同源。

### 4. 最终用到的技术
- `SpellKind`（`enum : byte`）：求值器眼里的三种语义 `Emit / Modify / Multicast`。
- `SpellDefinition : ScriptableObject`：一个法术的全部数据，字段按 Kind 分组（Emit 用 `ProjectilePrefab/BaseDamage/BaseSpeed/DamageType`；Modify 用 `ModDamageMul/ModSpeedMul/ModDamageAddFlat/ModSpreadAddDegrees`；Multicast 用 `ExtraDraws`）。带 `[CreateAssetMenu]` 可在编辑器右键新建；预留 `Sprite Icon`（仿 Noita 的技能图标，暂可空）。
- `CastModifierState`（`readonly struct`）：求值时"当前累积的修正"，含 `Default`（恒等：加 0、乘 1）和 `Apply(modify)`（叠加一个修正法术 → 返回新状态）。
- `EmitCommand`（`readonly struct`）：求值输出的一条"该产出什么"的纯数据（预制体 + 最终伤害/速度/类型/散射角）。

### 5. 架构设计与拆解（新增脚本）
全部放在 `Game.Skills` 程序集（依赖 Core + Combat，不碰角色/UI）：
- `Assets/_Project/Scripts/Skills/SpellKind.cs`
- `Assets/_Project/Scripts/Skills/SpellDefinition.cs`
- `Assets/_Project/Scripts/Skills/CastModifierState.cs`
- `Assets/_Project/Scripts/Skills/EmitCommand.cs`

**关键决策**：为什么"玩家看到 4~8 类法术（投射物/静态投射物/修正/多重/触发…），求值器只认 3 种语义"？因为"火球"和"黑洞"在求值器眼里都是 `Emit`——它俩区别（一个飞、一个不飞）是各自**预制体**的事，跟"运行程序"的逻辑无关。**把玩家可见的丰富分类，收敛成内核的少数语义**，是这套设计能小而干净的根本。

### 6. 踩坑与 Debug
- **恒等单位元的坑**：`CastModifierState.Default` 必须是"加法项 0、乘法项 1"。如果把 `DamageMul` 默认成 0，所有伤害会被乘没。同理一个"只改伤害的修正"，它的 `ModSpeedMul` 必须默认 1（而不是 0），否则会把速度乘成 0。`SpellDefinition` 的修正字段初值都设成恒等（Mul=1、Add=0）就是为此。
- **ScriptableObject 能不能在 EditMode 测试里造？** 能——`ScriptableObject.CreateInstance<SpellDefinition>()`。所以"求值器吃 SO 数组"依然可单测（测试跑在 Unity 里，能造 SO）。

### 7. 面试 Q&A
- **Q：为什么用 `readonly struct` 而不是 `class` 装修正状态？**
  A：修正状态在求值循环里频繁按值传递、还要给每个产出物做快照。`readonly struct` 在栈上、零 GC、天然不可变——快照就是一次值拷贝，安全且无堆分配；用 class 会制造大量短命堆对象、触发 GC，且快照得手动深拷贝防别名。
- **Q：玩家有那么多法术分类，为什么内核只有 3 种 Kind？**
  A：内核只关心"求值控制流"——产出、改状态、扩预算。产出物之后飞不飞、爆不爆、引不引燃，是产出物自己的运行时行为（预制体的事），求值器一律当 `Emit`。把可见分类收敛成少数语义，复杂度被收进"少量规则"而非散落在"每个效果"里。

## 二、求值器内核：产出 + 修正 + 预算（CastEvaluator）

### 1. 需求拆解
有了词汇，现在要"从左到右运行法杖"：读到修正就累积、读到投射物就产出（带上当前修正快照）、并且"一次开火产出几发"要可控。输出一组 `EmitCommand`。

### 2. 基础知识与原理
- **解释器模式（Interpreter Pattern, GoF）**：给一种"语言"（这里是法术序列）写一个求值器，逐条指令求值。
- **程序计数器 / 顺序求值**：从左到右遍历，维护"当前状态"（修正 + 预算）。
- **快照（snapshot）**：产出投射物的瞬间，把当前修正状态拷一份钉死——所以"修正只影响它后面、且本次开火内的产出"，这条直觉就是"状态从左往右累积、emit 时快照"的自然结果。

### 3. 业界常见方案
- **解释器**（直接遍历数据结构求值）—— 我们用的，最直接。
- **字节码 / 虚拟机**（《Game Programming Patterns》Bytecode 章）：把法术编译成字节码，跑一个小 VM。更适合超大规模/需要保存编译结果的场景，对我们是过度设计。
- **行为树 / 节点图**：可视化连线。适合 AI 决策，不适合"线性序列求值"。

### 4. 最终用到的技术
- `CastEvaluator`（`static` 纯类）：`Evaluate(spells, baseDraws, availableMana, incomingMods, output) → CastSummary`。
- 单遍 `for` 循环 + `switch(spell.Kind)`：Modify → `mods = mods.Apply(spell)`；Emit → 若预算>0 则 `output.Add(BakeEmit(spell, mods))` 且预算--。
- `BakeEmit`：把基础值×修正快照算成最终值（伤害 `(BaseDamage+DamageAddFlat)*DamageMul`、速度 `BaseSpeed*SpeedMul`、散射 `mods.SpreadDegrees`）。
- 产出列表 `List<EmitCommand>` 由调用方预分配传入、方法内 `Clear()` 复用（零每帧 GC）。

### 5. 架构设计与拆解（新增脚本）
- `Assets/_Project/Scripts/Skills/CastEvaluator.cs`（含返回摘要 `CastSummary` struct）。

**关键决策——签名一次立对，为"触发递归"留位**：`Evaluate` 从第一天就带一个 `incomingMods` 参数（本次求值的"起始修正状态"）。当前顶层调用传 `CastModifierState.Default`；**后期做"触发"法术（命中时再放一段子法术）时，只需在命中点用快照当起点再调一次 `Evaluate`，不用改求值器主体**。这就是"前期不实现、但签名先留位"——避免后期返工。

### 6. 踩坑与 Debug
- **纯函数 vs 实例化的矛盾（最关键的一次设计纠偏）**：spec 初稿一边说"求值器纯逻辑、可单测、对标 DamagePipeline"，一边说"emit 真实投射物"。这俩在 Unity 里**互相矛盾**——`Instantiate` 是 Unity 调用，一旦进求值器就不纯、不可 EditMode 测、也没法干净递归。纠正方案：**求值器只产出 `EmitCommand` 纯数据，真正 Instantiate 交给阶段 B 的 `SpellCaster`**。这一刀让"纯/可测/可递归/Unity 实例化"四个诉求同时成立（详见子功能五）。
- **修正作用域要测边界**：增伤在火球**前**（生效）/ **后**（不生效）必须各写一个断言，锁死"只影响其后"的语义。

### 7. 面试 Q&A
- **Q：为什么把求值器做成 `static` 纯函数，而不是一个 MonoBehaviour？**
  A：它不需要场景、不需要 Unity 生命周期，输入数据进、结果数据出。做成纯函数就能在 EditMode 里毫秒级单测（不进 Play、不建场景），还能被递归调用（触发法术）。这和项目里 `DamagePipeline` 同一种切法——把"能脱离引擎验证的逻辑"和"必须靠引擎的副作用"彻底分开。
- **Q：求值器怎么保证零 GC？**
  A：产出列表由调用方预分配、方法内 `Clear()` 复用；修正状态是 `readonly struct`（栈上）；用索引 `for` 而非 `foreach`（避免迭代器分配）；不用 LINQ、不装箱。施法是离散输入（按一下）触发，本就不是每帧热路径，但仍按零 GC 习惯写。

## 三、多重施法（Multicast）

### 1. 需求拆解
"一次开火产出多发"——双重 / 三重。这是组合涌现最直观的演示（三重 + 增伤 + 火球 = 3 颗增伤火球）。

### 2. 基础知识与原理
- **施法块 + 抽取预算（draw budget）**：一次开火 = 一个"块"，块有一个"投射物预算"，每产出一发消耗 1，预算耗尽就停。多重 = 把预算调大。这是 Noita 求值器最本质的骨架。

### 3. 业界常见方案
- Noita 真实机制更复杂（多重会"向前抽取"并能回绕/再触发，配合法力与施放延迟构成上限）。我们 MVP 选了**单遍读取、不回绕**的简化模型——简单、确定、好测，且避免"多重在短序列上回绕导致指数膨胀"的坑。

### 4. 最终用到的技术
- 在 `Evaluate` 的 `switch` 里加一个分支：`case SpellKind.Multicast: drawBudget += spell.ExtraDraws;`（双重 +1、三重 +2，不消耗预算）。

### 5. 架构设计与拆解（改动脚本）
- 仅改 `CastEvaluator.cs`（switch 加一个 case）。**这正是解释器式设计的红利**：加"多重"这个机制只动了一处求值规则，没碰任何法术定义。

### 6. 踩坑与 Debug
- **"三重 + 1 个火球"该出几发？** 我们的规则是"预算填不满即作废、不回绕"——`[三重, 火球]` 只出 1 发（预算 3 但只有 1 个投射物）；想要 3 发就摆 3 个火球 `[三重, 火球, 火球, 火球]`。这是有意为之的简化（写进了 spec），避免回绕时多重被反复重读导致投射物指数膨胀、爆 GC/帧率。
- **顺序敏感**：`[火球, 三重, 火球]` → 第 1 发出（预算 0）→ 三重（预算+2=2）→ 第 2 发出 → 共 2 发。这种"后置多重再开预算"的行为也写了测试锁死。

### 7. 面试 Q&A
- **Q：加一个全新机制（多重）要改多少地方？**
  A：一处——求值循环里加一个 `case`。法术定义、产出结构、运行时都不用动。这就是解释器式（规则集中）相对查表式（规则散落）的可扩展性优势。
- **Q：为什么不照搬 Noita 的回绕？**
  A：回绕 + 多重再触发会让短法杖产生指数级投射物，需要法力 + 施放延迟 + 硬上限一起兜底，复杂度高。MVP 用"单遍不回绕"换确定性和可测性，把回绕/牌组指针/充能列为后续阶段——scope 控制。

## 四、法力与 fizzle

### 1. 需求拆解
没有资源约束，所有组合都能无脑连发，系统失衡。需要每个法术耗法力，法力不够就中断本次施法（fizzle）。

### 2. 基础知识与原理
- **资源闸门**：法力是平衡"强组合"的唯一手段——越强的组合越贵。
- **提前中断**：求值到某发法力不够时，停止后续产出并标记 `Fizzled`。

### 3. 业界常见方案
- 法力/能量/怒气/冷却是 RPG 通用资源模型。Noita 用"法力上限 + 回复速度"约束连发。

### 4. 最终用到的技术
- `Evaluate` 增加 `availableMana` 参数 + 局部 `manaLeft/manaSpent/fizzled`；Emit 分支里"预算门 → 法力门 → 扣法力/累计 → 产出"；法力不足时 `fizzled=true` 并 `break` 跳出整个读取循环；返回 `CastSummary(manaSpent, fizzled)`。

### 5. 架构设计与拆解（改动脚本）
- 仅改 `CastEvaluator.cs`（替换 `Evaluate` 方法体）。**签名 Task 2 已定死**，本步只加逻辑不改签名——这是"前期把完整签名立对、后期只填实现"的好处。

### 6. 踩坑与 Debug
- **`break` 要跳出的是 for 循环，不是 switch**：法力不足时若只 `break` 出 switch，循环会继续读下一个法术。解法是 switch 里置 `fizzled=true; break;`，switch 之后再 `if (fizzled) break;` 跳出 for。
- **零成本边界**：用 `ManaCost > manaLeft`（严格大于）判定不足——这样"0 费法术 + 0 法力"仍能施放（`0 > 0` 为假）。写了 `ZeroCostSpells_NeverFizzle` 测试锁死。
- **阶段 B 暂未启用法力系统**：`SpellCaster` 先传一个大值 `_availableMana=9999`（不会 fizzle），把资源池/回复/UI 留给后续阶段——`Evaluate` 的法力参数和 `CastSummary` 接口都已就位，将来接上即可。

### 7. 面试 Q&A
- **Q：法力逻辑加进来，改了求值器的对外签名吗？**
  A：没有。`availableMana` 和返回的 `CastSummary` 在更早的任务就定进了签名，本步只填充实现。这是刻意为之——**先把完整签名一次立对，后续任务只加逻辑不改接口**，避免改签名牵连所有调用点。

## 五、纯逻辑 ↔ Unity 实例化的分离（EmitCommand + SpellCaster 桥）

### 1. 需求拆解
求值器算出"该产出什么"（一组 `EmitCommand`），但它不碰 Unity。得有个东西把这些纯数据**变成真实的飞行火球**，打到敌人身上。

### 2. 基础知识与原理
- **关注点分离（Separation of Concerns）**：把"算什么"（纯逻辑、可测）和"在引擎里造什么"（副作用、靠 Unity）分到两层。
- **命令模式（Command Pattern）的影子**：`EmitCommand` 是一条"产出指令"数据，由另一处"执行器"消费。
- **快照式参数注入**：投射物 `Init` 时把伤害/阵营/速度按值传进去，命中时不回查（可能已销毁的）施法者。

### 3. 业界常见方案
- 经典分层：**逻辑层产出意图数据 → 表现/引擎层执行**（如 ECS 里 system 产出 command buffer，引擎层回放）。
- 对象池：高频投射物用池化避免 Instantiate/Destroy 抖动（本 MVP 先用直接 Instantiate，池化是后续优化点）。

### 4. 最终用到的技术
- `EmitCommand`（纯数据，子功能一已建）作为两层之间的"契约"。
- `SpellCaster : MonoBehaviour`（挂法师身上）：`CastWand(spawnPos, aimPoint, team, attackerId, casterCollider)` → 调 `CastEvaluator.Evaluate` → 遍历 `EmitCommand` → `Object.Instantiate(预制体)` → `GetComponent<ProjectileBase>()` → `proj.Init(team, attackerId, cmd.Damage, cmd.DamageType, dir*cmd.Speed, casterCollider, useGravity:false)`。
- **复用现有投射物体系**：产出的就是现有 `Fireball` 预制体，命中走现有 `ProjectileBase` → `IDamageable.ReceiveHit`，不另造轮子。

### 5. 架构设计与拆解（新增脚本 + 程序集改动）
- 新增 `Assets/_Project/Scripts/Character/Spells/SpellCaster.cs`（放 `Game.Character`，因为它要 `Instantiate` 投射物 + 之后被法师调用）。
- 改 `Assets/_Project/Scripts/Character/Game.Character.asmdef`：新增对 `Game.Skills` 的引用（这样 Character 能用求值器/法杖/EmitCommand）。这是 spec 既定的依赖方向：**Character → Skills（角色用法术系统施法）**；Skills 不反向依赖 Character。

**为什么 SpellCaster 放 Character 而不是 Skills？** 因为它必须 `Instantiate`（Unity）+ 后续要和法师 FSM/动画配合。把"纯求值"留在 Skills（可测），把"落地实例化"放 Character（靠引擎），边界清晰。

### 6. 踩坑与 Debug
- **`ProjectilePrefab` 判空**：阶段 A 复审就预见了"配错的 Emit 法术没填预制体 → `Instantiate(null)` 抛异常"。所以 `SpellCaster` 在 Instantiate 前判空、跳过并 `GameLog.Warn`；拿不到 `ProjectileBase` 组件也 `Destroy` 兜底。
- **`CharacterController` 能当 `casterCollider` 传吗？** 能——`CharacterController` 派生自 `Collider`，所以 `proj.Init(..., _player.CharacterController, ...)` 合法（用于忽略与施法者自撞）。
- **`Init` 参数顺序**：复审专门核对了 `ProjectileBase.Init(byte, int, float, DamageType, Vector3, Collider, bool)` 的精确顺序与类型，逐个对齐——跨程序集调用最容易因参数顺序错位出隐性 bug。
- **`useGravity:false`**：用具名参数传，自文档、且免受将来参数位置变动影响。

### 7. 面试 Q&A
- **Q：为什么不让求值器直接 Instantiate，省一层？**
  A：那样求值器就绑死 Unity、不可单测、也没法递归（触发法术）。分层后求值器是纯函数（22 个测试护航），`SpellCaster` 是唯一"数据→引擎"的桥——逻辑变更不动引擎层，引擎层变更不动逻辑层。
- **Q：`SpellCaster` 将来能给敌人复用吗？**
  A：能。它不依赖玩家特有的东西，参数都是外部传入（生成点、瞄准点、阵营、施法者）。敌人也能挂一个 `SpellCaster` 跑自己的法杖——这正是把它做成独立组件而非塞进玩家控制器的价值。

## 六、多发散射（SpellAiming 纯函数）

### 1. 需求拆解
多重一次产出 N 发，如果方向全一样会重叠成一发。需要把它们散成扇形。

### 2. 基础知识与原理
- **均匀分布 + 线性插值（Lerp）**：把 N 发均匀铺在 `[-spread/2, +spread/2]` 的角度区间上。
- **整数除法陷阱**：`index/(count-1)` 若都是 int 会被截断成 0/1，必须 `(float)index/(count-1)`。

### 3. 业界常见方案
- 扇形/锥形散射是射击游戏标配。两种风格：**确定性扇形**（均匀角度，可预测、可测）vs **随机散布**（Noita 风格的随机抖动，更"枪械手感"但不可测）。我们选确定性扇形——可单测、且把"是否散开"交给玩家（加"散射"修正法术才散）。

### 4. 最终用到的技术
- `SpellAiming.SpreadOffsetDegrees(index, count, spreadDegrees)` 纯静态函数：`count<=1 || spread<=0 → 0`，否则 `Mathf.Lerp(-spread/2, +spread/2, index/(count-1))`。
- `SpellCaster` 里：`yaw = SpreadOffsetDegrees(i, count, cmd.SpreadDegrees)`，`dir = Quaternion.AngleAxis(yaw, Vector3.up) * baseDir`（绕世界 Y 偏航）。

### 5. 架构设计与拆解（新增脚本）
- 新增 `Assets/_Project/Scripts/Skills/SpellAiming.cs`（纯函数 → 放 Skills、可单测）。
- 新增测试 `Assets/_Project/Tests/Skills/SpellAimingTests.cs`（4 个：单发→0、零散射→0、三发/30°→-15/0/+15、两发/20°→-10/+10）。

**为什么把这点几何抽成纯函数？** 因为方向数学是"能脱离引擎验证"的逻辑——抽出去就能单测，`SpellCaster` 只管把角度套到 `Quaternion` 上。再次体现"可测逻辑 / 引擎副作用"的切分原则。

### 6. 踩坑与 Debug
- **整数除法**：最隐蔽的坑。`(float)index / (count - 1)` 的强转不能漏，否则三发扇形会算成"全在一边"。测试 `Three_Spread30_FansEvenly` 直接锁死 -15/0/+15。
- **spread=0 的退化**：函数返回 0（全发同向重叠）——这是正确数据行为：想散开就加"散射"修正。`Multicast_BudgetUnfilled`/散射相关用例覆盖。

### 7. 面试 Q&A
- **Q：散射为什么单独抽一个纯函数，而不是写在 SpellCaster 里？**
  A：它是纯数学、无副作用——抽出去能在 EditMode 精确断言（-15/0/+15 这种），回归时一眼看出有没有算错。混在 MonoBehaviour 里就只能靠肉眼 Play 看，回归脆弱。
- **Q：确定性扇形 vs 随机散布怎么选？**
  A：作品集要"可解释、可测"，选确定性；而且把"散不散"交给玩家的修正法术，更贴"组合涌现"的主题。随机散布更适合追求枪感的成品，可作为另一种"散射"法术后续加。

## 七、接入法师：左键运行法杖（WizardController + 施法状态）

### 1. 需求拆解
把法师左键从"硬编码火球 / 长按陨石"改成"运行当前法杖"。要保留：按下瞬间锁存准心方向（不漂移）、快速连点不丢、施法动画。

### 2. 基础知识与原理
- **手写状态机（FSM）**：法师的攻击是一个状态（纯 C# 对象，非 MonoBehaviour），由 `PlayerStateMachine` 驱动 `Enter/Update/Exit`。
- **动画时机驱动**：施法动画播到某个归一化时间点（`ArrowSpawnTime`）才"释放"——读 `Animator.GetCurrentAnimatorStateInfo(0).normalizedTime`，单点用 bool 去重。
- **输入缓冲（buffer）+ 冷却（cooldown）**：按下入队，冷却中不丢、冷却一过补发；按下那一刻锁存准心（消除"出手时相机/身体已转动导致方向漂移"）。

### 3. 业界常见方案
- "意图与执行解耦"：输入只产生"意图"（入队），状态机在合适时机消费——动作游戏常见，避免输入丢帧/误触。
- 攻击动画用 Animation Event 或 normalizedTime 触发命中/发射——本项目统一用 normalizedTime（无平行的 Animation Event 机制）。

### 4. 最终用到的技术
- `WizardController.UpdateAttackInput()`（基类每帧钩子，始终运行）：上升沿 `ResolveAimTargetPoint` 锁存 `ClickAimPoint` + `_castQueued=true` + 入队计时；过期作废。
- `TryStartAttack()` / `TryStartAirAttack()`：`_castQueued && AttackCooldownCounter<=0` → `FireQueuedCast()` → 切到施法状态。
- `PlayerWizardAttackState`：播施法动画（来自 `ComboDefinition` 段 0）→ `normalizedTime` 越过 `ArrowSpawnTime` → `ReleaseCast()` → `_wizard.SpellCaster.CastWand(spawnPos, ClickAimPoint, team, attackerId, CharacterController)`。

### 5. 架构设计与拆解（改动脚本）
- 全文件改写 `WizardController.cs`：
  - **移除**：`_fireballPrefab`/`_projectileSpeed` 字段及 getter（投射物改由法杖提供）、长按→陨石路由、`_attackHeldTime`/`_attackPressActive`（不再需要 tap/hold 判定）。
  - **新增**：`_spellCaster`（`GetComponent` 缓存）+ `SpellCaster` getter；`_castQueued`/`FireQueuedCast`。
  - **保留**：按下锁存准心 `ClickAimPoint` + 入队补发（之前修过的"第二发偏左/误触陨石"那套）。
- 全文件改写 `PlayerWizardAttackState.cs`：`SpawnFireball`（硬编码火球）→ `ReleaseCast`（调 `SpellCaster.CastWand`）；`ComboDefinition` 段 0 仍作"施法动画 + 释放时机"的驱动。

**陨石暂休眠**：`WizardController` 里陨石相关字段/状态/hash **保留但不再路由**（避免改预制体序列化引用），`PlayerWizardHeavyState` 文件保留不再进入。陨石将在后续阶段作为"地面目标型法术"重做。这是经你确认的"完全取代"决策。

### 6. 踩坑与 Debug
- **删序列化字段的连锁**：移除 `_fireballPrefab`/`_projectileSpeed` 后，唯一的旧消费者是 `PlayerWizardAttackState.SpawnFireball`，已被改写——所以安全。复审专门确认了"没有任何残留引用指向已删成员"。
- **陨石休眠的编译安全**：`PlayerWizardHeavyState` 仍引用 `MeteorData/MeteorPrefab/...` 等——这些成员必须**全部保留**，否则休眠的重击态编译不过。复审逐个核对了保留清单。
- **`.meta` 文件的坑（贯穿全程）**：用廉价模型批量转写代码时，它几次想手搓 `.cs.meta`（被规则禁止——Unity 才该生成带正确 GUID 的 meta）。控制器每次都拦下/删除并修正提交。手搓的 2 行 meta 没有合法 GUID，会让 Unity 重新导入时 GUID 不一致、可能断引用。
- **第二发偏左的历史 bug（相关背景）**：本系统之所以坚持"按下瞬间锁存 `ClickAimPoint`"，是因为更早火球版本曾在"松手/出手那一刻"才读相机，导致挥杖期间转动轨道相机时第二发火球飞偏。锁存在按下帧解决了它。（实际后来发现偏左还叠加了"火球碰撞体过大互撞"，已一并修复。）

### 7. 面试 Q&A
- **Q：把法师攻击从写死火球换成"运行法杖"，对原有状态机改动大吗？**
  A：不大。状态机骨架（按下→入队→冷却→进施法态→动画时机→释放）完全复用，只把"释放点做什么"从"生成 1 个写死火球"换成"调 SpellCaster 运行法杖"。`ComboDefinition` 从"投射物数据源"退化成"施法动画/节奏驱动"。这种"换内核不换骨架"的增量改造，正是前期把接缝（释放点）留干净的回报。
- **Q：为什么按下那一刻就锁存瞄准方向？**
  A：施法有前摇（动画播到 ArrowSpawnTime 才发射）。若发射那一刻才读相机，玩家在前摇里转动轨道相机就会让法术飞向"转后"的方向，手感错乱。锁存"按下瞬间"的准心，"所见即所得"。
- **Q：陨石为什么不直接删掉，留着不是死代码吗？**
  A：删序列化字段会丢失预制体上的引用、还牵连 Animator 连线，风险高；而陨石很快会作为"地面目标法术"回归。所以选择"休眠"（保留成员、不再路由）这种低风险中间态，并在计划/账本里明确标注是有意为之，避免审查误判。

## 八、贯穿全局的三件事（架构 / 隔离 / 可测）

1. **程序集分层与单向依赖**
   ```
   Game.Core   (EventBus/GameLog，地基)
   └ Game.Combat (伤害/投射物/状态效果)
       ├ Game.Skills    ← 法术系统：纯求值(CastEvaluator) + 数据(SpellDefinition/WandLoadout) + 纯几何(SpellAiming)
       └ Game.Character ← 角色：SpellCaster(桥) + WizardController + 施法状态；本期新增对 Skills 的引用
   ```
   求值/数据/几何在 `Game.Skills`（纯、可测、不碰 Unity 实例化）；"落地实例化 + 接角色"在 `Game.Character`。依赖只向下/向纯逻辑，不反向。

2. **可测试性是设计目标，不是附属品**
   能脱离引擎验证的逻辑（求值、修正数学、散射几何）全做成纯函数/纯类，22 个 EditMode 测试护航；靠引擎的副作用（Instantiate、动画、输入）才放 MonoBehaviour，由开发者 Play 验证。复审甚至专门补了"端到端验证 BakeEmit 把速度/散射/平铺加伤写进产出"的测试，堵住"state 层证过、output 层没证"的盲区。

3. **scope 控制 = 明确写下"不做什么"**
   砍掉材料/流体模拟、回绕、洗牌、追踪/弹跳/穿透、法力资源池、触发递归——全部写进 spec 的"不在本阶段范围"。把触发的**函数签名**先留位（`incomingMods`）、把陨石**休眠**、把法力**接口就位但传大值**——这些都是"为将来留好接缝、但现在不实现"的克制。

## 九、面试者视角：完整开发思路与流程讲解（口述段落）

各位面试官好，下面我完整讲一下这套"法杖编程"系统是怎么从零设计、分阶段落地的。

首先是**定基调**。需求是做一套类似 Noita 的法术系统——玩家把法术像积木一样排进法杖，按一次开火，系统从左到右执行，组合出各种效果。我做的第一件事不是写代码，而是先纠正一个最容易犯的认知错误：很多人会把"组合"理解成"查表"，以为 A+B 触发隐藏技能 C，每个组合手写一条规则。这条路 N 个法术要维护 N² 张表，必然崩。我把它定义成**解释器**：法杖是程序、法术是指令、开火是让一个求值器从左到右求值。这样我只需要定义少数几类指令的求值规则，所有花样都是求值过程自然涌现的，加一个法术就是加一条数据、零新规则。这个定性决定了后面所有架构。

然后是**找出系统的"脊椎"并一次立对**。我把玩家能看到的七八种法术分类，收敛成求值器只认的三种语义——产出、修正、多重，后期再加触发。我意识到"施法块 + 抽取预算"是这套求值器的脊椎，所以哪怕前期不放多重法术，我也让求值器从第一天就带预算概念；同样，我把求值函数的签名一次定全——包括当时还不实现的法力参数和给触发递归留的"起始修正状态"参数——这样后续任务只往里填实现，不改接口、不牵连调用点。

接着是**一个关键的设计纠偏**。spec 初稿里有个隐藏矛盾：一边要求求值器是纯逻辑、可单测，一边又说它去 emit 真实投射物。可 Instantiate 是 Unity 调用，一旦进求值器就不纯、不可测、也没法递归。我把它一刀切开：**求值器只产出 EmitCommand 纯数据，真正的实例化交给一个叫 SpellCaster 的桥接组件**。这一刀让"纯、可测、可递归、能在引擎里造东西"四个诉求同时成立——这也是整套设计我最满意的决定。它和我们项目里 DamagePipeline 那种"纯函数算伤害、HealthComponent 负责扣血"是同一种品味。

落地上我**分了两大阶段、用测试驱动**。阶段 A 是纯内核：数据词汇、求值循环、多重、法力，全程不碰 Unity，写了十几个 EditMode 测试逐条锁死语义——修正只影响其后、多重内修正一次成本作用于全部、法力不足从哪发开始 fizzle、零费法术的边界等等。阶段 B 才接运行时：把 EmitCommand 实例化成真实投射物，复用项目已有的 ProjectileBase 火球和命中结算，不另造轮子；把多发散射抽成一个纯函数单独测；最后把法师左键从写死火球改成运行法杖——这里我特意复用了原有状态机骨架，只把"释放点做什么"换掉，并保留了"按下瞬间锁存准心"的手感处理（因为施法有前摇，发射时才读相机会让玩家在前摇里转镜头导致法术飞偏）。

**工程上的几个取舍**也值得讲。整套按程序集严格分层：求值、数据、几何放在纯粹的 Game.Skills，落地和接角色放在 Game.Character，依赖只向纯逻辑那边走。我把"能脱引擎验证的逻辑"全做成纯函数当作设计目标而非附属品，所以才有 22 个测试护航;甚至在最终整支复审时,补了一个"端到端验证修正确实被写进产出"的测试,堵住了"在状态层证过、但产出层没证"的盲区。对暂时不做的东西我也很克制：材料模拟、回绕、洗牌、触发递归、法力资源池全部明确写进"不做"清单，但都留好了接缝——触发留了函数参数、法力留了接口、陨石做成可唤醒的休眠态。

**踩过的坑**主要有：求值器纯不纯的那个矛盾（已述）；多重在短序列上若回绕会指数膨胀投射物，我用"单遍不回绕"换确定性；法力不足要跳出的是外层循环不是 switch；散射的整数除法必须强转 float 否则扇形会算歪；跨程序集调 ProjectileBase.Init 时逐个核对了参数顺序；以及用廉价模型批量生成代码时它几次想手搓 .meta 文件，我都拦下让 Unity 自己生成带合法 GUID 的 meta。

最后说**这套系统的价值**：它对玩家是"可编程的法术"，对我是一次"把看似复杂的涌现系统，收敛成一个干净、可测的解释器内核"的练习。它最能体现的能力不是"做了很多技能"，而是抽象、分层、可测试性和 scope 控制——加一个全新机制（比如多重）只改求值循环里一个 case，加一个新法术只是新建一个数据资产，这种可扩展性就是整套设计的回报。后续我会沿这个骨架做触发递归（命中点再跑一段子法术，是唯一引入递归的机制，配性能护栏）、把陨石重做成地面目标法术、补上法力资源系统，以及那套拖拽编程法杖的 UI。

谢谢，以上就是这套法术编程系统的完整开发思路。
