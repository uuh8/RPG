# 触发类火球术设计解析：从命中效果倒推实现

> 创建时间：2026-07-02  
> 适用读者：战斗系统设计新手  
> 对应功能：法术编程系统 Phase D：触发类火球术  
> 视觉约定：Icon 左上角带“小太阳标志”的火球术表示“触发类火球术”；没有小太阳的是普通火球术。

## 0. 先从最终效果说起

这次完成的功能可以用一句话概括：

玩家把“触发类火球术”放进法杖编程框后，它本身会先被发射出去；当这颗触发火球命中敌人或环境时，它会在命中点继续施放它后面的法术。

例如编程框里是：

```text
[触发火球] [普通火球]
```

施法时并不是两颗火球一起从玩家手里飞出去，而是：

1. 先从玩家手里发射“触发火球”；
2. 触发火球命中目标或墙面；
3. 在命中点继续施放它后面的“普通火球”。

如果编程框是：

```text
[触发火球] [触发火球] [普通火球]
```

那么就会变成一条链：

1. 第一颗触发火球从玩家手里飞出；
2. 命中后，在命中点发射第二颗触发火球；
3. 第二颗触发火球命中后，在新的命中点发射普通火球。

这就是它和普通多重发射最大的区别：普通多重发射强调“现在从施法者位置一次性发出多个投射物”；触发类法术强调“命中以后，把后续法术带到命中点继续执行”。

从实现结果倒推，这个复杂功能可以拆成六个小功能：

1. 数据层：让普通火球和触发火球成为同一种发射法术的两个数据变体。
2. 求值层：当求值器读到触发火球时，捕获它后面的法术作为 payload。
3. 战斗层：投射物命中时发出一个“我命中了”的通用通知。
4. 角色施法层：SpellCaster 订阅这个通知，并在命中点递归施放 payload。
5. 安全收敛层：保证触发链不会无限递归。
6. UI/资产层：用小太阳图标让玩家区分触发火球和普通火球。

下面按这个开发顺序逐步拆解。

---

## 1. 小功能一：把“触发火球”建模成 Emit 法术的一个数据变体

### 1.1 基础知识和原理

在法术编程系统里，法术不是写死在某个 `MonoBehaviour` 里的，而是用 `ScriptableObject` 配置出来的。

`ScriptableObject` 的核心价值是：把“数据”和“行为解释器”分开。

普通火球和触发火球从玩家视角看很像：它们都有图标、伤害、速度、投射物 prefab、法力消耗。区别只在于：触发火球命中后会继续释放后续法术。

所以它不是一个全新的大类，而更像是“发射类法术 Emit 的一个附加属性”。

也就是说：

```text
普通火球 = Emit + IsTrigger=false
触发火球 = Emit + IsTrigger=true
```

这种设计保留了法术系统最重要的可扩展性：以后如果要做“触发箭矢”“触发冰球”“触发闪电球”，都不需要新增一套代码，只要在对应的 Emit 法术资产上勾选触发标记即可。

### 1.2 常见实现方案

游戏开发里实现这种“命中后继续施法”的效果，常见有几种方案。

第一种是为触发火球写一个专门脚本，比如 `TriggerFireball.cs`。它的优点是直观，缺点是扩展差。每多一种触发投射物，就要复制一份类似逻辑，最后会变成很多“长得差不多但细节不同”的脚本。

第二种是把触发做成一个新的 `SpellKind.Trigger`。这样求值器会把它当作和 `Emit / Modify / Multicast` 平级的新语义。优点是概念明确，缺点是会把法术语义越拆越碎。因为触发本质上不是“它自己能不能产生投射物”的问题，而是“这个投射物命中后要不要携带后续法术”的问题。

第三种是把触发做成 Emit 的一个标记，也就是本项目最终采用的方案。它把“发射什么投射物”和“是否携带后续法术”合在同一个 Emit 数据上。这样普通火球和触发火球可以共享投射物 prefab、伤害结算、飞行逻辑、UI显示逻辑，只在求值时多处理一个 `IsTrigger`。

### 1.3 最终用到的技术

本功能主要用到：

- `ScriptableObject`：用 `SpellDefinition` 表示一个法术资产。
- `CreateAssetMenu`：让法术定义能在 Unity Editor 里创建。
- `Sprite Icon`：UI 里显示法术图标。
- `GameObject ProjectilePrefab`：指定真正生成的投射物 prefab。
- `bool IsTrigger`：标记这个 Emit 是否是触发类投射物。

核心脚本是：

```text
Assets/_Project/Scripts/Skills/SpellDefinition.cs
```

关键字段：

```csharp
public SpellKind Kind = SpellKind.Emit;
public Sprite Icon;
public GameObject ProjectilePrefab;
public float BaseDamage = 10f;
public float BaseSpeed = 20f;
public DamageType DamageType = DamageType.Magical;
public bool IsTrigger = false;
```

这里最关键的是 `IsTrigger`。它只对 `Kind=Emit` 的法术有意义。

### 1.4 架构设计与拆解

`SpellDefinition` 位于 `Game.Skills` 模块。这个位置很重要，因为它是法术编程系统的数据层，不应该依赖角色控制器，也不应该依赖 UI 具体表现。

它和其他组件的关系是：

```text
SpellDefinition
  -> 被 SpellLibrary 收集，供 UI 调色板显示
  -> 被 WandLoadout 引用，表示当前法杖里的编程序列
  -> 被 CastEvaluator 读取，解释成 EmitCommand
  -> 被 SpellCaster 间接使用，最终生成 ProjectileBase
```

本小功能涉及的脚本：

| 脚本 | 所属模块 | 作用 |
|---|---|---|
| `SpellDefinition.cs` | `Game.Skills` | 增加 `IsTrigger`，让 Emit 法术能成为触发类法术 |
| `SpellLibrary.cs` | `Game.Skills` | 不需要理解触发逻辑，只负责把触发火球资产列入可用法术 |
| `SpellSlotView.cs` | `Game.UI` | 读取 `SpellDefinition.Icon` 显示图标 |

本设计的重点是：UI 不判断触发规则，Combat 不知道法术规则，角色控制器也不硬编码“触发火球”。所有语义都从 `SpellDefinition` 数据开始，由求值器解释。

### 1.5 踩坑与 Debug

第一个坑是：不要把触发火球写成完全独立的一套投射物逻辑。那样短期能跑，长期会导致普通火球、触发火球、触发冰球、触发箭矢各自维护一套命中和施法代码，很难扩展。

第二个坑是：Icon 和数据标记可能不同步。当前截图里的“小太阳”是视觉约定，但真正决定触发行为的是 `SpellDefinition.IsTrigger`。如果某个资产图标带了小太阳但 `IsTrigger=false`，玩家会以为它能触发，实际不会触发。反过来，如果 `IsTrigger=true` 但图标没带小太阳，功能会生效，但 UI 会误导玩家。

后续可以考虑在 UI 里根据 `IsTrigger` 动态叠一个角标，而不是完全依赖美术图标本身，这样能减少资产配置错误。

### 1.6 模拟大厂面试 Q&A

**Q：为什么不新增一个 `TriggerSpellDefinition` 子类？**

A：因为触发不是一种完全独立的法术类型，而是 Emit 法术的一个命中后行为。用子类会把数据结构拆得更复杂，UI、求值器、资产管理都要处理更多类型判断。当前用 `IsTrigger` 表达“发射类法术是否携带 payload”，更符合这个功能的语义。

**Q：为什么不用一个新的 `SpellKind.Trigger`？**

A：`SpellKind` 表示求值器的大语义，比如发射、修正、多重。触发火球仍然会发射一个投射物，所以它本质还是 Emit。把触发做成 Emit 的属性，可以复用发射法术的全部数据和流程，避免语义膨胀。

**Q：这个方案以后能不能扩展到触发冰球、触发箭矢？**

A：可以。只要它们是 `Kind=Emit`，有自己的 `ProjectilePrefab`，并勾选 `IsTrigger`，求值器和 SpellCaster 不需要知道具体是什么投射物。

## 2. 小功能二：求值器捕获 payload，并结束当前层施法

### 2.1 基础知识和原理

法术编程系统的核心是“解释器”。

玩家在法杖里摆放的是一串 `SpellDefinition`，但游戏运行时需要把这串数据解释成真正要生成的投射物命令，也就是 `EmitCommand`。

普通情况下：

```text
[火球] [火球]
```

求值器会从左到右读，读到一个 Emit 就产出一个 `EmitCommand`。

但触发火球不一样：

```text
[触发火球] [火球]
```

读到触发火球时，求值器要做三件事：

1. 产出触发火球自己的 `EmitCommand`；
2. 把它后面的 `[火球]` 保存成触发火球的 payload；
3. 停止当前层继续求值，避免 `[火球]` 也立刻从玩家手里发出去。

这里的“当前层”很关键。第一次施法是从玩家手里发出；命中后是从命中点再开一个新层。触发火球后面的法术不是消失了，而是被延迟到命中时再解释。

### 2.2 常见实现方案

第一种方案是只捕获下一个法术。比如 `[触发火球] [火球] [火球]` 只让第一颗火球成为 payload。这个方案简单，但不够像 Noita，也很难表达更复杂的链式组合。

第二种方案是捕获触发火球后面的整个后缀，也就是本项目采用的方案。这样玩家能写出 `[触发火球] [修正] [火球]` 或 `[触发火球] [触发火球] [火球]` 这样的组合，系统自然支持链式触发。

第三种方案是把法术序列编译成一棵图或 AST。大型法术系统可能会这样做，适合非常复杂的控制流、条件、分支和循环。但当前阶段还不需要那么重的结构，用线性序列加后缀 payload 已经足够表达核心玩法。

### 2.3 最终用到的技术

核心脚本：

```text
Assets/_Project/Scripts/Skills/CastEvaluator.cs
Assets/_Project/Scripts/Skills/EmitCommand.cs
```

`EmitCommand` 增加了两个和触发相关的数据：

```csharp
public readonly IReadOnlyList<SpellDefinition> Payload;
public readonly CastModifierState PayloadMods;
public bool HasPayload => Payload != null && Payload.Count > 0;
```

它们的含义是：

- `Payload`：命中后要继续施放的法术列表。
- `PayloadMods`：触发火球生成时，已经累积到它身上的修正快照。

为什么需要 `PayloadMods`？

因为法术序列里可能有修正法术：

```text
[伤害 x2] [触发火球] [火球]
```

玩家通常会期待：触发火球本身吃到 `伤害 x2`，它命中后放出来的 payload 也继承这个上下文。否则触发前的修正就会在命中后“断掉”，组合体验会变得很不一致。

### 2.4 架构设计与拆解

`CastEvaluator` 是纯逻辑层，不生成 GameObject，不读 Transform，不播放音效，不依赖场景。

它的职责只是：

```text
输入：SpellDefinition 序列 + 施放数 + 法力 + 起始修正
输出：EmitCommand 列表 + 施法摘要
```

当它读到 `spell.IsTrigger` 时，会：

```text
1. 把当前 Emit 烘焙成 EmitCommand
2. CaptureSuffix(spells, i + 1)，把后续法术保存为 payload
3. 把当前修正状态保存为 PayloadMods
4. ended = true，停止本层求值
```

本小功能涉及的脚本：

| 脚本 | 所属模块 | 作用 |
|---|---|---|
| `CastEvaluator.cs` | `Game.Skills` | 解释法术序列，遇到触发 Emit 时捕获后缀 payload 并结束本层 |
| `EmitCommand.cs` | `Game.Skills` | 承载最终发射命令，同时携带触发 payload |
| `CastEvaluatorTests.cs` | `Game.Skills.Tests` | 用 EditMode 测试验证触发 payload 的纯逻辑 |

测试覆盖了几个关键规则：

- 普通 Emit 没有 payload；
- 触发 Emit 会捕获它后面的后缀；
- payload 不包含触发火球自己；
- 触发火球在序列末尾时，payload 为空，命中后不会继续生成东西。

### 2.5 踩坑与 Debug

第一个坑：如果触发火球不结束当前层，后面的法术会立刻发射。

例如：

```text
[触发火球] [普通火球]
```

如果不 `break` 当前层，那么普通火球会同时从玩家手里发出去，同时又被 payload 保存，结果变成“立即发一次，命中再发一次”。这不是我们想要的触发语义。

第二个坑：payload 不能捕获整个法杖序列。

如果命中后重新执行整个 wand：

```text
[触发火球] [普通火球]
```

触发火球命中后又会重新生成同一个触发火球，然后命中后继续重新生成，形成无限递归。当前方案使用 `CaptureSuffix(spells, i + 1)`，只捕获触发火球后面的法术，因此触发链会越来越短，自然收敛。

第三个坑：修正状态必须快照。

如果 payload 只保存后缀法术，不保存 `PayloadMods`，那么命中后重新求值时就会丢掉触发前已经累积的伤害、速度、散射等修正。保存快照后，触发链的上下文就能延续。

### 2.6 模拟大厂面试 Q&A

**Q：为什么触发火球后面的法术不在当前层继续执行？**

A：触发语义的核心是“延迟执行”。后续法术应该被带到命中点执行，而不是从施法者位置立即执行。所以求值器在遇到触发 Emit 时会捕获后缀，然后结束当前层。

**Q：为什么 payload 捕获后缀而不是只捕获下一个？**

A：捕获后缀能支持更丰富的组合，例如触发后接修正、接多重、接另一个触发。只捕获下一个虽然简单，但会明显限制法术编程的表达力。

**Q：如何避免无限递归？**

A：payload 捕获的是触发火球后面的后缀，不包含触发火球自己。每次触发后，下一层可执行的序列都会变短，所以链条会自然结束。

## 3. 小功能三：ProjectileBase 提供命中通知，但不认识法术系统

### 3.1 基础知识和原理

触发火球要在“命中时”继续施放 payload，所以必须知道投射物什么时候命中、命中在哪里、命中方向是什么。

命中检测属于 Combat 层，因为 `ProjectileBase` 已经负责：

- Rigidbody 飞行；
- Collider 碰撞；
- 队伍过滤；
- 伤害结算；
- 命中后销毁；
- 火球爆炸、箭矢插入等子类扩展。

但架构上 Combat 不能依赖 Skills 或 Character。也就是说，`ProjectileBase` 不能直接调用 `SpellCaster`，否则 Combat 就反向依赖了上层模块。

解决办法是观察者模式：

```text
ProjectileBase 只发出“我命中了”的通用事件
SpellCaster 如果关心触发，就订阅这个事件
```

这样 Combat 只提供通用命中通知，不知道它会被用来触发法术、生成特效，还是做别的表现。

### 3.2 常见实现方案

第一种方案是让火球命中时直接调用 `SpellCaster.CastPayload()`。这个方案最直接，但会破坏模块边界。Combat 会知道 Character/Skills 的存在，后续很难维护。

第二种方案是使用 UnityEvent。它适合设计师在 Inspector 里连表现事件，但这个功能需要运行时动态绑定 payload，而且每颗投射物的 payload 都不同，UnityEvent 反而不如 C# 事件清晰。

第三种方案是 C# event，也就是本项目采用的方案。`ProjectileBase` 定义一个 `Impacted` 事件，命中时把 `hitPoint` 和 `hitDir` 发出去。上层谁需要谁订阅。

### 3.3 最终用到的技术

核心脚本：

```text
Assets/_Project/Scripts/Combat/Projectiles/ProjectileBase.cs
```

关键技术：

- `Rigidbody`：投射物物理飞行。
- `Collider`：触发 `OnCollisionEnter`。
- `Collision.GetContact(0).point`：获取命中点。
- `Rigidbody.linearVelocity`：获取命中瞬间速度方向。
- C# `event Action<Vector3, Vector3>`：把命中点和命中方向通知上层。

事件定义：

```csharp
public event System.Action<Vector3, Vector3> Impacted;
```

命中流程简化后是：

```text
OnCollisionEnter
  -> 同队目标：忽略碰撞，直接 return，不触发
  -> 计算 hitPoint / hitDir
  -> 如果是敌人：ReceiveHit
  -> OnImpact：火球爆炸、箭矢插入等子类表现
  -> Impacted?.Invoke(hitPoint, hitDir)
  -> 标记 consumed 并销毁
```

### 3.4 架构设计与拆解

`ProjectileBase` 处在 `Game.Combat` 模块，是所有投射物的基类。

它和触发系统的关系是：

```text
ProjectileBase
  -> 不认识 SpellCaster
  -> 不认识 CastEvaluator
  -> 不认识 SpellDefinition
  -> 只在命中时发出 Impacted(hitPoint, hitDir)

SpellCaster
  -> 如果 EmitCommand.HasPayload
  -> 给本次实例化出来的 ProjectileBase 订阅 Impacted
```

本小功能涉及的脚本：

| 脚本 | 所属模块 | 作用 |
|---|---|---|
| `ProjectileBase.cs` | `Game.Combat` | 增加通用命中事件 `Impacted`，向上层暴露命中点和方向 |
| `Fireball.cs` | `Game.Combat` | 仍然只负责火球自己的命中特效、爆炸、燃烧等行为 |
| `Arrow.cs` / `NovaFireball.cs` | `Game.Combat` | 继承投射物基类，不需要理解触发系统 |

这个设计的重点是“打开一个 seam”，也就是在低层模块留下一个干净的扩展接口。Combat 只描述“发生了命中”，具体命中后要做什么由上层决定。

### 3.5 踩坑与 Debug

第一个坑：同队穿过不能算触发。

项目里同队投射物和队友会被过滤。如果同队碰撞也触发 payload，那么玩家快速连发时火球之间互相擦碰可能就会导致错误触发。因此当前逻辑是：同队目标只忽略碰撞并返回，不调用 `Impacted`。

第二个坑：超时自毁不能算触发。

投射物飞太久会被 `_maxLifetime` 销毁，这是防止场景里残留垃圾对象。超时不代表命中，所以不能触发 payload。当前 `Impacted` 只在 `OnCollisionEnter` 的真实命中路径里调用。

第三个坑：事件触发顺序要清楚。

当前顺序是先结算伤害和 `OnImpact`，再发 `Impacted`。这样触发 payload 时，原投射物的命中逻辑已经完成，命中点也已经确定。对玩家来说，它更像“命中产生后续法术”，而不是“后续法术抢在命中前发生”。

### 3.6 模拟大厂面试 Q&A

**Q：为什么不用 EventBus 做这个命中通知？**

A：EventBus 适合跨模块广播，比如伤害事件、死亡事件。这里的命中通知是“某一颗具体投射物携带某一份具体 payload”，订阅关系是实例级的。如果用全局 EventBus，还要额外做 projectile id 和 payload 映射，复杂度更高。C# event 更适合这种一对一运行时绑定。

**Q：为什么 Combat 层不直接处理触发？**

A：Combat 层应该只负责通用战斗结算。触发法术属于法术编程语义，如果放进 Combat，会让 Combat 依赖 Skills/Character，破坏当前 asmdef 的单向依赖原则。

**Q：命中环境也触发吗？**

A：是的。当前 `OnCollisionEnter` 命中敌人或环境都会走 `OnImpact` 和 `Impacted`。同队目标除外，因为同队穿过不算真实命中。

## 4. 小功能四：SpellCaster 在命中点递归运行 payload

### 4.1 基础知识和原理

前面三步已经准备好了：

```text
SpellDefinition 知道某个 Emit 是触发类
CastEvaluator 能把后续法术保存成 payload
ProjectileBase 能在命中时告诉上层 hitPoint / hitDir
```

剩下的问题是：谁把 payload 真正变成下一颗投射物？

答案是 `SpellCaster`。

`SpellCaster` 是角色侧的“法术运行时桥梁”。它一边拿到 `WandLoadout` 和 `SpellDefinition` 数据，一边能访问角色位置、瞄准方向、队伍、施法者 collider，并最终 `Instantiate` 投射物。

所以它最适合负责：

1. 第一次施法：从玩家位置运行当前法杖；
2. 触发施法：从命中点运行 payload。

### 4.2 常见实现方案

第一种方案是给触发写一个单独方法，比如 `CastTriggerPayload()`。这很容易和普通施法逻辑重复：法力、修正、音效、投射物初始化都要写两遍。

第二种方案是把普通施法和触发施法统一成一个内部方法，本项目采用这种方式。外部的 `CastWand()` 只负责从法杖开始，内部的 `RunCast()` 负责“给我一串法术、一个起点、一个方向，我就求值并生成投射物”。

第三种方案是做一个施法任务队列或协程系统。大型项目里如果触发法术很多、带延迟、带异步加载，可能会这样做。但当前阶段触发发生在命中瞬间，递归调用更简单直接。

### 4.3 最终用到的技术

核心脚本：

```text
Assets/_Project/Scripts/Character/Spells/SpellCaster.cs
```

关键技术：

- `CastEvaluator.Evaluate(...)`：把法术序列解释成 `EmitCommand`。
- `Object.Instantiate(...)`：生成投射物 prefab。
- `ProjectileBase.Init(...)`：注入队伍、攻击者、伤害、速度、施法者 collider。
- `ProjectileBase.Impacted += ...`：订阅命中事件。
- 闭包捕获 payload 和 payload mods：保证每颗触发投射物携带自己的后续法术。
- `HashSet<AudioClip>`：一次施法中相同音效只播放一次，避免多重发射声音叠得太吵。

触发路径可以简化理解为：

```text
RunCast(...)
  -> CastEvaluator.Evaluate(...) 得到 emits
  -> foreach emit
       -> Instantiate projectile
       -> ProjectileBase.Init(...)
       -> 如果 emit.HasPayload
            -> 订阅 projectile.Impacted
               -> 命中时 RunCast(payload, 1, payloadMods, hitPoint, hitDir, ...)
```

这里 `baseDraws=1` 很关键。命中后执行 payload 时，不是重新拿法杖原本的施放数，而是从一个基础施放预算开始，再由 payload 里的 Multicast 自己增加预算。

### 4.4 架构设计与拆解

`SpellCaster` 位于 `Game.Character` 模块。

它处在整个功能的汇合点：

```text
Game.Skills
  CastEvaluator / EmitCommand / SpellDefinition
        ↓
Game.Character
  SpellCaster：把纯数据命令变成场景里的投射物
        ↓
Game.Combat
  ProjectileBase：飞行、碰撞、伤害、命中通知
```

注意这个依赖方向是允许的：`Game.Character` 可以依赖 `Game.Skills` 和 `Game.Combat`，因为它是上层玩法控制模块。但 `Game.Combat` 不能反过来依赖 `Game.Character`。

本小功能涉及的脚本：

| 脚本 | 所属模块 | 作用 |
|---|---|---|
| `SpellCaster.cs` | `Game.Character` | 把普通施法和触发施法统一到 `RunCast`，在命中点递归运行 payload |
| `CastEvaluator.cs` | `Game.Skills` | 每次 `RunCast` 都通过它重新解释当前层法术 |
| `ProjectileBase.cs` | `Game.Combat` | 提供命中事件，让 `SpellCaster` 能触发下一层 |

### 4.5 踩坑与 Debug

第一个坑：递归施法必须保留队伍和攻击者信息。

触发出来的火球仍然应该属于原施法者。如果丢了 `team`，可能会打到自己或队友；如果丢了 `attackerId`，伤害事件、击杀归属、后续统计都会不准确。

第二个坑：必须继续忽略施法者 collider。

触发 payload 从命中点生成，通常不会和施法者重叠，但保持同一个 `casterCollider` 仍然是正确的防御式设计。它能避免某些近距离反弹、地形边缘或特殊法术把投射物生成回施法者附近时发生自撞。

第三个坑：闭包捕获要捕获局部变量。

如果在循环里直接使用可变的循环变量，多个投射物可能共享到错误的 payload。正确做法是在订阅前把 `cmd.Payload` 和 `cmd.PayloadMods` 存成局部变量，再让回调闭包捕获这些局部变量。

第四个坑：音效去重的作用范围要明确。

当前 `_playedSfx` 是一次 `RunCast` 内去重。这样多重发射不会把同一个音效叠成很响；触发 payload 发生在命中后，会进入新的 `RunCast`，可以重新播放属于那一层的音效。

### 4.6 模拟大厂面试 Q&A

**Q：为什么递归施法发生在 SpellCaster，而不是 CastEvaluator？**

A：CastEvaluator 是纯逻辑解释器，它不应该知道场景、Transform、Prefab 和物理。递归本质上是“把 payload 在另一个空间位置再运行一次”，这需要命中点、方向和投射物实例化，因此应该放在 SpellCaster 这种运行时桥接层。

**Q：命中后方向为什么用 hitDir？**

A：hitDir 来自投射物命中瞬间的速度方向。这样 payload 会沿着触发投射物原本飞行方向继续展开，玩家能直觉理解为“火球命中后继续往前释放”。如果以后想做反射、法线喷发、散射等效果，可以在这里扩展方向策略。

**Q：为什么 payload 的 baseDraws 是 1？**

A：触发语义表示“命中后开始执行它携带的后续序列”，默认应从一次基础施放开始。如果 payload 自己包含 Multicast，它仍然可以扩展发射数量。这样规则简单，也和当前 wand 的 base draw 语义分开。

## 5. 小功能五：让链式触发自然收敛

### 5.1 基础知识和原理

所有递归系统都要考虑一个问题：会不会停不下来？

触发火球天然容易产生递归，因为它命中后会继续施法，而继续施放出来的法术也可能是触发火球。

例如：

```text
[触发火球] [触发火球] [普通火球]
```

这个组合应该允许，因为它很有趣。但它必须最终停止。

本项目的停止规则来自一个简单不变量：

每次触发只捕获“自己后面的后缀”，所以越往后执行，剩余序列越短。

第一次：

```text
[触发火球A] [触发火球B] [普通火球]
```

A 的 payload 是：

```text
[触发火球B] [普通火球]
```

B 的 payload 是：

```text
[普通火球]
```

普通火球不是触发，所以链结束。

### 5.2 常见实现方案

第一种是加递归深度限制，比如最多触发 8 层。很多商业项目都会加这种硬保护，尤其是玩家能创造循环时。

第二种是加全局投射物数量上限，比如最多同时存在 100 个投射物。这个方案更偏性能保护，不直接表达语义。

第三种是设计语义本身保证收敛，本项目当前采用这种方案。只要 payload 一定是严格更短的后缀，就不会无限递归。

未来如果加入“循环”“复制整根法杖”“回到开头”这类高级法术，就必须重新加入深度上限、输出上限或法力约束，否则玩家很容易创造无限链。

### 5.3 最终用到的技术

关键点仍然在 `CastEvaluator.CaptureSuffix(spells, i + 1)`。

这个 `i + 1` 是安全性的核心。它表示 payload 从触发火球之后开始，不包含触发火球自己，也不回到序列开头。

此外，`EmitCommand.HasPayload` 只有在 payload 非空时才为 true：

```csharp
public bool HasPayload => Payload != null && Payload.Count > 0;
```

所以如果触发火球放在最后：

```text
[触发火球]
```

它仍然会像一颗普通投射物一样飞出，但命中后没有后续法术可执行。

### 5.4 架构设计与拆解

这个小功能不是一个独立脚本，而是由两层共同保证：

```text
CastEvaluator
  -> 只捕获后缀，保证 payload 越来越短

SpellCaster
  -> 只有 HasPayload 的投射物才订阅 Impacted
```

涉及脚本：

| 脚本 | 作用 |
|---|---|
| `CastEvaluator.cs` | 通过后缀捕获建立收敛规则 |
| `EmitCommand.cs` | 用 `HasPayload` 表示是否需要绑定触发回调 |
| `SpellCaster.cs` | 只对有 payload 的投射物订阅命中回调 |

### 5.5 踩坑与 Debug

最危险的坑是“为了方便”把 payload 改成整个 wand。

这样做表面上可以让触发火球命中后重新施放整根法杖，但它会立刻破坏收敛规则。只要序列里有一个触发火球，就可能无限触发。

另一个坑是未来加入 Shuffle 或随机抽牌时，后缀长度可能不再是简单线性减少。到那个阶段就不能只靠 `CaptureSuffix` 保证安全，需要引入更明确的触发预算或递归深度限制。

### 5.6 模拟大厂面试 Q&A

**Q：当前没有递归深度上限，安全吗？**

A：在当前语义下是可控的，因为 payload 是严格后缀，每次触发都会减少可执行序列长度。不过这只是当前功能阶段的安全性。如果未来加入循环、回卷、复制整根法杖等机制，就需要额外加入深度上限和投射物数量上限。

**Q：为什么不现在就加全局上限？**

A：可以加，但当前阶段的主要目标是验证触发语义。过早加入全局上限可能会掩盖求值规则本身的问题。后续进入资源、冷却、性能治理阶段时，再统一做 projectile cap 和 cast budget 会更自然。

**Q：如果玩家连续放很多触发火球，会不会性能爆炸？**

A：当前每条链最多受法杖槽位长度限制，且同队投射物碰撞已经有过滤。真正的性能风险会来自未来更复杂的复制、多重和循环组合，因此后续需要在系统层加入施法预算和全局上限。

## 6. 小功能六：用小太阳图标建立玩家可读性

### 6.1 基础知识和原理

战斗系统不只是“代码能跑”，还要让玩家看得懂。

触发火球和普通火球在视觉上都像火球。如果 UI 没有区分，玩家会很难理解为什么一个火球命中后会继续施法，另一个不会。

所以当前采用了一个简单直观的视觉约定：

```text
左上角带小太阳 = 触发类火球术
没有小太阳 = 普通火球术
```

从截图可以看到，调色板和编程框里都能看到这个标记。这样玩家在拖拽时就能识别这个法术的特殊语义。

### 6.2 常见实现方案

第一种方案是为触发法术单独制作带角标的 Icon，也就是当前方案。优点是简单、稳定，UI 代码不需要改。缺点是资产配置要小心，Icon 和 `IsTrigger` 可能出现不同步。

第二种方案是 UI 根据 `IsTrigger` 动态叠加一个角标。优点是数据和视觉更不容易错；缺点是 UI prefab 要多一个 badge 层，代码也要多一点绑定逻辑。

第三种方案是使用 tooltip、边框颜色、标签文字等多重提示。适合后续法术种类变多以后补充，但当前阶段优先保证图标上能快速识别。

### 6.3 最终用到的技术

当前 UI 链路主要是：

```text
SpellLibrary.Available
  -> SpellPaletteView.Rebuild
  -> SpellSlotView.Bind
  -> SpellSlotView.Refresh
  -> Image.sprite = SpellDefinition.Icon
```

涉及 Unity 技术：

- UGUI `Image`：显示法术图标。
- UGUI `Text`：无图标时显示名字兜底。
- `EventSystem` 拖拽接口：`IBeginDragHandler / IDragHandler / IEndDragHandler / IDropHandler`。
- `HorizontalLayoutGroup` 或类似布局组件：排列调色板和编程框格子。

核心脚本：

```text
Assets/_Project/Scripts/UI/SpellSlotView.cs
Assets/_Project/Scripts/UI/SpellPaletteView.cs
Assets/_Project/Scripts/UI/WandEditorController.cs
```

其中 `SpellSlotView` 只读 `spell.Icon`，不判断 `IsTrigger`。这意味着当前小太阳是资产层的视觉约定，而不是 UI 运行时自动生成的角标。

### 6.4 架构设计与拆解

UI 层在架构中的职责是“展示和编辑法术序列”，不是“解释法术行为”。

它和触发功能的关系是：

```text
Game.UI
  -> 读取 SpellDefinition.Icon，展示小太阳图标
  -> 通过拖拽把 SpellDefinition 引用插入 WandLoadout
  -> 不判断触发逻辑，不生成投射物

Game.Skills
  -> 根据 IsTrigger 决定触发语义
```

涉及脚本：

| 脚本 | 作用 |
|---|---|
| `SpellSlotView.cs` | 显示单个法术格，读取 `SpellDefinition.Icon` |
| `SpellPaletteView.cs` | 从 `SpellLibrary` 生成调色板 |
| `WandEditorController.cs` | 控制打开面板、拖拽、插入、移动、移除 |
| `WandFrameView.cs` | 展示当前法杖编程框并写回 `WandLoadout` |

这部分最重要的设计点是：UI 只负责让玩家正确编辑数据。真正的触发语义仍然由 `CastEvaluator` 和 `SpellCaster` 执行。

### 6.5 踩坑与 Debug

第一个坑：图标 Image 的 alpha 或 GameObject active 状态可能导致图标看不见。

`SpellSlotView.Refresh()` 里专门处理了：

```text
_icon.sprite = _spell.Icon
_icon.color.a = 1
_icon.enabled = true
_icon.gameObject.SetActive(true)
```

这是为了避免 prefab 里图标默认透明或子物体 inactive，导致绑定了 Icon 但 UI 仍然不显示。

第二个坑：拖拽影子也要用同一个 Icon。

`WandEditorController.BeginSpellDrag()` 会把 `_dragGhost.sprite` 设置成当前法术的 Icon。这样拖动触发火球时，小太阳标记也会跟着鼠标走，玩家不会在拖拽过程中丢失识别。

第三个坑：当前小太阳不是自动 badge。

这不是 bug，但需要记住。现在如果美术资源配错，系统不会自动报错。后续可以补一个 Editor 校验：如果 `IsTrigger=true` 但 Icon 命名或角标缺失，就给出警告。

### 6.6 模拟大厂面试 Q&A

**Q：为什么 UI 不根据 `IsTrigger` 自动显示小太阳？**

A：当前阶段为了快速验证玩法，使用带角标的图标资产最简单，不需要改 UI prefab 结构。它的代价是资产配置需要人工保证一致。后续如果法术数量增加，我会把小太阳做成动态 badge，直接由 `IsTrigger` 驱动，减少配置错误。

**Q：UI 层为什么不直接判断触发法术？**

A：UI 层可以展示触发标记，但不应该解释触发行为。触发的运行语义属于 Skills 求值器和 Character 施法器。这样 UI 改版不会影响战斗规则，战斗规则扩展也不会强迫 UI 写特殊逻辑。

**Q：玩家如何理解这个法术？**

A：通过图标角标建立第一层识别，通过实际效果建立第二层反馈：带小太阳的火球命中后会继续施放后续法术，普通火球只完成自己的命中效果。

## 8. 从大厂面试角度完整讲解开发思路

如果我在面试里介绍这个功能，我会这样讲：

这次我做的是法术编程系统里的“触发类火球术”。它的设计目标是让玩家能把一个火球作为后续法术的载体：火球先从玩家手里发射，命中敌人或环境后，再在命中点继续释放它后面的法术。这个玩法的重点不是简单多发一颗火球，而是把“法术执行位置”从施法者手上转移到命中点，从而形成链式组合。

我首先没有把触发火球做成一个新的硬编码技能，而是把它建模成 Emit 法术的一个数据变体。普通火球和触发火球共享 `SpellDefinition` 的基础字段，包括 prefab、伤害、速度、音效和 Icon，区别只是在 Emit 上增加了 `IsTrigger` 标记。这样以后如果要做触发冰球、触发箭矢，也不需要复制新逻辑，只要配置对应的发射法术资产。

然后我在纯逻辑层的 `CastEvaluator` 里扩展求值规则。普通 Emit 会直接产出一个 `EmitCommand`，但当求值器读到 `IsTrigger=true` 的 Emit 时，会把它自己产出为当前层的投射命令，同时把它后面的法术后缀保存成 payload，并结束当前层求值。这样 `[触发火球] [普通火球]` 不会让普通火球立即从玩家手里飞出去，而是会把普通火球延迟到触发火球命中后执行。这里 payload 捕获的是严格后缀，不包含触发火球自己，所以链式触发会自然收敛，不会在当前规则下无限递归。

接着我在 Combat 层打开了一个干净的命中通知 seam。`ProjectileBase` 仍然只负责投射物飞行、碰撞、同队过滤、伤害结算和销毁，但它在真实命中敌人或环境时发出 `Impacted(hitPoint, hitDir)` 事件。Combat 不知道法术系统，也不直接调用 `SpellCaster`，这样保持了模块依赖方向。`SpellCaster` 在生成一颗带 payload 的投射物时，才给这颗具体投射物订阅 `Impacted`。

最后，`SpellCaster` 把普通施法和触发施法统一到内部的 `RunCast` 方法里。第一次施法时，它从当前法杖运行整段序列；触发火球命中时，它从命中点、沿命中方向，再运行 payload 序列。这个过程会保留原施法者的队伍、攻击者 id、caster collider 和修正快照，所以伤害归属、友军过滤和修正继承都是一致的。

在 UI 上，我没有让 UI 参与法术规则判断，而是用资产图标建立玩家识别：左上角带小太阳的是触发火球，没有小太阳的是普通火球。UI 仍然只是读取 `SpellDefinition.Icon` 并把法术引用拖进 `WandLoadout`，真正的触发语义仍由 Skills 的求值器和 Character 的施法器负责。这保证了表现层和玩法规则的边界清晰。

整个功能的开发顺序是：先定义数据语义，再扩展纯求值器，再给投射物补通用命中事件，最后在运行时施法器里把 payload 递归执行出来。这样每一步都有明确边界，低层模块不反向依赖高层模块，触发逻辑也能通过纯单元测试先验证，再交给 Unity 场景里的投射物命中链路完成最终效果。
