## 1. 触发是什么？（用大白话）

普通投射物：**火球飞出去 → 命中 → 造成伤害 → 消失。** 一锤子买卖。

**触发**给投射物加一条"遗言"：

> **触发投射物：飞出去 → 命中 → 造成伤害 → 在命中那一点，再施放它后面跟着的法术 → 然后才消失。**

例子。法杖摆成：

```
[ 触发火球 | 魔法箭 ]
```

- 按开火 → 飞出一颗**火球**。
- 火球**命中** → 在命中点**再射出一支魔法箭**。

"触发火球后面跟的法术(魔法箭)"，不是开火时发射，而是**等火球命中、在命中点才发射**。这段跟着触发走的后续法术，叫**载荷(payload)**。

**一句话**：触发法术后面那个，就是"它命中时会发生的事"。

---

## 2. 为什么它叫"递归"？（命门）

我们整套系统的核心：**开火 = 一个"求值器"从左到右运行法杖。**

普通法术，求值器跑一遍就完。但**触发是唯一会让"运行求值器"再发生一次的机制**：

- 开火：运行求值器 → 产出触发火球（身上**带载荷**）。
- 火球命中：**再运行一次求值器**（这次跑载荷）→ 在命中点产出新投射物。

"运行求值器"在命中时又调用了它自己 = **递归**。还能链：载荷里若又是触发，新投射物命中时又运行一次……一层接一层（**这正是 Noita 那种连锁的来源**）。

```
开火 ─运行求值器─► 触发火球(载荷=魔法箭) ─命中─► 运行求值器(魔法箭) ─► 魔法箭
        （若魔法箭也是触发，则它命中时继续往后链……）
```

---

## 3. 它怎么复用我们已做好的东西（纯增量，不重构内核）

阶段 A 就为它留了位子：

1. **求值器的"可重入"接缝**：`CastEvaluator.Evaluate(..., incomingMods, ...)` 第一天就带了 `incomingMods`（"本次运行的起始修正状态"）。普通开火传"默认"；**触发命中时传火球当初捕获的修正快照**，载荷就能继承当时的增伤/加速。这个参数就是为递归留的。
2. **投射物命中钩子**：`ProjectileBase.OnImpact`（命中后、销毁前的扩展点）已存在——触发就在这一刻"再运行一次求值器、在命中点产出"。
3. **施放器**：阶段 B 的 `SpellCaster` 已能"把求值结果变真实投射物"，触发的"命中再施放"复用它，只是生成点改成命中点。

所以触发 = 给法术加一个"我是触发"的标记、给产出指令加一个"我带的载荷"、命中时回调一次施放。**内核循环不动。**

---

## 4. 一次触发，数据怎么流（落到具体）

以 `[触发火球, 魔法箭]` 为例：

```
按开火 → 求值器从左到右读：
   读到「触发火球」(emit 且 IsTrigger=true)：
     - 像普通火球一样产出一颗（带当前修正快照）
     - 因为它是触发：把它"之后的法术后缀"[魔法箭] 捕获为载荷，连同当前修正快照，挂到这颗火球的产出指令上
     - 触发会"吃掉"它后面的法术作为载荷 → 本层求值就此结束（后面的不在开火时单独发射）
   SpellCaster 生成火球，并把"载荷"存到火球身上
火球飞行 → 命中
   OnImpact：
     - 正常爆炸/结算伤害（不变）
     - 因为带载荷：在命中点再运行一次求值器
         Evaluate(载荷, baseDraws=1, incomingMods=火球的修正快照)   ← "预算 1" = 只放出下一个
       → 产出一支魔法箭 → 在命中点射出
       （若载荷的第一个又是触发，则它再捕获自己的后缀 → 链式继续）
火球销毁
```

三个要点：
- **载荷在"开火那一刻"捕获并定型**（含修正快照）——所见即所得，与火球"按下瞬间锁存准心"同哲学。
- **命中点 = 载荷的新生成点**。
- **以"预算 1"求值载荷** = "放出下一个法术"的语义；若下一个是触发就继续链（和 Noita 一样）。

## 5. 为什么本阶段"不需要"递归护栏（你提的关键点，成立）

直觉是"触发能套娃 → 会指数爆炸"，但**在我们的设定里不会**，两个原因：

1. **法杖长度有限**：编程框 `Capacity`（如 8）限死一根法杖最多几个法术。
2. **载荷是"严格更短的后缀"**：触发捕获的载荷 = 它**之后**的法术，比当前序列短；递归一层，待处理序列就更短一截。

两者合起来 → **递归深度 ≤ 法杖长度（有限），每层产出也被有限法杖限死 → 不可能无限/指数爆炸。** 所以**不设递归深度上限、不设单次产出上限**——有限法杖本身就是天然预算。

> ⚠️ 这个"安全"依赖一条**不变量**：载荷必须是"比当前序列更短的后缀"。只要将来别把载荷改成"整根法杖"或"会再生法术"，递归就永远收敛。实现时把这条写进注释，别哪天手贱破坏了。
>
> （唯一可能值得加的，是与递归无关的"场上活跃投射物全局上限"——那是防"疯狂连发 + 长寿命投射物"堆积，不是防递归。但有射速冷却 + 投射物自带寿命兜底，MVP 先不做，等 Profiler 真发现问题再加。）

## 6. 架构草图（这轮点到为止，细节留给实现计划）

> 触发 = 投射物的一个**变种(affix)**，不是新的第五类法术（玩家眼里是"带触发的火球"，求值器眼里仍是 `Emit`，只是多带一段载荷）。

预计改动点：

- **`SpellDefinition`**：给 Emit 类加 `bool IsTrigger`（这条投射物命中时运行其载荷）。
- **`EmitCommand`**：加"载荷"——捕获的载荷法术序列（后缀）+ 修正快照。普通产出这块为空。
- **`CastEvaluator`**：读到"触发 emit"时，把其后的后缀捕获为载荷塞进 `EmitCommand`，并**结束本层求值**（后缀不在本层单独产出）。
- **`SpellCaster`**：生成触发投射物时把载荷"挂"到投射物上；并提供"命中时按载荷在命中点以预算 1 再施放一次"的回调。
- **`ProjectileBase`（Game.Combat）**：需要一个最小的**命中通知接缝**，让上层（Game.Character 施放侧）在投射物命中时被叫到去跑载荷——同时保持模块隔离（Combat 不反依赖 Skills/Character）。这是本阶段唯一在 Combat 侧开的小口子，实现计划里设计得最小化。

---

## 7. 范围（MVP）

**做：**
- "命中触发"：触发投射物命中时，在命中点施放其载荷。
- 载荷 = 触发法术之后的法术后缀（开火时捕获 + 修正快照）；命中时以"预算 1"求值，即"放出下一个"；下一个若也是触发则继续链（**和 Noita 一样**）。
- 链式触发递归，由有限法杖天然收敛，**不加护栏**。
- 复用 `incomingMods` 接缝让载荷继承修正。

**不做（MVP 外）：**
- **定时触发**（飞行 X 秒后自动触发）——下一步再加，机制同源。
- 触发的特效打磨、UI 上对"触发法术"的特殊标识。
- 与递归无关的"全局活跃投射物上限"（除非 Profiler 发现需要）。

---

## 8. 决策（2026-06-29 已拍板）

1. **载荷 = 紧跟的下一个法术（和 Noita 一样的链式触发）**：实现为"捕获其后的后缀、命中时以预算 1 求值"——每次只放出下一个；若下一个又是触发则继续往后链。
2. **只做"命中触发"**；定时触发留后续。
3. **不设递归护栏**：有限法杖 `Capacity` + 载荷是严格更短的后缀 → 递归天然收敛（见第 5 节不变量）。
4. **触发 = emit 法术 + `IsTrigger` 标记**（如新建"触发火球"资产，预制体仍是火球，勾上 IsTrigger）。

> 一处需你点头确认的细微解释（决策 1 内）：你说"只捕获下一个"+"和 Noita 一样"。这两者在**深层链式**上略有张力——真正像 Noita 的链式触发（触发→触发→触发）需要"捕获后缀、每次只放一个、下一个是触发就继续"。所以我把"只捕获下一个"实现成上面这种**能链式**的版本（每命中只放出下一个，但链条能延续）。如果你其实想要"只放一个、不延续链"的更简版，告诉我即可。
