## 0. 怎么读这份文档

这份文档刻意写得**偏基础**，目标是让你（战斗设计新手）彻底看懂这套系统。读法：

1. 先读 **第 1~3 节**：用"法师现在怎么打火球"对比"新系统怎么打火球"，建立直觉。
2. 再读 **第 4~5 节**：理解"法杖 = 程序、开火 = 运行程序"这个核心模型 + 一次完整执行追踪。
3. **第 6 节**是给我（写代码的人）看的架构落地，但我也尽量讲清楚，你能看个大概。
4. 第 7 节往后是分期、决策、风险、名词表。

每一节凡是涉及"它和现在有什么不一样"，我都会用 **【现状】/【新方案】** 两栏对照。

## 1. 一句话：我们要做的是什么

> 不做"一堆技能按钮"，而是做一套 **像 Noita 那样：把若干"法术"按顺序摆进一个"法杖"，按一次开火键，系统从左到右"运行"这串法术，组合出花样百出的效果。**

打个最贴近的比方：

- **传统技能系统** = 你有 Q/E/R 三个按钮，每个按钮 = 一个写死的技能。想要新效果，就得程序员再写一个新技能。
- **法术编程系统** = 你有一排"积木"（火球、增伤、分裂、追踪…），自己把积木排成一行，按开火键就按这一行**从左到右执行**。换个排法 = 换个效果，**不需要程序员加新代码**。

这套系统真正的卖点（也是简历含金量所在）不是"我做了很多技能"，而是"我做了一个**能把简单零件组合出复杂效果的小型解释器**"——这和你项目里 `DamagePipeline`（纯函数、可单测）是同一种工程品味。

---

## 2. 先看清楚：法师现在是怎么打火球的【现状】

在动新系统之前，先把**现在**的火球流程摊开，后面所有对比都基于它。

【现状】按一次左键 → 打出一颗火球，整条链路是写死的：

```
你按左键
  → WizardController.UpdateAttackInput()   读到"按下"，锁存准心方向(ClickAimPoint)，把这一发"入队"
  → WizardController.TryStartAttack()       发现队列里有一发、冷却已过 → 进入攻击状态
  → PlayerWizardAttackState                 播放挥杖动画，到 0.4 时刻调 SpawnFireball()
  → SpawnFireball()                         Instantiate 一颗 Fireball 预制体
  → fireball.Init(队伍, 攻击者, 伤害, 类型, 速度, ...)   把伤害/速度等"快照"进去
  → 火球飞出去，命中敌人 → 走 IDamageable.ReceiveHit → 扣血
```

关键点：**"按一次 = 一颗火球"是被硬编码死的**。火球的伤害是固定的（来自 `AttackDefinition`），数量是固定的（1 颗），没有"组合"的余地。想做"一次打三颗"或"火球更强"，今天只能改代码。

这就是新系统要打破的地方。

## 3. 新系统怎么打火球（一步步加料）【新方案】

新方案下，火球**不再是"按键直接生成"**，而是"**运行一段法杖程序，而这段程序恰好包含一个火球法术**"。我们从最简单的情况开始，逐步加料，你会亲眼看到"组合涌现"。

### 3.1 最小情况：法杖里只有 `[火球]`

【现状】按键 → 写死生成 1 颗火球。
【新方案】按键 → 运行法杖 `[火球]` → 程序从左到右只读到一个"火球"指令 → 生成 1 颗火球。

**结果一模一样**。这一步的意义是：新系统在最简单的情况下，行为和现在完全相同——所以它不是推倒重来，而是"把写死的那一步，换成跑一个只有一条指令的程序"。

### 3.2 加一个"修正"：法杖 `[增伤, 火球]`

【新方案】按键 → 程序从左到右：
1. 读到"增伤" → 这不是生成东西，而是**修改接下来要生成的火球的属性**（比如伤害 +50%）。
2. 读到"火球" → 生成一颗火球，但它带上了"刚才那个 +50% 增伤"。

**结果：一颗更强的火球。** 注意"增伤"必须放在"火球"**前面**才有用——因为程序是从左往右读的，修正只影响它**后面**的法术。这条规则你要记住，它是整套系统直觉的来源。

### 3.3 加一个"多重"：法杖 `[三重施法, 火球]`

【新方案】按键 → 程序从左到右：
1. 读到"三重施法" → 它告诉系统："这一次开火，要多生成几发"。
2. 读到"火球" → 因为受三重影响，**一次生成 3 颗火球**（呈扇形散开）。

**结果：一次打 3 颗火球。**

### 3.4 组合起来：法杖 `[三重施法, 增伤, 火球]`

【新方案】按键 → 三重(要 3 发) + 增伤(后面的都 +50%) + 火球 → **3 颗都带增伤的火球**。

看到没？"3 颗" + "都增伤"这两个效果，是**程序自然跑出来的**，没有人专门写过"三重增伤火球"这个技能。这就是"涌现"——**你定义少量零件的规则，组合效果自动产生**。

> 这就是和传统技能系统最本质的差别：传统系统里"三连增伤火球"是一个需要程序员手写的技能；这里它只是 `[三重, 增伤, 火球]` 三块积木的摆放顺序。

## 4. 核心模型：法杖是"程序"，开火是"运行它"

把上面的直觉正式化成三个词：

| 术语 | 大白话 | 例子 |
|------|--------|------|
| **法术（Spell）** | 一条"指令" | 火球、增伤、三重施法 |
| **法杖（Wand / Loadout）** | 一串按顺序排好的指令 = 一段"程序" | `[三重, 增伤, 火球]` |
| **开火** | 让一个"解释器"从左到右**运行**这段程序 | 按一次左键 |

### 4.1 法术其实只有三种"语义"（这是关键的简化）

你在 Noita 里会看到很多分类（投射物、静态投射物、修正、多重、触发…），但**对"运行程序的解释器"来说，前期只需要认识 3 种**：

| 你/玩家看到的分类 | 解释器眼里的语义 | 它做什么 |
|---|---|---|
| 投射物（火球、魔法箭…）、静态投射物（黑洞、火环…） | **emit（产出）** | 生成一个飞行/存在物，带上"当前累积的修正" |
| 修正（增伤、加速、追踪…） | **modify（修改）** | 改"当前修正状态"，影响它**之后**产出的东西 |
| 多重（双重/三重施法） | **multicast（多抽）** | 让这一次开火**多产出几发** |
| （后期）触发/定时 | emit + 携带一段"子程序" | 唯一会引入"递归"的；前期不做 |

**重要洞察**：玩家看到 4~8 类，解释器前期只需处理 **3 种语义**。"火球"和"黑洞"在解释器眼里都是 `emit`——它俩的区别（一个飞、一个不飞）是各自**预制体**的事，跟"运行程序"的逻辑无关。把丰富的玩家分类收敛成内核的少数语义，正是这套设计干净的原因。

## 5. 一次完整的执行追踪（看懂这个，你就懂了 80%）

假设法杖是：

```
[ 三重施法 | 增伤 | 火球 | 魔法箭 | 魔法箭 ]
```

按一次开火，解释器（我们叫它 `CastEvaluator`）这样跑（右边一栏标注它和现状的差别）：

```
读到「三重施法」  → multicast：这次开火"要产出 3 发"           （现状没有这个概念）
读到「增伤」      → modify：当前修正.伤害 +50%（先记着，不产出）  （现状没有"累积修正"）
读到「火球」      → emit：产出 1 颗火球，带上"伤害+50%"的快照     （现状=SpawnFireball，但现在是写死的）
                    还差 2 发 → 继续往后读
读到「魔法箭」    → emit：产出 1 支魔法箭，带"伤害+50%"           还差 1 发
读到「魔法箭」    → emit：产出 1 支魔法箭，带"伤害+50%"           凑够 3 发 → 本次开火结束

最终：1 火球 + 2 魔法箭 同时射出，三者都带 +50% 增伤
```

几条**必须理解的规则**（它们都是这一个循环自然得出的，不是特例）：

- **修正只影响"它后面、且本次开火内"的产出**。所以增伤要放前面。
- **多重的修正只花一次、却作用于全部**——因为修正改的是"当前状态"，本次产出的每一发都读这个状态。
- **预算决定产出几发**：三重把"投射物预算"加到 3，之后每读到一个投射物就产出一发、预算 -1，预算耗尽就不再产出。**MVP 采用单遍读取、不回绕**：如果产出物不够填满预算（比如三重后面只有 1 个火球），剩余预算直接作废——想要 3 发就摆 3 个投射物（`[三重, 火球, 火球, 火球]`）。这条规则简单、确定、好测，也避免了"多重在短序列上回绕导致指数膨胀"的坑（回绕/牌组指针/充能这些 Noita 细节留到后续阶段再议）。

## 6. 它怎么长进你现有的代码（架构）【现状 vs 新方案】

这一节是落地核心。**最重要的一个设计决定**：

> **解释器（CastEvaluator）只"算出该产出哪些东西"，它本身不 Instantiate 任何 Unity 物体。真正的生成交给一个 MonoBehaviour。**

为什么这么切？因为你已经有一个完美的范本——`DamagePipeline`：

| | `DamagePipeline`（你已有的） | `CastEvaluator`（新的，同款思路） |
|---|---|---|
| 是什么 | 纯函数：输入伤害请求 → 输出伤害结果 | 纯函数：输入法杖序列 → 输出"该产出哪些东西" |
| 碰 Unity 吗 | 不碰，纯 C# | 不碰，纯 C# |
| 能单测吗 | 能（EditMode 测试） | 能（这正是它的价值） |
| 谁来"落地" | `HealthComponent` 拿结果去扣血 | `SpellCaster` 拿结果去 Instantiate 投射物 |

### 6.1 数据流（新方案）

```
你按开火
  → SpellCaster（MonoBehaviour，挂在法师身上）读取当前法杖 WandLoadout
  → 调 CastEvaluator.Evaluate(法杖序列, 施法器参数)   ← 纯函数，零 Unity
       返回一个列表：List<EmitCommand>
       （EmitCommand = "该产出什么" 的纯数据：哪个法术 + 修正快照[+后期:子程序]）
  → SpellCaster 遍历这个列表，对每条 EmitCommand：
       Instantiate 对应预制体（火球/魔法箭…）
       把修正快照应用上去（伤害×倍率、速度×倍率、扇形偏角…）
       调用现有的 ProjectileBase.Init(...) —— 完全复用你现有的投射物体系
  → 投射物飞出去命中 → 走现有 IDamageable.ReceiveHit / CombatDamage seam → 扣血
```

注意最后两步：**新系统不另造投射物轮子**，它产出的火球就是你现在的 `Fireball` 预制体，命中走的还是 `ReceiveHit` 那条老路。新系统只是替换了"决定产出什么"的那一层大脑。

### 6.2 复用对照表（让你看到它有多贴合现有代码）

| 需求 | 现有可复用的东西 | 怎么用 |
|---|---|---|
| 解释器内核放哪 | `Game.Skills`（目前空壳，依赖 Core+Combat） | `CastEvaluator` / `SpellDefinition` 放这里 |
| 产出火球 | `Fireball` / `NovaFireball` 预制体 + `ProjectileBase` | emit 时 Instantiate 它们 |
| 投射物之间不互撞 | `ProjectileBase` 的同队碰撞注册表（前几天刚加的） | 直接生效，无需改动 |
| 命中结算伤害 | `IDamageable.ReceiveHit` + `CombatDamage.Deal`（预留 seam） | emit 出的投射物命中后照走 |
| 区域/持续伤害类法术 | `AreaDamage` / `BurnStatus` | 某些法术 emit 的就是带这俩的预制体 |
| 充能/冷却 | 现有 `AttackCooldownCounter` + buffer-counter 模式 | 施法器参数（见第 8 节）复用同款心智 |
| 瞄准方向 | 现有 `ClickAimPoint`（按下锁存的准心） | SpellCaster 用它当所有 emit 的基准方向 |

### 6.3 需要新建的东西（一览，细节留给 implementation plan）

- `SpellDefinition`（ScriptableObject）：一个法术的数据。字段大致：类型（emit/modify/multicast）、预制体引用（emit 类才有）、修正参数（modify 类才有，如伤害加成/速度加成）、多抽数量（multicast 类才有）、法力消耗、图标、名字。**新增一个法术 = 新建一个这种资产，不写代码。**
- `WandLoadout`（ScriptableObject）：一个 `SpellDefinition[]`（法杖里从左到右的法术序列）。早期写死在 Inspector，后期由 UI 拖拽编辑。
- `CastModifierState`（readonly struct）：解释器跑的时候"当前累积的修正"。例如 `{ 伤害加成, 速度倍率, 扇形角度 }`。是 struct 所以零 GC、可快照。
- `EmitCommand`（struct）：解释器输出的一条"产出指令"——纯数据，告诉 SpellCaster"产出哪个法术、带什么修正快照"。
- `CastEvaluator`（static 纯类）：核心解释器。`Evaluate(序列, 施法器参数) → List<EmitCommand>`。**可 EditMode 单测。**
- `SpellCaster`（MonoBehaviour，放 `Game.Character`，挂法师身上）：调 Evaluator，拿结果去 Instantiate，应用修正，管充能/法力。**这是连接"纯逻辑"和"Unity 世界"的桥。**

### 6.4 一个需要你知道的分层小麻烦（不影响理解，标记给后续）

后期做"拖拽编程框 UI"时会遇到：`Game.UI` 要能读/写法杖序列，但法杖数据 `SpellDefinition` 在 `Game.Skills` 里。解决办法（我倾向的）：让 `Game.UI` 也引用 `Game.Skills`（它本来就引用了 `Game.Combat`，多引用一个数据层不破坏你的原则），并用一个**共享的 `WandLoadout` ScriptableObject** 当"UI 改它、SpellCaster 读它"的中间媒介。这个等做到 UI 阶段再细化，前期完全不影响。

## 7. 触发（后期再做，但现在要"留好位子"）

你之前指出"触发本质是投射物的一个变种"——**完全正确**。普通火球命中就消失；"触发火球"命中时会多做一件事：**把跟在它后面的法术当作"载荷（payload）"再放一次**。

例：法杖 `[触发火球, 魔法箭]` → 火球飞出去 → 命中墙 → 在命中点再射出一支魔法箭。

它在**数据上**只是火球的一个字段（"我带不带触发"）。但在**运行逻辑上**，它是整套系统里**唯一引入"递归"的机制**：

- 普通 emit：产出完，解释器的活就干完了。
- 触发 emit：产出火球时，要把"后面那段法术"**打包存到火球身上**；等火球命中，**再调一次同一个解释器**去跑这段打包的程序。"解释器调用自己" = 递归。

**为什么前期不做、但现在就要留位子？** 因为只要前期把 `CastEvaluator.Evaluate` 设计成一个**可被反复调用的纯函数**（输入：序列+起点+预算+"继承来的修正"；输出：产出列表+读到哪了），那后期加触发就只是"命中时再调一次 Evaluate"，**不用改解释器主体**。如果前期没留这个签名，后期就得返工。所以这是**前期就要立对、但前期不实现**的东西。

## 8. 必须保留的"施法器参数"（CasterConfig）

我们说"早期不做法杖系统（不做多把法杖、各有属性、可拾取切换）"——这个 scope 控制是对的。**但有一组参数不能跟着砍掉**，否则解释器根本跑不起来：

| 参数 | 作用 | 对应现状 |
|---|---|---|
| **施放延迟（Cast Delay）** | 两次开火之间的间隔 | ≈ 现有 `AttackCooldownCounter` |
| **充能时间（Reload）** | 法杖序列从头读到尾后，重置回开头的冷却 | 新概念，但同样用计数器 |
| **法力（Mana）** | 资源约束：每个法术耗法力，没法力就发不出 | 全新（现在没有资源系统） |

**法力是关键闸门**：没有它，所有组合都能无脑连发，系统失去平衡。早期把这组参数做成一个固定的 `CasterConfig`（SO 或直接挂法师身上的几个字段），**只有一套、不可切换**——但它必须存在。

## 9. 分期计划（每阶段你都能在编辑器里验证）

依赖方向决定顺序——**UI 依赖能正确运行的内核，内核不依赖 UI**：

- **阶段 A — 解释器内核（纯逻辑）**
  - 建 `CastEvaluator` + `SpellDefinition` + `CastModifierState` + `EmitCommand`。
  - 含 multicast 的"块 + 多抽预算"骨架（见决策点，我推荐一次立对）。
  - 法杖序列先写死在 Inspector。
  - **EditMode 单测**验证求值正确（见第 11 节）。这阶段几乎不需要场景，纯逻辑。
- **阶段 B — 接运行时**
  - 建 `SpellCaster`，把 Evaluator 的输出变成真的 `Fireball` 等投射物，走 `CombatDamage` seam 打到敌人。
  - 法师左键改成"运行法杖"。**手感验证。**
- **阶段 C — 编程框 UI + 背包**
  - 拖拽法术进框，框内容 = 喂给 Evaluator 的序列。UI 这时才上场。
- **阶段 D — 触发变种 + 性能护栏**
  - 递归求值 + payload 捕获 + 护栏（递归深度上限 / 单次开火产出上限 / 活跃投射物总数上限）。

## 10. 关键设计决策

### 已拍板（你和网页版 Claude 已确认）
- **是解释器，不是查表**：定义少量语义规则，组合自动涌现。
- **砍掉材料/流体模拟**：3D RPG 没有这个底座，用 `AreaDamage`/`BurnStatus` 保留风味即可。
- **重机制完备、轻数量**：MVP 约 12~15 个法术，覆盖 emit×几种 + modify×几种 + multicast×1~2 + （后期）trigger×1。
- **触发后期做、但签名前期留位**。
- **做"背包格 + 编程框"，不做"多法杖可切换对象"，但保留 CasterConfig 参数**。

## 11. 阶段 A 验收：单测清单（体现可测性）

`CastEvaluator` 是纯函数，所以这些都能在 EditMode 里断言（无需进 Play）：

- 空法杖 → 产出 0 个。
- `[火球]` → 产出 1 个，修正为默认值。
- `[增伤, 火球]` → 产出 1 个，伤害带加成。
- `[火球, 增伤]`（增伤在后） → 产出 1 个，伤害**不**带加成（验证"只影响其后"）。
- `[三重, 火球]` → 产出 3 个。
- `[三重, 增伤, 火球, 火球]` → 产出 3 个，全部带增伤（验证"多重内修正作用于全部"）。
- `[三重, 火球]`（多重后只有 1 个投射物）→ 产出 1 个（预算填不满即作废，验证"单遍不回绕"）。
- 法力不足 → 验证 fizzle 行为符合定义（产出到没法力为止 + `Fizzled=true`）。

## 12. 风险与护栏

- **触发递归会爆炸**（指数级产出 → 爆 GC / DrawCall / 帧率）。护栏分两层：
  - 纯解释器内：递归深度上限、单次开火产出总数上限（可在 Evaluate 里计数拦截，可测）。
  - 运行时 `SpellCaster`/全局：活跃投射物总数上限。
- **热路径零 GC**：修正快照用 `readonly struct`；递归用预分配的栈/对象池，避开你 CLAUDE.md 里 new/boxing 红线。emit 是离散输入触发的一次性分配，可接受。
- **scope 蔓延**：材料、洗牌、多法杖、追踪/弹跳/穿透——全部明确划到 MVP 之外，避免一上来摊子铺太大。

## 13. 名词表（新手速查）

| 词 | 意思 |
|---|---|
| 解释器 / 求值器（Evaluator） | 从左到右"运行"法杖序列的那段逻辑（`CastEvaluator`） |
| emit（产出） | 法术语义之一：生成一个投射物/存在物 |
| modify（修正） | 法术语义之一：改"当前修正状态"，影响其后的产出 |
| multicast（多重/多抽） | 法术语义之一：让本次开火多产出几发 |
| trigger（触发） | 后期机制：投射物命中时再跑一段子程序（唯一的递归） |
| payload（载荷） | 触发投射物身上打包的那段"子程序" |
| 快照（snapshot） | 产出投射物的瞬间，把"当前修正状态"拷一份钉死给它（之后再改修正不影响它） |
| 施法块（cast block）/ 多抽预算（draw budget） | "这一次开火要产出几发"的计数；multicast 增大它 |
| CasterConfig | 施法器全局参数：施放延迟、充能、法力 |
| WandLoadout | 法杖里从左到右的法术序列（一个 SO） |
| EmitCommand | 解释器输出的"该产出什么"的纯数据条目 |
